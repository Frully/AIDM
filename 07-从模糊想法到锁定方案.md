# 从模糊想法到锁定方案

一句「为 Relay 接入自动退款」里藏着两个能让后面全盘返工的陷阱。第一个是问题本身就错了：用户真正的痛点可能不是「退款慢」，而是「人工审批占用了大量客服坐席」。第二个是方案没有被第二意见审视过：Relay 早期 Agent 写完退款方案后，在同一个会话里进行自我评审，它确认无误后直接执行编码；但在并发环境下，同一笔退款因未加锁触发了二次出款——漏洞在事后回看清晰可辨，但写方案的 Agent 在自我确认的瞬间，根本跳不出「当前设计完全正确」的逻辑偏置。

Plan 阶段就是一条收敛漏斗，专门堵这两个口子。

**动手前必须走完「发散质疑问题 → 多视角评审收敛 → 锁定并写成设计文档」这条漏斗；锁定的机制是作者与评审分离，绝不在同一上下文里自批。**

在 AI 驱动开发下，“先想清楚”的工程杠杆被显著放大。Agent 编写速度极快，你给一句语义含糊的指令，它几分钟就能拼出上千行功能看似健全、实则背离核心边界的代码。人类团队尚可通过“边写边改”来摸索方向，是因为人类的编写速度相对可控，方向发生偏差时能够被及时觉察；而 Agent 的高吞吐特征会将错误的架构逻辑在短时间内放大数倍，使得返工成本呈量级上升。所以漏斗的每一步都在动手前完成，产出全是文档。


## 第一步：六问发散，质疑问题本身

一个模糊的业务意图在工程层面至少遗留三类未决要素：隐藏的技术需求（「自动退款」背后是订单数据一致性、审计流水、冲正回滚）、边界安全指标（资金划扣的拦截水位、风控黑名单策略）、架构选型权衡（AI 直接判断并调用扣款，还是 AI 生成建议交规则引擎过滤执行）。发散这一步的任务，是用结构化追问把它们全部逼出水面。

追问由六个强制问题构成骨架，每一问都是一把手术刀，专切一类思维幻觉：

* **需求证据**：是否有真实用户在为此付费、抱怨或流失？必须引用具体数据，戳破“我觉得用户需要”的臆想。
* **现状痛点**：用户今天怎么解决？摩擦力多大？不做会有什么实质损失？戳破“问题不够严重”的托词。
* **精准用户画像**：最迫切的种子用户是谁？拒绝泛泛的“所有商户”，要明确到“日均处理 50 笔以上退款工单、且无力扩编客服的中型 B 端商户”，戳破“服务所有人等于不服务任何人”的空洞。
* **最小可行切入点**：一周内能上线并解决核心摩擦的最窄切口是什么？戳破完美主义大包裹。
* **独特观察**：为什么是现在？有什么旁人没注意到的技术或业务红利？戳破“随时都能做”的时机幻觉。
* **长期适配性**：三年后这方案还成立吗？模型能力跃升或外部 API 变化会让它增强还是过时？戳破“短期有效等于永远有效”的惯性。

六问收尾后，Agent 会汇总分析答案，主动挑战初始的问题框架。通过挑战则进入方案选型，否则必须退回修正并重新澄清。

在 Relay 里，团队最初的构想只有一句“我们需要接入自动退款功能”。Agent 以问询模式介入：

> **Agent**：你有没有真实数据证明这是值得现在解决的问题？
> **团队**：有。客服数据显示每月 12,000+ 退款工单中，65% 是金额小于 5,000 分的小额退款，每笔人工处理约 8 分钟。
>
> **Agent**：用户今天怎么处理这件事？
> **团队**：完全人工。客服查单、判断条件、手动操作网关，全程没有自动化。
>
> **Agent**：你说“自动退款”，但你描述的核心痛点是“人工审批占用了客服 60% 的时间”。你真正想解决的是退款慢，还是客服资源被退款工单绑架？

经过这一轮，问题框架从“接入退款功能”收窄为“将小额退款从人工工单队列中自动分流”。方案方向也随之明确：不是“让 AI 直接决策并执行退款”（非确定性输出，资金安全无法在工程上闭合），而是“LLM 意图分类 + 硬编码规则引擎兜底”。Agent 最终产出设计文档 `docs/design/refund-v1.md`：

```markdown
# Design: Relay 自动退款 v1
Status: DRAFT | Branch: feat/auto-refund

## Problem Statement
Relay 商户客服坐席平均每天处理 80+ 退款工单，其中 65% 为小额退款（< 5,000 分），
每笔人工耗时约 8 分钟，严重挤占高价值工单的响应资源。

## Premises Challenged
- ❌ 初始框架：“我们需要一个退款功能”
- ✅ 修正框架：“我们需要把小额退款从人工工单队列里彻底移走”

## Approaches Considered
### 方案 A：LLM 全权决策并执行退款
模型不确定性无法在工程上保证 100% 不越权划扣，资金安全无法闭合。否决。

### 方案 B：LLM 意图分类 + 硬编码规则引擎兜底（推荐）
AI 解析意图判断类别，规则引擎做金额卡口与黑名单拦截，最终执行由规则决定。
可控、可审计、可回滚。采纳。

## Success Criteria
- 小额退款自动化率 ≥ 90%
- 误退款率 < 0.1%
- 平均处理时间 < 30 秒（从现状 8 分钟）
```

这份文档是漏斗下一步的直接输入。


## 第二步：多视角评审收敛，作者与评审分离

发散给出了一个候选方案，但候选方案要锁定，必须先过评审——并且评审绝不能由作者自己在同一会话里做。

在结构上，自写自审必定失败。大语言模型在单次会话中存在显著的启动效应（Priming Effect）：当 Agent 在同一个上下文里推导并形成初始方案后，其注意力权重已高度倾向于证明自身逻辑的正确性。如果让其进行自我评审，往往只能抓取拼写或接口命名的浅表瑕疵。有效的评审必须通过干净的、物理隔离的上下文，甚至引入与编写模型不同底座的异构模型，让评审 Agent 站在客观的批判视角，降级容错、并发竞态、幂等缺失等隐性漏洞才会显现。

评审分为三个独立视角，混在一起会使每个层面都浅尝辄止：

1. **CEO / 业务视角**：确保方案在 Spec 刚性边界内运行，拦截范围蔓延（Scope Creep），防止 Agent 超前引入不必要的系统复杂度。
2. **工程视角**：专审数据流的异常边界——所有外部写操作（数据库、第三方支付、外部通知）是否有幂等键；并发下是否存在资源竞争、悲观锁或分布式锁是否配置妥当；事务边界是否正确；网络超时降级是否明晰。
3. **设计视角**：仅在涉及用户交互的变更中启用，验证新增界面与交互是否符合设计系统规范，防止 Agent 引入非标组件。

评审投入与改动风险成正比：小型内部重构只跑工程视角，不涉及 UI 就裁掉设计视角，业务目标在 Spec 阶段已明确就免去 CEO 视角。视角间产生冲突时（如工程视角要求引入异步队列，而业务视角因排期要求同步极简），收敛的终点由 Plan 阶段的人类负责人裁决——裁决结论连同被否决的替代方案，必须显式写进设计文档，作为可审计的决策资产。

在 Relay 里，退款方案提交工程视角评审时，评审 Agent 在全新上下文中读取方案，输出整改报告：

```bash
$ gstack review --plan ./docs/plans/auto_refund_flow_v1.md --view eng

[WARN] 金融副作用接口缺失幂等键
  - 缺陷：超时重试分支直接调用 `stripe.refunds.create`，504 超时下重试会重复出款。
  - 整改：本地生成交易凭证，透传基于 sha256(order_id + refund_action_sequence) 的唯一幂等键。

[WARN] 并发下存在资源竞态 (Race Condition)
  - 缺陷：缺少锁机制，两个并发请求可能同时通过金额阈值校验，造成重复出款。
  - 整改：金额校验前用 SELECT ... FOR UPDATE 对订单行加悲观锁。
```

这两条高危报错在方案编写会话里完全没有被捕获。修改设计并重跑工程评审、报错消除后，方案才正式锁定。`Plan done` 的定义就此清晰：**方案锁定（多视角评审通过）+ 写成设计文档**，两者齐备才算锁定。


## 设计文档先行：应对会话断裂的交接契约

锁定后、编码前，最终决策必须沉淀为设计文档。代码只记录当前实现，无法记录“为什么不采用同步方案”这类被否决的替代路径。后来接手的 Agent 无法理解复杂的异步结构，很容易将其重构回有缺陷的同步逻辑。设计文档是应对 Agent “会话记忆断裂”的交接物：大模型每次开工都是冷启动，设计文档是重建架构心智的唯一事实源。

文档流动的方向必须是刚性的：**Spec（系统做什么）→ Design（如何构建）→ Code（实现）**。Spec 是 Design 的刚性输入，严禁跳过 Spec 直接做架构。一份优秀设计文档的标志，就是冷启动的 Agent 据此能直接动手编码。


## gstack 参考实现

发散质疑在 `gstack` 中由 `/office-hours` 运行；收敛阶段由 `/plan-review` 将多视角评审封装成独立命令，确保上下文物理隔离：

```bash
$ gstack office-hours          # 六问质疑 + 产出设计文档
$ gstack plan-review --plan ./docs/plans/auto_refund.md --views eng,ceo
```

需要更彻底的逻辑对抗时，可在 `/codex` 中配置与编写模型不同底座的推理模型作为评审裁判，防止第二意见携带相同的注意力偏置。设计文档统一存放在 `./docs/designs/`，骨架固定：背景与架构取舍（含已否决方案）、副作用与幂等设计（加锁字段、幂等键格式）、异常流与评审结论。即使不使用 `gstack`，这一方法同样适用：动手前开一个专门的会话让 Agent 作为质疑者，评审时更换干净上下文作为批判者。


## 典型反模式

* **跳过发散直接写技术规范**：直接让 Agent 编写接口定义和边界参数。如果框架方向偏离，后续所有的工程技术实现都将构建在错误地基上。
* **六问形式化**：每一问都敷衍回应，使用“用户想要更流畅的体验”代替真实的数据支撑，产出的设计文档不具备任何实质约束力。
* **同一会话自写自审**：Agent 被自身的编写思路误导，放过架构层面的关键漏洞。
* **评审结论不落地**：发现的问题仅在聊天记录中以口头形式存在，没有回写到设计文档，导致后续冷启动的 Agent 无法承接该修改要求。
* **设计文档只复述现状**：只记录“如何实现”，漏掉了“被否决的备选方案”，导致后来接手的 Agent 再次踩入旧方案的坑。


## 本章要点

* **Plan 阶段是收敛漏斗**：发散质疑问题（六问）→ 多视角评审收敛 → 锁定并写成设计文档。
* **高吞吐放大架构偏离代价**：先把问题框架做对，比先把代码写快更为关键。
* **锁定基于作者与评审分离**：自审在结构上必定失败，评审必须引入干净上下文或异构模型。
* **三视角分层评审**：业务、工程、设计视角分工查错，视角冲突由人类负责人裁决并写入设计文档。
* **方案锁定加设计文档即为 Plan Done**：Spec → Design → Code 顺序不可跳过，文档是冷启动 Agent 唯一的持久化契约。
