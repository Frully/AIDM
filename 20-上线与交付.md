# 上线与交付

在 AI 驱动开发的模式下，当我们将部署（Ship）动作交由 Agent 编排触发时，整个发布架构的设计核心，就在于如何控制逻辑错误的下限。当发布一旦涉及底层的关系型数据库结构变更（Schema Migration）与多实例流量切换时，不可逆性（Irreversibility）是将系统推向灾难的最主要诱因。

**Agent 并行交付的底线前提是系统具备绝对的可逆性，包括 Expand-and-Contract 的双向迁移机制与灰度部署自动回滚网关。**

## 风险对称性与可逆性设计

不可逆的操作在博弈论中属于“劣势策略”：一旦执行产生偏差，系统无法通过低阻抗回撤，只能以高昂的代价进行线上热修。

为了让 Agent 安全地驱使生产部署，我们在底层架构上必须进行如下可逆性重构：

### 1. 扩展式数据库迁移（Expand-and-Contract）

`ALTER TABLE DROP COLUMN` 在多实例滚动部署里不是普通的 DDL，而是一颗定时炸弹。当新代码已经不读旧列、旧实例还在读的时候，数据库接受了 `DROP`，旧实例立刻开始收到 `column does not exist` 错误，而此时距离回滚窗口关闭可能只有三十秒。Relay 早期就踩过这个坑：一次退款金额字段的重构直接 `DROP` 了旧列，滚动部署过程中约有 30 秒的窗口期让部分实例返回 500，客服后台报警炸了一屏。

根本原因不是 Agent 粗心，而是约束没有写进 `AGENTS.md`。Agent 生成迁移文件时没有任何阻力，它选了最短路径：一个 SQL 文件，`ADD COLUMN` 之后紧跟 `DROP COLUMN`，一步到位。

#### 约束写在 `AGENTS.md` 里（人类写）

Relay 的修复从 `AGENTS.md` 开始，而不是从提示词开始：

```markdown
## 数据库迁移约束（硬性红线）
- 禁止在同一个迁移文件里同时出现 ADD COLUMN 和 DROP COLUMN 的同名列
- 所有列删除和表删除操作必须拆成两个独立的迁移文件：
  第 1 步（Expand）：添加新列/新表，开始双写，不删旧数据
  第 2 步（Contract）：确认新列/新表稳定后，移除旧列/旧表；
  第 2 步的迁移文件只有在第 1 步的迁移已合并至 main 分支后才允许提交
- 所有迁移文件必须包含 up（向前）和 down（回滚）两个方向
- 大表（超过 100 万行）的 ALTER TABLE 必须在注释里标注预计执行时长
```

这段约束的效果不依赖 Agent 记忆，而依赖 CI 拦截。约束落地分两层：一层是 Agent 读 `AGENTS.md` 生成迁移文件时的行为输入，另一层是 CI 里的 migration-linter 把违规变成构建失败。

#### CI migration-linter（AI 根据约束生成配置，人 review 后提交）

```yaml
# .migration-lint.yml
rules:
  - name: no-direct-drop
    description: "禁止在单个迁移文件里同时出现 ADD COLUMN 和 DROP COLUMN 的同名列"
    pattern: "DROP (COLUMN|TABLE)"
    context: "not_paired_with_expand"
    severity: error

  - name: require-rollback
    description: "每个迁移文件必须有 down 方向"
    check: "has_down_migration"
    severity: error
```

当 Agent 没有读到或绕过了 `AGENTS.md` 里的约束、在第一步迁移里直接写了 `DROP`，CI 会给出如下报错：

```
migration-lint: 1 error found
✗ [no-direct-drop] migrations/20240301_add_amount_cents.sql
  Line 12: DROP COLUMN amount_float
  ERROR: Found DROP COLUMN without prior expand migration merged to main.
  Expand migration 'add_amount_cents' must exist in git history before contract step.
```

这条报错不依赖 code review 是否认真，它直接阻断了合并。Agent 拿到报错后，重新读 `AGENTS.md`，理解了"两步"要求，重新生成了拆分后的两个文件。

#### Relay 的实际迁移：`amount_float` 换成 `amount_cents`

Relay 早期存储退款金额用的是 `amount_float DECIMAL(10,2)`，换成整数分 `amount_cents INTEGER` 是一次典型的 expand-and-contract。迁移文件由 Agent 根据 `AGENTS.md` 约束生成，工程师做 review。

**第 1 步 Expand**（由 Agent 生成，review 通过后合并）：

```sql
-- migrations/20240301_add_amount_cents.sql
-- up
ALTER TABLE refunds ADD COLUMN amount_cents INTEGER;
-- 双写：应用层在写入时同时维护 amount_float 和 amount_cents
-- AGENTS.md 约束：不得删除 amount_float，观察期不少于两周

-- down
ALTER TABLE refunds DROP COLUMN amount_cents;
```

review 时工程师重点检查两点：`down` 方向能否干净回滚（可以，新列直接删），以及双写逻辑是否完整（应用层代码同步 PR 覆盖了所有写入路径）。

**第 2 步 Contract**（两周后，由 Agent 根据 `AGENTS.md` 约束生成）：

```sql
-- migrations/20240315_remove_amount_float.sql
-- up
-- 前置条件：amount_cents 已通过应用层双写覆盖所有历史记录后执行
UPDATE refunds SET amount_cents = ROUND(amount_float * 100) WHERE amount_cents IS NULL;
ALTER TABLE refunds ALTER COLUMN amount_cents SET NOT NULL;
ALTER TABLE refunds DROP COLUMN amount_float;
-- 估计执行时长：refunds 表约 80 万行，预计 40-60 秒（已在 staging 测量）

-- down
-- 注意：此 down migration 不可自动执行，需 DBA 介入从备份恢复
-- 影响范围：refunds 表全量
```

review 时工程师重点核查三件事：`WHERE amount_cents IS NULL` 的数据修补是否完整（要先跑 `SELECT COUNT(*) FROM refunds WHERE amount_cents IS NULL` 确认残余行数为零）、`down` 方向是否明确标注了"需 DBA 介入"的风险、大表操作是否附上了 staging 实测时长。最后一点在 `AGENTS.md` 里有明确红线，Agent 通常会带上，但 review 时仍需确认数字是真实测量的而非估算。

这套流程的根源不是两个 SQL 文件的格式，而是 `AGENTS.md` 里那条"禁止直接 DROP"的红线：它让 Agent 生成文件时有约束可循，让 CI linter 有规则可检，让工程师 review 时有结构化的关注点，而不是从头读一遍 SQL 猜意图。

### 2. 部署自动回退（Auto-Rollback）
新容器镜像推向集群后，流量控制网关（如 Ingress / Service Mesh）必须以灰度比例（如 5% -> 20% -> 100%）逐步注入真实流量。在灰度观测周期内，一旦可观测性日志大盘抛出异常飙升或健康检查端点报错，编排器必须瞬间执行自动回撤，将路由切回老版本，让新版镜像原地挂起。

## 黄金数字定额测试与幂等拦截

针对退款、扣款等金融/发钱级别的黄金路径，上线前CI必须强制通过两项拦截：
* **黄金数字定额测试**：喂入固定测试数据，验证在浮点数四舍五入运算下，金额精确到分（Cents）且无精度丢失。
* **API 调用绝对幂等（Idempotency）**：模拟在网络严重抖动下，同一笔退款请求在 1 秒内连续发送 3 次。系统必须利用 `idempotency_key` 物理锁定逻辑锁，确保 Stripe 只有一次扣款动作执行，另外两次返回幂等冲突响应。

## 调试挫败细节：预发环境下因重放造成的二次退款

在一次升级退款模块的部署演练中，开发团队在 CI 阶段配置了“模拟高并发网络抖动重放测试”。

由于 Agent 在底层逻辑中将 Stripe API 调用返回的 `idempotency_key` 的哈希生成规则从 `uuid` 重构为了不含时间戳的静态 `order_id`，且未能在本地缓存上施加分布式锁（Redis lock）。

在预发灰度部署后的第一分钟，自动化红蓝对抗引擎模拟了一起断网后客户端重试。由于前一笔退款写入尚未在数据库完成提交，重试请求在微秒级间隙内绕过了老字段查询，Stripe 接口被物理调用了两次，直接向测试用户返回了两笔足额退款。

正是因为发布防线强设了“黄金测试与高并发幂等断言”，测试用例在预发部署的第一关便直接判定报错：`AssertionError: Multiple refunds created for order_id: 19283`。这痛击了团队，促使团队重新收紧了对退款执行器的强校验，杜绝了线上重复发钱事故的发生。

## gstack 参考实现

在 gstack 体系中，自动化部署与数据库迁移被集成在发布技能 `/ship` 与合并控制中。

在大仓本地或 CI 流水线中，工程师通过调用发布指令，自动触发迁移文件的 lint 扫描以及灰度流量比例划分：

```bash
# 执行受控的灰度上线与健康回退检测
$ gstack ship --image relay-app:v2.1.0 --canary-rate 10
```

这套部署脚本和流量路由方案与云厂商的实际交付网关相对接，构成了可替换的物理运行层。其核心原则是不论对接何种基础设施，系统都必须通过健康探测（如 `/health`）对新实例进行实时检验，并在报错超标时通过自动化回滚机制自动回撤流量。

## 典型反模式

* **直接跑一刀切物理迁移**：发布脚本中直接包含 `ALTER TABLE DROP COLUMN` 等不可逆变更。一旦新代码上线出现 bug，回滚时会由于表结构已被物理破坏而彻底瘫痪。
* **全量一把梭部署**：新版本镜像直接 100% 覆盖老服务实例，没有任何灰度和自动回退保护，导致逻辑崩溃时只能手忙脚乱地重新编译构建，线上大面积挂红数十分钟。
* **缺少发钱幂等拦截**：对支付等发钱通道不提供基于 `idempotency_key` 的强契约测试，全靠模型临场发挥，导致用户遇到网络波动疯狂点击时产生双倍财务扣损。

## 本章要点

* 可逆性是向 Agent 授权部署的核心前置。迁移必须采用双向可逆设计（Expand-and-Contract），不搞一刀切破坏变更。
* 部署必须配置灰度注入与可观测性闭环，遇到核心错误指标超标时必须在数十秒内自动回滚，压缩故障时间。
* 涉及资金流的核心防线，上线前必须强通过黄金数字精度测试与基于幂等键的高并发并发重构验证。

> 一个核心特性从最初的设计Spec、本地隔离多线开发、CI 可观测性拦截一直走到了线上的可逆发布。但这并不意味着研发回路已经结束。为了让下一次迭代变得更加高效和精准，我们应该如何沉淀这轮开发中的全部教训，并喂回我们的 Harness 宪法？我们将在第九部分 `反射（Reflect）` 中展开讨论。
