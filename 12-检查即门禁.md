# 检查即门禁

在单体大仓与分层规则就绪后，我们必须为项目搭建一道自动化收网的防线。大模型协同研发中，如果我们仅向 Agent 提供一系列"应当遵守"的检查建议，它是自动不会在编写完代码后主动运行校验的。

**静态编译、风格扫描与各级自动化测试必须固化为流水线上不可绕过的刚性检查门禁，禁止任何未通过全绿验证的代码合入主干。**

将检查由"建议跑一下"的软弱提醒，升级为"不跑过就无法提交"的刚性物理门禁，是实现高质量 Agent 自治研发的底盘防线。

## 门禁流水线的多维分层

一道合格的刚性门禁不仅要做到拦截，还必须根据"反馈速度与资源消耗"在时间线上执行科学的分层编排。如果我们将高耗时的测试用例全部堆在本地代码提交的一瞬间，会导致每次提交流程缓慢不堪，迫使开发团队和 Agent 采用绕过 hook 的手段。

因此，门禁必须采用由快到慢、逐层递进的多级设计：

* **代码提交卡口（Pre-commit Hook）**：仅允许运行毫秒级响应的极速任务，如格式化校验（Format）、代码风格扫描（Lint），以及对于资金等红线字段的安全规则扫描。
* **推送前置卡口（Pre-push Hook）**：运行秒级响应的静态规则校验，如强类型编译检查（TS Compile）、依赖流向守卫扫描（Dependency Cruiser）以及核心逻辑单元测试。
* **合并准入流水线（CI Pipeline）**：在远端容器中启动全量的黑盒测试与数据库集成测试。这是合并入主干前的最后一道物理大闸。

通过这一科学的分层设计，Agent 能够在开发的每一个微小步骤（如每次 `git commit`）中尽早获取反馈，而不用等到 PR 合并阶段才被一堆低级格式错误阻塞。

## 报错信息是 Agent 唯一的反馈信号

对于 Agent 而言，检查门禁不仅是用来阻断错误提交的阻拦索，更是它实现自愈（Self-healing）的核心反馈通路。

如果门禁仅抛出类似于 `Process exited with code 1` 这样模糊且缺乏上下文的报错，Agent 会因为完全不知道错在何处而失去自愈依据，在随后的会话中陷入盲目的瞎猜和无意义的代码重试中，白白耗费大量的 Token 额度。

因此，配置质量检查工具时，我们必须要求其报错输出具备高度的可读性与确定性。报错必须精确指出：**在哪一行（具体物理位置）**、**违反了哪条红线（规则原因）**、以及**期望的正确输入与实际输入之间的 Diff 差值**。门禁能拦得越死，且说得越明白，Agent 自我修复代码的效率就越高。

## 门禁配置从哪里来

这里有一个容易被忽视的工作流问题：这些 `lefthook.yml`、`.github/workflows/ci.yml` 是谁写的？

答案是 Agent 生成的。你在 `AGENTS.md` 里用人类语言写下约束，Agent 据此生成配置文件，你 review 后提交。这是分工的关键：**人负责定义约束，Agent 负责把约束翻译成机器可执行的形式**。

手写一份完整的 GitHub Actions workflow 容易出错，还要记住各种 action 的版本号和参数细节。而如果你只写约束描述，Agent 会帮你处理这些细节，你的精力放在 review 语义是否正确——步骤顺序对不对、缺没缺检查项、覆盖范围够不够。

这个工作流在 Relay 项目上运行了数十次迭代：每当架构约束更新，在 `AGENTS.md` 里加一行说明，Agent 就同步更新对应的配置文件，整个过程没有手动维护 YAML 的心智负担。

## 从约束到配置：Relay 的实际路径

### `AGENTS.md` 里的约束描述（人类写）

```markdown
## CI 门禁约束
- 所有提交（commit）前必须通过：ESLint 检查、TypeScript 类型检查
- 所有 push 到主分支前必须通过：单元测试、E2E 冒烟测试（核心路径）
- 违反任何一项直接阻断，不允许 --no-verify 绕过（AGENTS.md 里写明）
```

### Agent 生成的 `lefthook.yml`

```yaml
# lefthook.yml（AI 根据 AGENTS.md 中的门禁约束生成，人类 review 后提交）
pre-commit:
  parallel: true
  commands:
    lint:
      glob: "*.{ts,tsx}"
      run: pnpm eslint {staged_files} --max-warnings 0
    typecheck:
      run: pnpm tsc --noEmit

pre-push:
  commands:
    unit-test:
      run: pnpm test --passWithNoTests
    e2e-smoke:
      run: pnpm playwright test --grep @smoke
```

`pre-commit` 下的两个命令并行执行，整个阶段 1–3 秒内完成：lint 只扫当次暂存的文件（`{staged_files}`），不全量扫；typecheck 是全量的，但 `tsc --noEmit` 对有缓存的项目通常一两秒内出结果。`--max-warnings 0` 的含义是警告即失败：不允许"先提交、warning 以后再说"，因为积累下来的 warning 最终会让 lint 形同虚设。

`pre-push` 串行执行，耗时较长但只在推送时触发：`unit-test` 跑全量单元测试，`e2e-smoke` 只跑打了 `@smoke` 标签的核心路径（如登录、主要工作流），不跑完整 E2E 套件。把重测试放在 push 阶段而非 commit 阶段，这是防止 Agent 绕过 hook 的关键——hook 太慢，Agent 就会加 `--no-verify`。

## 远程 CI：本地 hook 兜不住的那一层

本地 hook 有一个物理上无法消除的漏洞：`git push --no-verify`。技术上任何人（或 Agent）都可以绕过它。

远程 CI 不可绕过。PR 没有通过 CI 就无法合并，这不是惯例，是 GitHub 的分支保护规则在技术层面强制执行的。

### `AGENTS.md` 里的远程门禁约束（人类写）

```markdown
## CI 约束（远程不可绕过）
- 每次 PR 必须通过：全量 lint、全量类型检查、全量单元测试、架构依赖守卫
- main 分支的 push 额外跑：E2E 全量测试、安全扫描
```

### Agent 生成的 `.github/workflows/ci.yml`

```yaml
# .github/workflows/ci.yml（AI 根据 AGENTS.md 生成，人类 review 后提交）
name: CI

on:
  pull_request:
    branches: [main]
  push:
    branches: [main]

jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v3
        with:
          version: 9
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'pnpm'

      - name: 安装依赖
        run: pnpm install --frozen-lockfile

      - name: Lint 检查
        run: pnpm eslint . --max-warnings 0

      - name: 类型检查
        run: pnpm tsc --noEmit

      - name: 依赖守卫（架构层检查）
        run: pnpm depcruise --validate .dependency-cruiser.js src

      - name: 单元测试
        run: pnpm test --coverage

  e2e:
    runs-on: ubuntu-latest
    needs: validate
    if: github.ref == 'refs/heads/main'
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v3
        with:
          version: 9
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'pnpm'

      - name: 安装依赖
        run: pnpm install --frozen-lockfile

      - name: E2E 全量测试
        run: pnpm playwright test
```

这里有几个设计选择值得拆开说。

`validate` 和 `e2e` 分成两个独立 job。`validate` 跑 lint、类型检查、依赖守卫、单元测试，2–3 分钟内出结果——这是每次 PR 都要过的门槛。`e2e` 依赖 `validate`（`needs: validate`），且只在推到 `main` 分支时触发（`if: github.ref == 'refs/heads/main'`）。E2E 跑完通常要 10 分钟以上，PR 阶段不必每次都等它，push 到 main 才是最后一道闸。

`pnpm install --frozen-lockfile` 是 CI 里的铁律：禁止 pnpm 在 CI 环境里更新 lockfile。这意味着如果有人本地新加了依赖但没有提交更新后的 `pnpm-lock.yaml`，CI 会直接报错，而不是悄悄拉个不一致的版本装上去然后测试通过。

依赖守卫（`depcruise --validate .dependency-cruiser.js src`）在 CI 里而非 hook 里，正是因为 hook 可以绕过而 CI 不能。架构层规则——比如 `packages/workers` 不得直接引用 `packages/api` 的内部模块——只要有一次被绕过就会留下技术债，所以它必须放在不可绕过的位置。

## 主线演练：Relay 退款 Lint 拦截与自愈迹线

Relay 在加这套门禁之前，有一次架构违规是在 code review 阶段才被发现的：某个 Agent 在实现退款逻辑时，在 `packages/workers` 里直接 import 了 `packages/api` 的内部 helper，越过了严格的层级边界。问题被发现时，这个 PR 已经积累了 30 多个 commit，依赖关系纠缠在三个文件里，拆解花了将近半天。

加了阶梯式门禁之后，同样的问题在本地 `pre-push` 阶段就被 `depcruise` 拦截了，当时 Agent 只写了两个文件。

在 `packages/workers/src/handlers.ts` 中试图编写如下代码时：

```typescript
// 越线代码示例
const refundValue = 12.5; // 浮点金额
console.log(`initiating refund for user: ${userId}`); // 裸日志
```

在 Agent 执行 `git commit` 的那一刻，本地 Pre-commit Hook 会强行启动并将其拦截，在终端输出如下高度可定位的报错日志：

```bash
$ git commit -m "feat: trigger refund process"
RUNNING PRE-COMMIT LINTERS...

✖ packages/workers/src/handlers.ts:24:9
  Error: Floating point numbers are forbidden for financial values.
  Actual: refundValue: number = 12.5
  Fix: Please use integer Cents (int) and name variables with a 'Cents' suffix (e.g. refundCents = 1250).

✖ packages/workers/src/handlers.ts:25:9
  Error: Standard 'console.log' is forbidden in production code.
  Fix: Please use structural logger 'pino' (e.g. logger.info({ userId }, "initiating refund")).

Lefthook: [pre-commit] failed to pass. Commit aborted.
```

这行直白明确的报错日志构成了 Agent 唯一的反馈输入。在读取这行报错迹线后，Agent 会迅速锁定并精准改写代码，通过将变量改写为 `const refundCents = 1250` 并使用 `logger.info` 重新提交，让门禁顺利通行。这极大减轻了人工 Reviewer 逐行检查的包袱。

如果 Agent 在 push 时触发了依赖守卫，报错同样指向具体位置：

```bash
$ git push origin feat/refund-v2
RUNNING PRE-PUSH CHECKS...

✖ packages/workers/src/handlers.ts → packages/api/src/internal/helpers.ts
  Error: Dependency rule violation — 'workers' must not depend on 'api/internal'.
  Rule: packages/workers may only import from packages/shared.
  See: .dependency-cruiser.js rule 'no-workers-to-api-internal'

Lefthook: [pre-push] failed to pass. Push aborted.
```

Agent 看到这条报错，知道需要把这个逻辑移到 `packages/shared` 里，或者通过 `packages/api` 的公共接口调用。整个修复在同一个会话里完成，没有跨 PR 的上下文丢失。

## gstack 参考实现

在 gstack 体系中，本地代码提交的前置检查使用 `lefthook` 进行调度，而静态代码风格扫描推荐使用开箱即用的 `Biome` 工具链。

在项目根目录下维护的 `lefthook.yml` 文件中，声明在 `pre-commit`（本地提交前）和 `pre-push`（本地推送前）需要强制执行的任务卡口：

```yaml
# lefthook.yml
pre-commit:
  parallel: true
  commands:
    lint-format:
      glob: "*.{js,ts,tsx}"
      run: npx biome check --write {staged_files}
    typecheck:
      run: npx tsc --noEmit
```

在 CI 环境中，gstack 使用统一的命令行接口调用预置检查，确保本地 Hook 配置与远端 CI 流水线规则完全咬合。这些配置是由规则解释层编译而成的物理实现层，关键在于在合并 PR 之前利用不可绕过的 GitHub Actions 流水线，对 Agent 提交的每一行代码实施刚性兜底。

## 典型反模式

* **依靠人肉 Review 充当质量拦截器**：在高频的 Agent 并发开发流中，人工眼力根本无法把关每一行 Lint 和架构引用，资深工程师的有限精力不应消耗在重复性的格式检查上。
* **门禁报错缺乏有效定位信息**：测试失败只给出一个 `Test suite failed` 报告，没有指出具体的数据 Diff，导致 Agent 在错误的逻辑分支中无头苍蝇式地乱试。
* **将全量集成测试极度堆在前置 Hook**：导致每次本地提交代码都需要运行数分钟的外部 API 联调，因反馈周期过长而避开门禁。
* **只有本地 hook，没有远程 CI**：任何人（包括 Agent）都可以用 `--no-verify` 绕过本地 hook，没有 CI 的项目等于没有门禁。

## 本章要点

* 检查必须是物理上不可被绕过的流水线刚性拦截门禁，过不了检查就绝对无法合并。
* 门禁必须按运行快慢在提交、推送与 CI 各个时机进行分层调度，确保反馈闭环极速敏捷。
* 门禁的报错是给 Agent 的核心反馈，报错必须做到精准定位、明示规则，提供清晰的修复指引。
* `AGENTS.md` 写约束，Agent 生成配置，人 review 后提交——这是配置维护的正确分工，不是手写 YAML。
* 架构依赖守卫必须放进远程 CI，而非只放本地 hook；hook 可以绕过，CI 不能。
