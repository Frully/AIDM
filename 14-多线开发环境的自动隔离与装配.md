# 多线开发环境的自动隔离与装配

多线并行开发的技术可行性依赖于运行环境的物理隔离。如果我们仅在代码层面隔离了目录，却要依赖人工去手动挑空闲端口、建测试数据库以及修改环境变量，繁琐的环境装配流程很快就会彻底打消团队使用并行的积极性，重新退回到低吞吐的单线开发模式。

**必须在会话启动钩子（SessionStart Hook）中部署并发安全的资源分配与环境装配脚本。使新开辟的每一个 Worktree 目录在首轮开工时，一键自动化装配出独立且不撞车的网络端口与沙箱数据库。**

自动化并不是一个用来点缀的次要工具，而是决定并发 Worktree 体系能否在真实工业研发中应用的关键分水岭。

## 动态装配的四个硬性原则

要让 Agent 在新的 Worktree 中做到“插电即用、无感就绪”，装配脚本的编写必须严格遵循如下四项原则：

* **会话启动自动触发**：使用 Git Hook 拦截 Agent 第一次切入工作目录或启动客户端的时刻，自动触发装配，无需开发人员手动运行配置命令。
* **决定性端口派生**：根据当前并发安全分配的“槽位（Slot）”编号，以 `基准端口 + 槽位偏移` 的算术公式静态派生出该会话的专有端口。严禁使用随机端口探测（那样会导致每一次重启端口都会漂移，使联调极为困难）。
* **分配机制并发安全**：几条并行线可能在同一秒内被多开启动，必须使用物理文件锁（flock）对共享的槽位表进行排他性加锁读取与占用，防止两个会话抢到同一个槽位发生端口碰撞。
* **装配逻辑完全幂等**：装配脚本必须是可重入的。同一 Worktree 目录多次重复运行脚本，结果必须保持完全一致，已起好的环境自动复用，严禁每跑一次就重复派生或修改配置文件。

## 密钥复用与端口隔离

在动态环境隔离中，我们只需隔离那些会引发物理资源冲突的实体（如 API 监听端口、沙箱数据库名、Redis 缓存实例），而应当复用不冲突的全局资产（如大模型调用 API 密钥、云平台存储凭证）。

开发人员不应为每一个 Worktree 各自去申请和配置一套外部 API 密钥，这不仅引入了繁琐的安全运维开销，也增加了密钥泄露的风险。

正确的做法是采用“从主模板派生配置”的方案。装配脚本自动读取根目录下的 `.env.template`（包含全局共享的只读密钥），通过字符串替换（如 `sed` 命令）仅修改其中的 `PORT` 与 `DATABASE_URL` 字段，并生成专属于该目录的 `.env` 配置文件。这既保证了资源隔离的纯净度，又规避了到处手动配置密钥的麻烦。

## 主线演练：Relay 自动化装配 Hook 脚本与执行迹线

在 Relay 示例项目中，大仓提供了一键装配脚本 `scripts/setup_worktree.sh`。该脚本在 Agent 切入工作区或初始化环境时自动调用：

```bash
#!/usr/bin/env bash
# scripts/setup_worktree.sh
set -euo pipefail

# 1. 确定当前 Worktree 目录名作为唯一标识
WT_NAME=$(basename "$(pwd)")
LOCK_FILE="/tmp/relay_worktree_allocation.lock"

# 2. 并发安全地分配槽位 (Slot)
exec 9>"$LOCK_FILE"
flock -x 9

SLOT_FILE="/tmp/relay_worktree_slots.db"
touch "$SLOT_FILE"

# 查找当前 Worktree 是否已分配槽位，若没有则分配一个空闲的 (1-5)
SLOT=$(grep "^${WT_NAME} " "$SLOT_FILE" | cut -d' ' -f2 || true)
if [ -z "$SLOT" ]; then
  # 找出 1-5 中未被占用的最小数字
  USED_SLOTS=$(cut -d' ' -f2 "$SLOT_FILE" | sort -n)
  SLOT=1
  for s in {1..5}; do
    if ! echo "$USED_SLOTS" | grep -q "^$s$"; then
      SLOT=$s
      break
    fi
  done
  echo "${WT_NAME} ${SLOT}" >> "$SLOT_FILE"
fi

flock -u 9

# 3. 计算端口与数据库映射
PORT=$((3000 + SLOT))
DB_NAME="relay_sandbox_${WT_NAME}"

# 4. 基于模板生成专属环境变量配置文件
if [ ! -f .env ]; then
  sed -e "s/{{PORT}}/${PORT}/g" \
      -e "s/{{DB_NAME}}/${DB_NAME}/g" \
      -e "s/{{REDIS_DB}}/${SLOT}/g" \
      .env.template > .env
  echo "[INFO] Generated isolated environment config in .env for ${WT_NAME} (Port: ${PORT}, DB: ${DB_NAME})"
fi
```

当 Agent 第一次进入 `wt-refund-01` 目录开展工作时，该脚本自动执行并输出：

```bash
$ bash ../scripts/setup_worktree.sh
[INFO] Generated isolated environment config in .env for wt-refund-01 (Port: 3001, DB: relay_sandbox_wt-refund-01)
```

通过这套机制，各线开发环境实现了零人工干预的就绪，为并发开发彻底扫清了环境装配摩擦力。

## Worktree 装配脚本的生产规范

上面的 `setup_worktree.sh` 解决了并发安全的通用问题，但实际项目还需要两个更具体的脚本：一个负责把新 Worktree 从零装配到可运行状态，一个负责任务完成后把所有资源干净地回收掉。这两个脚本的需求可以直接写进 `AGENTS.md` 或任务描述，由 AI 生成，人类 review 后提交。

Relay 在 `AGENTS.md` 里有这样一段脚本需求描述：

```markdown
## Worktree 脚本需求

wt-setup.sh：为新 Worktree 装配独立运行环境
- 接受参数：worktree 名称（如 RELAY_REFUND_01）
- 根据名称提取末尾编号，分配端口（基础端口 3000 + 编号）
- 创建独立的 PostgreSQL 数据库（命名规则：relay_[worktree名小写]_dev）
- 写出 .env 到 Worktree 根目录（包含 PORT、DATABASE_URL、WORKTREE_NAME）
- 输出装配摘要；已存在的资源跳过而非报错

wt-clean.sh：清理 Worktree 运行环境
- 接受参数：worktree 名称
- 停止对应的开发服务器进程（若在运行）
- 删除 PostgreSQL 数据库
- 删除 .env 文件
- 执行 git worktree remove 移除该 Worktree
```

AI 根据这段需求生成了 `wt-setup.sh`，核心逻辑如下：

```bash
#!/bin/bash
# .claude/wt-setup.sh — AI 根据上述规范生成，人类 review 后提交进代码库
set -e

WORKTREE_NAME=$1
if [ -z "$WORKTREE_NAME" ]; then
  echo "Usage: ./wt-setup.sh <worktree-name>"
  exit 1
fi

# 从名称末尾提取编号（RELAY_REFUND_01 → 1），计算专属端口
SUFFIX=$(echo "$WORKTREE_NAME" | grep -oE '[0-9]+$')
PORT=$((3000 + 10#${SUFFIX}))   # 10# 强制十进制，避免 08/09 被解析为八进制
DB_NAME="relay_$(echo "$WORKTREE_NAME" | tr '[:upper:]' '[:lower:]')_dev"

echo "装配 Worktree: $WORKTREE_NAME"
echo "  端口: $PORT"
echo "  数据库: $DB_NAME"

# 创建数据库，已存在则跳过
createdb "$DB_NAME" 2>/dev/null || echo "  数据库已存在，跳过"

# 写出 .env（已存在则跳过，保证幂等）
if [ ! -f ".env" ]; then
  cat > ".env" << EOF
PORT=$PORT
DATABASE_URL=postgresql://localhost:5432/$DB_NAME
WORKTREE_NAME=$WORKTREE_NAME
EOF
  echo "  .env 已生成"
else
  echo "  .env 已存在，跳过"
fi

echo "Worktree $WORKTREE_NAME 装配完成"
```

这个脚本由 AI 生成，但 review 时有三处需要人工确认：端口计算里 `10#${SUFFIX}` 的八进制保护逻辑是否合适、数据库命名格式是否符合 DBA 的约定、`.env` 里是否漏掉了 Relay 其他服务需要的环境变量（比如 `REDIS_URL` 或 `OPENAI_API_KEY` 的引用方式）。通过 review 后提交进代码库，之后所有 Worktree 用同一份脚本装配，彼此行为一致。

## SessionStart Hook：Agent 的冷启动上下文

装配脚本解决的是环境资源的问题。但每次在某个 Worktree 里打开新的 Claude Code 会话时，Agent 面对的是一个完全空白的上下文——它不知道自己在哪条任务线上，不知道当前分支的意图，也不知道这个 Worktree 要跑在哪个端口。

这个问题通过 Claude Code 的 `SessionStart` hook 解决。每次 Agent 会话启动时，hook 自动执行一段脚本，把 Worktree 的当前状态注入进来。配置本身也提交进代码库，所有 Worktree 共享同一份：

```json
// .claude/settings.json
{
  "hooks": {
    "SessionStart": [
      {
        "matcher": "",
        "hooks": [
          {
            "type": "command",
            "command": "bash .claude/wt-session-start.sh"
          }
        ]
      }
    ]
  }
}
```

对应的 `wt-session-start.sh` 同样由 AI 生成：

```bash
#!/bin/bash
# .claude/wt-session-start.sh — AI 生成，人类 review 后提交
# 每次 Agent 会话启动时执行，输出当前 Worktree 的状态上下文

echo "=== Relay Worktree 会话上下文 ==="
echo "当前目录: $(basename "$(pwd)")"
echo "当前分支: $(git branch --show-current)"
echo ""
echo "=== 项目约束 ==="
cat AGENTS.md
echo ""
echo "=== 当前任务 ==="
if [ -f "TODOS.md" ]; then
  grep -E "^\- \[.\]" TODOS.md | head -10
else
  echo "（无 TODOS.md）"
fi
echo ""
if [ -f ".env" ]; then
  echo "=== 本 Worktree 环境 ==="
  grep -E "^(PORT|DATABASE_URL|WORKTREE_NAME)=" .env
fi
```

在 Relay 的 `RELAY_REFUND_01` Worktree 里打开新会话时，Agent 在第一条消息之前就看到：

```
=== Relay Worktree 会话上下文 ===
当前目录: RELAY_REFUND_01
当前分支: feat/refund-idempotency

=== 项目约束 ===
[AGENTS.md 全文]

=== 当前任务 ===
- [ ] 退款接口幂等键校验逻辑
- [ ] 退款金额上限红线接入 CI

=== 本 Worktree 环境 ===
PORT=3001
DATABASE_URL=postgresql://localhost:5432/relay_relay_refund_01_dev
WORKTREE_NAME=RELAY_REFUND_01
```

没有这个 hook 之前，Relay 团队遇到过一个具体的问题：开发者 A 在 `RELAY_REFUND_01` 里改好退款逻辑后，第二天再开一个会话时 Agent 不知道自己在哪个任务线上，把尚未完成的退款幂等键校验当作"上一个任务的残留"跳过了，直接开始写新功能。等到 code review 时才发现 `TODOS.md` 里那条任务从未真正完结，只是被沉默地绕过了。SessionStart hook 上线后，Agent 一打开会话就读到了 `TODOS.md` 里未完成的条目，不需要人来提醒。

## 完整使用流程

把两块工具串起来，一个 Worktree 的完整生命周期是：

1. 在 `AGENTS.md` 里写好 `wt-setup.sh` 和 `wt-clean.sh` 的需求描述，让 AI 生成脚本，review 后提交到 `.claude/` 目录。
2. 在 `.claude/settings.json` 里配好 `SessionStart` hook，让 AI 生成 `wt-session-start.sh`，review 后提交。
3. 新建任务线：`git worktree add ../RELAY_REFUND_01 -b feat/refund-idempotency`，然后进入目录执行 `bash .claude/wt-setup.sh RELAY_REFUND_01`，端口和数据库自动装配完毕。
4. 在这个目录里打开 Claude Code 会话：`SessionStart` hook 自动触发，Agent 看到分支名、`AGENTS.md` 全文、未完成 TODO 和端口配置。
5. 任务完成并合入主线后：执行 `bash .claude/wt-clean.sh RELAY_REFUND_01`，数据库删除、`.env` 清理、Worktree 从 `git worktree list` 移除。

这是一次性的 harness 投入。两个脚本和一个 `settings.json` 加起来不超过 80 行，但之后每新建一条 Worktree 任务线都从中受益，每次 Agent 会话都从完整的上下文开始，不再需要人工口头交代"你现在在做什么"。

## 典型反模式

* **随机端口盲目匹配**：每一次重启环境都使用系统随机分配的临时端口，导致前后端联调以及数据库连接的配置路径频繁漂移，无法稳定复现网络问题。
* **槽位分配缺乏进程互斥锁**：当三条并行线同时被自动化流程拉起时，因槽位读取冲突，多线同时分到了相同的端口，导致运行时抛出 `EADDRINUSE` 错误崩溃。
* **环境初始化缺乏幂等校验**：重复执行启动命令导致本地测试库被清空重置，破坏了 Agent 在上一次调试时积累的测试脏数据状态。
* **不提交 SessionStart hook**：把 `wt-session-start.sh` 放在本地不提交，换台机器或新成员加入时完全不知道这套机制，Agent 回到冷启动状态，任务上下文全靠人工口头补充。

## 本章要点

* 并行开发必须配备在会话启动时自动触发的动态环境装配机制，做到新线插电即用。
* 动态隔离聚焦于网络端口、隔离数据库等冲突实体，全局共享的外部接口密钥凭证走主模板派生，不要为每个 Worktree 单独申请。
* 装配脚本必须具备并发安全加锁、静态端口派生以及幂等可重入三条特质，消除手动维护成本。
* 装配脚本的需求描述写进 `AGENTS.md`，由 AI 生成，人类 review 后提交——这是 harness 里文档驱动代码生成的一个具体体现。
* `SessionStart` hook 配合 `TODOS.md`，让每个 Agent 会话从完整上下文开始，不再依赖口头交代任务线状态。

> 并行环境的自动装配确保了开发线的高吞吐与零碰撞。但当代码最终合入并投入运行后，我们必须确保它产生的行为在生产环境是可追踪和可诊断的。下一章讲如何设计机器友好的结构化日志系统。
