# 多视角计划评审：作者与评审分离

在系统工程中，设计方案的主写者往往是其自身逻辑盲区的最大维护者 ── 盲区之所以存在，正是因为主写者在构思时未能将其纳入认知。如果让同一个 Agent 角色在一个会话上下文里进行自我方案评审，它只会倾向于确认自己的逻辑合理性，而无法挑战自身逻辑的惯性。这并非工作态度问题，而是所处上下文的位置使然。

**写方案与审方案必须在物理与心智上下文上绝对解耦，严禁让同一个 Agent 或处于同一对话上下文的角色为自己的方案签字盖章。**

实现作者与评审分离的核心在于换位。我们需要通过切换角色、甚至换用不同参数或系列的大模型，让评审者在一个干净、全新的上下文里加载方案。新上下文没有“我刚刚写下了这行代码”的惯性包袱，它审视这份设计，就如同审视一份完全陌生的代码库提交一样冷静。

## 三视角协同防线

一份具备工程承重力的方案，其逻辑缺陷通常分布在不同的维度，需要不同的专业眼光执行交叉验证。在锁定方案并启动编码（Build 阶段）前，方案必须经历三道评审防线：

* **业务与范围视角**：评估方案的物理边界是否与 Spec 一致，严禁无谓的技术超前设计（如引入不必要的中间件或复杂的过度解耦），同时核查是否遗漏了核心业务流程。
* **工程与架构视角**：复核数据流向的合理性，评估高并发、重试幂等、网络超时和数据库事务的边界处理，并审查测试矩阵是否完整覆盖了资金或安全相关的关键路径。
* **设计系统视角**：对于涉及人机协同或前台界面的功能，审查其是否严格遵循设计系统的单一事实源，避免产生非标组件或分裂的用户交互链路。

通过这三种视角的独立评审，将修改意见强制收敛并修改回原方案中。一旦完成收敛，方案即被锁定，作为下一步任务分解与并行开发的刚性输入。

## gstack 评审套件参考

在 `gstack` 体系中，这三种不同的评审视角被解耦为独立的原子 Skills。利用 `/autoplan` 工作流，可在 Build 编码启动前，自动化调度这些评审角色并合并其产出的 Markdown 评审意见：

* `/plan-ceo-review`：加载业务视角，负责砍掉冗余的技术超前设计，强制让功能边界对齐 Spec 定义。
* `/plan-eng-review`：加载工程视角，专门用于诊断系统数据流的完备性、寻找逻辑漏洞并验证异常处理矩阵。
* `/plan-design-review`：加载设计视角，负责扫描界面层面的交互设计，防止出现偏离组件库的事后硬编码。

为了引入彻底的心智对立，还可以通过 `/codex` 调度与主写 Agent 完全不同的底层大模型（如以推理模型写方案，以长上下文或轻量模型挑漏洞），获取客观的“第二意见”（Second Opinion）。

## 主线演练：Relay 自动退款方案评审

在 Relay 示例项目的自动退款方案设计中，主写 Agent 生成了一份看起来逻辑自洽的方案。但当该方案被提交给三个独立的评审 Skill 并在全新上下文中解析时，暴露出多处严重的工程隐患。以下为工程评审 Skill `/plan-eng-review` 产出的部分报告副本：

```markdown
### 评审对象: auto_refund_flow_v1
### 评审视角: 工程/架构评审

[WARN] 金融副作用接口缺失幂等键（Idempotency Key）设计。
- 缺陷诊断：方案第 24 行在下游超时处理中，直接调度了第三方退款接口 `stripe.refunds.create`。当网络抖动发生 504 网关超时（Gateway Timeout）时，重试模块（第 32 行）可能会发起二次请求，引发重复退款事故。
- 整改要求：所有涉及资金流出或数据库副作用的外部接口调度，必须在本地持久化创建唯一交易凭证，并在调用第三方接口时强透传基于 `order_id` 与动作序列号哈希生成的唯一幂等键。

[WARN] 未充分考虑高并发状态下的资源竞争（Race Condition）。
- 缺陷诊断：缺少悲观锁机制，当同一用户短时间内快速点击两次退款时，两条并行处理链路可能绕过金额阈值校验，直接触发重复出款。
- 整改要求：在执行金额校验前，必须使用数据库的 `SELECT ... FOR UPDATE` 锁机制或分布式锁（Redis lock）锁定对应的订单状态。
```

这些因方案主写者专注于“让功能跑通”而遗漏的隐性漏洞，在通过多视角物理隔开的上下文审计后被成功捕获。方案根据评审整改要求完成修改与合并，最终锁定入库，为开发提供了无安全隐患的 Spec。

## 典型反模式

* **同一会话内自写自审**：在同一个 Agent 运行对话中让它“再检查一下”，此时它受制于前文的上下文惯性与自我论证倾向，盲区被原样保留。
* **评审意见仅作为日志挂起**：评审报告生成后未反哺到设计文档和 `AGENTS.md` 的刚性约束中，导致 Agent 开发时依旧沿用包含漏洞的初版设计。
* **过度依赖工程单一视角**：仅关注技术细节与接口定义，忽略了业务边界的越线与交互体验的一致性，产出空有高可用性但难用或跑偏的废弃系统。

## 本章要点

* 方案主写者与评审者必须绝对隔离，利用干净的独立上下文或异构模型提供不受偏见影响的客观评审。
* 三视角防线相辅相成：业务看范围、工程看架构与边界漏洞、设计看体验一致性。
* 评审结论必须闭环反哺，修改意见落入方案后锁定，严禁在编码阶段边写边修改基本方向。

> 设计方案经过三视角评审并锁定后，我们还需要一份能把架构取舍和设计决策显性化记录的实体设计文档。我们将在第 9 章 `设计文档先行` 中讨论其构建规范。
