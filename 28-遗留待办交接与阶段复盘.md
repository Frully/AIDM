# 遗留待办交接与阶段复盘

在 AI 驱动开发的语境中，任何一次开发会话（Session）在物理上都是无记忆的。当 Agent 提交完代码并销毁本次 Worktree 时，所有尚未完成的设计遗留项、以及调试期暴露但够不上本分支合并范围的问题，必须被结构化地存储下来。这构成了系统在时间维度上的“遗留待办交接（Deferred Working Memory）”。

**凡推迟到后续迭代的研发任务与技术债，必须编写包含技术背景、拦截边界、依赖前置与推迟理由的“冷启动可接手级”`TODOS.md` 文档。同时，项目团队必须建立阶段性复盘（Post-mortem）机制，将高频出现的逻辑缺陷固化为 CI 里的刚性检查项与 `AGENTS.md` 宪法条目。**

## 延迟记忆的冷启动准则

在大模型参与的敏捷团队中，一个非常普遍的痛点是 TODO 描述的含糊不清。例如：

```markdown
# 错误示范（无法被冷启动接管）
- TODO: 优化退款接口
```

当一个新的 Agent 进程冷启动拿到这行 TODO 时，由于严重缺失技术背景，它根本无从得知究竟是需要优化并发延迟、还是重构参数类型、抑或是过滤敏感日志。

因此，遗留待办交接的记录，必须严格遵循五要素规范：
1. **做什么（Action）**：明确的物理动作。
2. **为什么（Rationale）**：当时的业务痛点或优化背景。
3. **技术边界（Scope）**：被修改的代码文件范围。
4. **前置依赖（Dependencies）**：阻碍其立刻开始的物理条件。
5. **推迟缘由（Deferred Reason）**：为何不在此次合并中一并完成。

### 1. 结构化 TODOS 模板片段

在 Relay 中，我们在根目录下维护一个全局的 [TODOS.md](TODOS.md) 文件，其内容遵循上述格式：

```markdown
<!-- TODOS.md -->
## [DEFER] 自动退款拦截器向 VIP 策略倾斜
* **做什么**：将 `src/refund/decision-engine.ts` 中的额度检查改为按客户历史消费动态派生。
* **为什么**：VIP 用户的退款限额应放宽到 2000 元以改善大客户的满意度，目前固定 1000 元过死。
* **技术边界**：修改 `decision-engine.ts` 并更新 `refund-decision.eval.ts` 里的 VIP 样本评分。
* **前置依赖**：需要先等 CRM 服务导出 `getUserTotalSpent` 的只读 API（`TODOS.md#L45`）。
* **推迟理由**：Stripe 自动退款 v1 刚上线，需要先用固定阈值运行两周以确认稳定性。
```

### 2. Agent 基于 TODOS.md Cold-start 派生新 Plan 的迹线

依靠该标准，后续新拉起的 Agent 会话读到 TODOS 即可在零上下文丢失的情况下直接工作：

```bash
$ npx gstack task-pickup --todo-id VIP_REFUND_UPGRADE

[INFO] Scanning TODOS.md for ID: VIP_REFUND_UPGRADE... Found.
[INFO] Extraction complete. Generating isolated workspace...
[INFO] Checking dependencies: CRM service API. Status: READY.
[INFO] Synthesizing Implementation Plan...

--- Generated Plan Preview ---
1. Read file: src/refund/decision-engine.ts
2. Import client: src/clients/crm-client.ts
3. Update threshold evaluation logic from static 1000 to user-class derived
4. Update VIP eval sample in refund-decision.eval.ts
5. Verify E2E Playwright.

[INFO] Plan successfully generated. Proceeding to execution phase...
```

## 调试挫败细节：被误删的支付通知组件

在 Relay 早期开发时，开发人员在重构账单结算模块时发现“Stripe 异步回调成功后，需要发送一条 Slack 内部通知”的逻辑在本地联调时总是超时。

为了不延误退款核心功能的 PR 合并，Agent 随手在代码中写了一行行内注释 `// TODO: Fix notification` 并把对应的通知模块屏蔽。随后该 PR 绿灯合入了主干。

在下个迭代中，另一个被新派发了“重构订单回流”任务的 Agent 启动了独立 Worktree。由于行内注释 `TODO: Fix notification` 缺乏任何背景和拦截依赖申明，新 Agent 在进行冗余清理时，误认为这是一个由于历史遗留无用的失效逻辑（Dead Code），直接将该发送 Slack 的重要通知组件物理删除并合入了线上。

这导致线上退款成功后通知链路彻底断开，引发了长达三天的财务同步断层。

这起事故彻底痛击了团队，促使团队决定：**严禁在代码中使用任何行内 `TODO` 注释。所有延迟工作必须全部提炼至根目录的 `TODOS.md` 中，并严格遵循五要素模板校验，否则 CI 静态扫描器将视其为语法报错直接拦截。**

## 典型反模式

* **无背景 TODO 垃圾**：大仓代码里塞满了 `// TODO`、`// FIXME` 却没有任何一行解释原因，最终没人知道这些代码是谁留下的、能不能删。
* **无迭代复盘**：每个周期只管往前推代码，从来不总结为什么上一周遇到了三次重复退款报错，导致大模型缺陷防线始终停在初级水准。
* **旧决策悄然删除**：重构了系统的核心架构，但在大宪章 `AGENTS.md` 里只是把旧规则删除并写上新规则，没有任何变更说明（Supersede Log），导致后续 Agent 读取宪法时失去历史演进逻辑。

## 本章要点

* 遗留待办交接是保障无状态 Agent 能够在多线开发中持续交接的关键，必须集中在 `TODOS.md` 中且遵循五要素规范。
* 严禁在代码内部埋设模糊行内 TODO。所有任务必须通过全局记忆清单进行显式登记与依赖阻断。
* 阶段性复盘必须将发生过的错误沉淀为 CI 静态拦截与 `AGENTS.md` 中的宪法限制条目，环境的持续变强才是 AI 驱动下系统长治久安的绝对基石。
