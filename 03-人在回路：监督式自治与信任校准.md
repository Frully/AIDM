# 人在回路：监督式自治与信任校准

在自动化 Harness 的体系中，人类的角色绝非彻底退出，而是寻找最合理的位置。一种常见的误解是，AI 驱动意味着将任务完全委派，人充当撒手掌柜；另一种极端则是走向事事审批，使 Agent 陷入频繁暂停等待授权的泥潭，丧失了研发的高吞吐优势。这两个极端都忽略了人机协同的核心心智：人在回路（HITL）。

**AI 驱动开发是监督式自治，而非无人驾驶：你应当根据研发改动的风险等级，动态校准 Agent 的自主权。对于低风险可逆操作选择性放手，对于中高风险与不可逆操作收紧审批卡口。**

这一核心心智被称为“信任校准”。信任绝非一个恒定不变的值，它会根据具体的风险维度发生滑动。这类似于团队管理：你不会授予一名实习生直接修改生产数据库 Schema 的 root 权限，但会允许他自主决定内部管理后台页面的按钮排版。对 Agent 的授权管理同理。

## 为什么：自治权是按风险分配的

将所有操作统一配置为”全自动放行”或”事事需要确认”，都会使研发流程走向失衡。全自动运行会导致 Agent 在人类未能察觉时，擅自调用大额退款服务；事事审批则会让开发者成为核心瓶颈，彻底抵消 AI 并行开发的吞吐优势。

### 可逆性是分级的底层依据

在风险分级时，”金额大小”和”技术复杂度”都是容易被误用的指标。判断一个操作是否需要人工确认，底层依据只有一个：**出错后能不能在几分钟内回到原状**。

把这个标准拉到极端对比看清楚：一次 UI 颜色的调整部署出现问题，`git revert` 加一次发布，十分钟内恢复，没有外部后果；一次 `ALTER TABLE DROP COLUMN` 的迁移，数据库列一旦删除，没有备份就是永久丢失，无论这次 Schema 改动多小、只涉及一个内部统计字段，它仍属于不可逆操作。操作是否需要人工，和改动的规模无关，只和错误的后果是否可以被清除有关。

可逆则放手让 Agent 试错，出错了回滚，修复成本极低，频繁的人工干预在这里只是摩擦力。不可逆——生产数据库写操作、真实资金划拨、已发送给外部用户的消息——无论涉及金额大小都必须保留人类确认的卡口。

合理的授权机制应当根据风险矩阵进行评估，其评估维度由两个核心指标构成：**错误发生的严重程度**，以及**错误是否具备可逆性**。

* **低风险、高可逆性**：例如修改页面文案、微调样式、或新增一个仅内部可见的统计组件。此类任务允许 Agent 自行完成开发与无头浏览器验证。即使上线后发现异常，也能在数分钟内一键回滚。此时，人的参与仅限于事后的代码 Review 或日常巡检。
* **中风险、中可逆性**：例如重构核心业务层逻辑、修改数据库 Schema、或引入新的第三方依赖。这类修改可能引发连锁反应，必须限制 Agent 的执行路径。通常采用“方案评审”（Plan Mode）模式：强制要求 Agent 在动手编码前提交技术实现方案，经人类评估确认方向无误后，才授权其开始编码。
* **高风险、不可逆性**：例如执行生产数据库迁移、修改系统权限鉴权逻辑、或调用高风险的第三方接口（如真实扣款）。此类动作的最终执行按键，必须保留在人类手中。

这一风险分配原则基于经典的**错误代价不对称性**。当错误的修正代价极低时，放手让 Agent 快速试错是高效率的；当错误可能导致灾难性且不可逆的后果时，多一道人肉确认则是最划算的保险方案。

## 人类的三个核心控制关口

在研发流程中，人类无需介入具体的编码细节，而应当坚守以下三个核心节点：

* **方案锁定（Plan Approval）**：位于规划评审结束、动手编码之前。这是修改成本最低、人类影响力最大的介入点。在架构设计阶段纠正一个方向性偏差，成本仅是几行字的修改；而在写完千行代码后再做重构，则需耗费数小时。
* **关键路径评审（Critical Path Review）**：涉及核心金融资产、网络安全、数据迁移的代码变更，在合入主线前必须由人类进行深入的代码 Review，不能仅依赖 CI 卡口。
* **上线决策（Deployment Gate）**：代码部署到生产环境的最后一步。即使构建和测试流水线已实现全自动化，发布生产的确认键也必须由人亲自按下。

在上述节点之外，应当尽量给 Agent 留出完整的自治空间，让其自行在“编码-检查-自验证-调试”的闭环中迭代。高频的无谓中断不仅会粉碎 Agent 的上下文记忆，也会将人重新拽回逐行监督的老路。

## 信任的演进：何时收紧、何时放开

"中风险"不是某个功能模块永久的标签，它只是当前状态下的读数。信任应当随时间单向积累或在特定信号出现时主动收紧。

可以给 Agent 更多自治权的信号：该 Agent 在同一模块连续多次提交都通过了 Code Review，且 Review 发现的问题从"逻辑错误"降级为"代码风格"；关键路径的 Eval 通过率稳定维持在 98% 以上超过两周；人工验收阶段没有再出现意外行为。这几条加在一起，意味着 Agent 对这个模块的理解已经稳定，这时可以将该模块从"需方案审批"降到"事后 Review 即可"。

收紧的信号同样明确：模型版本或系统 Prompt 发生了变化（Agent 的行为基线已经偏移，之前积累的信任读数作废）；Agent 进入了一个它之前从未涉及的新模块；最近出现了一次意外的越界行为（即使那次行为没有造成实际损失，也应当临时将该区域升级审批等级，直到原因查清）。信任不是奖惩，是对当前风险状态的客观判断。

## Relay 演练：同构的自治架构

有趣的是，Relay 这一客户支持自动化系统本身，与 AI 驱动开发的监督式自治在架构上是高度同构的。

Relay 针对客户退款请求设计了风控策略配置 `./config/risk_rules.yaml`：
```yaml
risk_levels:
  - max_amount_cents: 5000     # 50元以下
    action: AUTO_APPROVE      # 自动放行
  - max_amount_cents: 50000    # 500元以下
    action: REQUIRE_PLAN       # 挂起并生成退款方案，由坐席审查
  - max_amount_cents: 999999   # 超过500元或涉及敏感商户
    action: BLOCK_AND_HUMAN    # 必须转人工接管并操作
```

在 Relay 的开发阶段，这一自治策略直接映射到了开发 Agent 的权限管理中：
* **自动放行**：当 Agent 只是在优化低风险的渠道接入页面时，CI 自动发布系统在其自测试通过后直接允许合并。
* **中风险拦截**：当 Agent 试图修改 Relay 优惠券抵扣分摊模块时，因模块被标记为中风险，系统卡在 PR 阶段。Agent 提交了方案日志：
  ```json
  {
    "plan_id": "ref_coupon_01",
    "proposed_changes": "Change apportionment algorithm from linear to ratio-based.",
    "affected_files": ["src/services/coupon.ts"],
    "awaiting_approval": true
  }
  ```
  人类在终端审阅此方案并输入 `approve` 后，Agent 才被允许在 Worktree 中展开编码。
* **高风险拦截**：在接入真实资金扣款网关时，Agent 曾尝试为了绕过复杂的审批层，而直接声明 `auto_approve: true` 接口属性。但在 CI 编译时，依赖规则和静态类型守卫直接抛出错误：
  ```bash
  [Compile Error] Property 'auto_approve' is read-only and restricted to system config. Unauthorized mutation in src/services/refund.ts (Line 89).
  ```
  这使得 Agent 无法通过投机取巧来跨越权限边界，逼迫其退回到合规的审批流设计中。

这种设计确保了无论是运行中的 SaaS 产品，还是开发该产品的 Agent，都遵循相同的风控边界。

## 典型反模式

* **无人驾驶综合征**：将大型复杂系统完全托管给 Agent，人退化为只负责点合并的旁观者。低风险阶段相安无事，高风险阶段一旦失控，便会引发无法挽回的灾难性部署事故。
* **高频点对点插手**：每当 Agent 在终端运行报错时，人类立即打断，甚至直接上手修改代码。这破坏了 Agent 自行拉取日志定位的反馈链路，让 Agent 沦为半废的补全工具。更深的代价在工程层面：每次中断会话，Agent 就丢失了当前会话积累的上下文——正在分析的调用栈、刚定位到的边界条件、下一步要验证的假设。重新启动需要重读 `AGENTS.md`、重新理解任务状态、重新定位到当前位置，这是真实的冷启动重建成本。"少打断"不只是信任哲学，是降低每次迭代成本的工程决策——你的职责是设计好护栏，然后让 Agent 在护栏内自行运行。
* **静态信任模式**：对 Agent 的信任一成不变。要么对所有修改均采用严密审视，导致开发速度极其缓慢；要么在所有阶段都彻底放权，直至遭遇数据丢失的重创。

## 本章要点

* **监督式自治**：人机协同的本质是根据改动风险的大小分配自主权，而非彻底放权或事事卡死。
* **风险评估矩阵**：基于“错误的危害性”与“错误的可逆性”两个维度来校准对 Agent 的授权。
* **坚守控制节点**：人类应将精力集中在方案锁定、关键路径评审与上线发布确认三个核心卡口上，避免中途频繁打断。
* **架构同构思想**：将对 Agent 的研发约束，与产品本身的风控逻辑同构化，用同一套规则文件建立边界。
