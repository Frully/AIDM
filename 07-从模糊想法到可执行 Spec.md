# 从模糊想法到可执行 Spec

在研发生命周期的起点（Think 阶段），首要目标不是写规范，而是弄清楚你是否在解决正确的问题。一句”为 Relay 接入自动退款”背后可能藏着一个错误的前提：用户真正的痛点可能不是”退款慢”，而是”人工审批占用了大量客服坐席”。如果直接将这句话委派给 Agent 动手，它会顺着字面意图推进，而一旦方向偏离，高吞吐的编码速度反而会带来昂贵的返工代价。

**在写下第一行代码前，必须先经历一轮对问题框架的压力测试：由 Agent 连续抛出六个强制问题挑战初始假设，并在最终的设计文档（Design Doc）中记录问题陈述、已验证假设与方案取舍。这并非技术规范，而是 Plan 阶段多视角评审的直接输入。**

设计文档不是应付流程的静态文本。它是从模糊想法到工程落地的唯一桥梁：Plan 阶段基于它进行三视角评审并精炼为工程 Spec；Build 阶段据此落实边界约束；Review 阶段用它检验代码是否偏离原始方向。一旦设计文档模糊，后续所有的门禁和拦截都将失去判据。

## 在 AI 驱动开发下，“先想清楚”的工程杠杆

在 AI 驱动开发的语境下，“先想清楚”的工程重要性被显著放大。

由于 Agent 编写代码的速度极快，你给出一句语义含糊的粗糙指令，它几分钟内便能拼凑出上千行功能看似健全、实则背离核心边界的代码。人类团队开发中“先写写看”的低约束模式之所以可行，是因为人类编码速度慢，偏离方向时能够被及时察觉并纠错；而 Agent 的高吞吐特性会导致错误逻辑在极短时间内迅速膨胀，拉高未来的重构成本。

通常，一个模糊的业务意图在工程层面至少遗留了三类未决要素：
* **隐藏的技术需求**：例如“自动退款”背后隐藏着对“订单数据一致性”、“退款记录审计流水”与“冲正回滚逻辑”的需求。
* **边界安全指标**：例如资金划扣的拦截水位、风控黑名单处理策略。
* **架构方案选型与权衡**：例如到底是由 AI 直接判断并调用扣款，还是由 AI 生成建议，交由底层的规则引擎过滤执行。

Think 阶段的核心工作，就是通过结构化的追问将这三类未决要素全部浮出水面，并在设计文档中记录清楚——让 Plan 阶段的评审者能据此做出有依据的取舍决策，而不是凭猜测锁定方案。

## 六个强制问题：Think 的追问骨架

Think 阶段的追问不是随机的，而是由六个强制问题构成的骨架。每一个问题都是一把手术刀，专门切除某一类常见的思维幻觉：

* **需求证据**：是否有真实用户在为此付费、抱怨或流失？必须引用具体数据，拒绝“我觉得用户需要”之类的感性描述（用于戳破“想当然”的需求幻觉）。
* **现状痛点**：用户今天如何解决该问题？现状的摩擦力有多大？如果不做，会有什么实质损失（用于戳破“问题不够严重”的轻视幻觉）。
* **精准用户画像**：最迫切的种子用户是谁？拒绝泛泛的“所有商户”，必须明确为“日均处理 50 笔以上退款工单、且无力扩充客服编制的中型 B 端商户”（用于戳破“服务所有人等于不服务任何人”的泛化幻觉）。
* **最小可行切入点**：能在一周内上线并解决核心摩擦力的最窄切口是什么（用于戳破“必须一次性做完”的完美主义幻觉）。
* **独特观察**：为什么选择在当下解决这个问题？有什么旁人未曾注意到的技术或业务红利（用于戳破“随时都可以做”的时机幻觉）。
* **长期适配性**：该方案在三年后是否依然成立？大模型能力的跃升或外部 API 的变化会让它增强还是过时（用于戳破“短期有效等于永远有效”的惯性幻觉）。

六问结束，Agent 汇总答案并主动挑战你的初始框架。如果框架经得住追问，则进入方案对比；如果经不住，则先修正框架，再重新确认。

## gstack 参考实现：`/office-hours`

在 gstack 的参考实现中，Think 阶段的核心载体是 `/office-hours` 指令，分两种模式：

* **Startup 模式**：面向新产品或新方向的探索。Agent 以 YC Office Hours 风格介入，依次跑完六个强制问题，在每一问结束前挑战你的回答，不允许含糊作答。问询结束后，Agent 会主动陈述“你说的是 X，但你描述的实际上是 Y”，帮助你看清自己真正在解决什么。最终产出设计文档并自动保存为本地 `.gstack/projects/{repo}/{branch}-design.md`，后续的 `/plan-ceo-review` 与 `/plan-eng-review` 命令会自动读取该文档。
* **`Startup` 模式**：面向新产品或新方向的探索。Agent 以 YC Office Hours 风格介入，依次跑完六个强制问题，在每一问结束前挑战你的回答，不允许含糊作答。问询结束后，Agent 会主动陈述“你说的是 X，但你描述的实际上是 Y”，帮助你看清自己真正在解决什么。最终产出设计文档并自动保存为本地 `.gstack/projects/{repo}/{branch}-design.md`，后续的 `/plan-ceo-review` 与 `/plan-eng-review` 命令会自动读取该文档。
* **`Builder` 模式**：面向小项目、Hackathon 或已有产品的新特性快速验证。节奏更轻，跳过需求证伪，直接聚焦“这个想法酷在哪、怎么实现最快”。

即使不使用 gstack，这一理念同样通用：在动手前开一个专门的问询会话，由 Agent 扮演质疑者，按六个强制问题逐一追问，人类负责诚实回答，最后将结果整理为 `docs/design/` 下的设计文档。

## 主线演练：Relay 退款特性的 Think 问询过程

Relay 团队最初的构想只有一句话：“我们需要接入自动退款功能。”Agent 以 `/office-hours` 的 `Startup` 模式介入，引导团队回答上述问题：

> **Agent**：你有没有真实数据证明这是一个值得现在解决的问题？  
> **团队**：有。我们的客服数据显示，每月 12,000+ 退款工单中，65% 是金额小于 5,000 分的小额退款，每笔人工处理耗时约 8 分钟。  
>
> **Agent**：用户今天是怎么处理这件事的？  
> **团队**：完全人工。客服查单、判断条件、手动操作网关，全程没有自动化。  
>
> **Agent**：你说"自动退款"，但你描述的核心痛点是"人工审批占用了客服 60% 的时间"。你真正想解决的是退款慢，还是客服资源被退款工单绑架？

经过这一轮追问，问题框架从"接入退款功能"收窄为"将小额退款从人工工单队列中自动分流"。方案方向也因此明确：不是"让 AI 直接决策并执行退款"（非确定性输出，资金安全无法保证），而是"AI 意图分类 + 硬规则风控兜底"。

问询结束后，Agent 产出设计文档 `docs/design/refund-v1.md`：

```markdown
# Design: Relay 自动退款 v1

Status: DRAFT | Branch: feat/auto-refund

## Problem Statement
Relay 商户的客服坐席平均每天处理 80+ 退款工单，其中 65% 为小额退款（< 5,000 分），
每笔人工处理耗时约 8 分钟，严重挤占高价值工单的响应资源。

## Demand Evidence
- 月退款工单量 12,000+，自动化率 0%
- 客服访谈（3 家商户）：退款工单占客服工时 50–70%

## Premises Challenged
- ❌ 初始框架："我们需要一个退款功能"
- ✅ 修正框架："我们需要把小额退款从人工工单队列里彻底移走"

## Approaches Considered
### 方案 A：LLM 全权决策并执行退款
LLM 判断是否退款、金额是否合规并直接调用网关。风险：模型不确定性无法在工程上
保证 100% 不越权划扣，资金安全无法闭合。否决。

### 方案 B：LLM 意图分类 + 硬编码规则引擎兜底（推荐）
AI 解析意图并判断类别，规则引擎做金额卡口与黑名单拦截，最终执行由规则决定。
可控、可审计、可回滚。采纳。

## Success Criteria
- 小额退款自动化率 ≥ 90%
- 误退款率 < 0.1%
- 平均处理时间 < 30 秒（从现状 8 分钟）
```

这份设计文档是 Plan 阶段的直接输入：CEO 视角审查其业务范围与价值取舍；工程视角将其精炼为带边界参数的技术 Spec（如具体的金额阈值、接口签名）；设计视角检验交互流程的合理性。Think 阶段在此处收尾：确立问题边界与解决路径，后续工作转交至 Plan 阶段。

## 典型反模式

* **跳过 Think 直接写技术规范**：拿到一句模糊需求直接让 Agent 写接口定义和边界参数。问题框架一旦偏离，后续所有技术工作都建立在错误的地基上，技术规范写得再精确也是沉没成本。
* **Think 走过场，不真正追问**：走完六问但每一问都浅尝辄止，用“用户想要更好的体验”替代真实数据，用“我们会处理边界情况”替代具体的约束定义。走过场的 Think 产出的设计文档和没有 Think 的效果一样。
* **把 Think 当架构设计**：在 Think 阶段就开始讨论接口命名、数据库字段和并发策略。Think 阶段只问”解决什么问题”和”走哪条路”，架构细节属于 Plan，不要提前消耗。

## 本章要点

* **Think 的核心产出**：设计文档（Design Doc）。它记录问题边界、已验证假设、方案对比与推荐路径，是 Plan 阶段多视角评审的直接输入，而非技术 Spec。
* **速度放大了风险**：Agent 高吞吐特质使得方向错误的代价极高，先把问题框架弄对，比先把代码写快更重要。
* **六问是结构，不是清单**：每一问针对一类常见思维幻觉，走完六问的目的是发现“你以为你在解决 X，实际上需要解决的是 Y”。
* **Think 不做架构**：问题边界和方案选择属于 Think；接口定义、边界参数、数据流设计属于 Plan。两者不要混。
