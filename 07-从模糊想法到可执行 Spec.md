# 从模糊想法到可执行 Spec

在研发生命周期的起点阶段（Think 阶段），首要目标是将模糊的产品意图具象化为确定的规范。例如，一句简单的“为 Relay 接入自动退款”背后隐藏着大量未明确的非功能性指标与边界逻辑。如果直接将此任务委派给 Agent，它会顺着字面意图自行猜测补全，而一旦猜测偏离预期，高吞吐的编码速度反而会带来昂贵的返工代价。因此，我们必须在动手前，将想法淬炼为一份没有歧义的 Spec 需求文档。

**在编码开工前，必须将模糊的业务想法固化为一份具备“冷启动可执行性”的 Spec 需求文档。其研判标准是：一个不具备任何前情背景的 Agent 读完此文档后，能够直接展开设计与编码，且无需中断并向人类提问。**

Spec 绝非应付流程的静态文本，而是 Think 阶段的交付物，作为后续 Plan 阶段多视角评审的标的，以及 Build 阶段并发编码的边界输入。Spec 逻辑含糊，后续的开发与测试卡口均会失去基准。

## 在 AI 驱动开发下，“先想清楚”的工程杠杆

在 AI 驱动开发的语境下，“先想清楚”的工程重要性被显著放大。

由于 Agent 编写代码的速度极快，你给出一句语义含糊的粗糙指令，它几分钟内便能拼凑出上千行功能看似健全、实则背离核心边界的代码。人类团队开发中“先写写看”的低约束模式之所以可行，是因为人类编码速度慢，偏离方向时能够被及时察觉并纠错；而 Agent 的高吞吐特性会导致错误逻辑在极短时间内迅速膨胀，拉高未来的重构成本。

通常，一个模糊的业务意图在工程层面至少遗留了三类未决要素：
* **隐藏的技术需求**：例如“自动退款”背后隐藏着对“订单数据一致性”、“退款记录审计流水”与“冲正回滚逻辑”的需求。
* **边界安全指标**：例如资金划扣的拦截水位、风控黑名单处理策略。
* **架构方案选型与权衡**：例如到底是由 AI 直接判断并调用扣款，还是由 AI 生成建议，交由底层的规则引擎规则过滤执行。

Think 阶段的核心工作，就是将这三类未决要素全部挖掘、对齐并固化。

## 需求淬炼：挑战意图的追问矩阵

将业务想法转化为工程 Spec，需要采取批判性思维，通过一组追问矩阵去挑战初始假设：

* **问题对准度**：此特性旨在解决什么核心痛点？“自动退款”是实现手段，其解决的痛点是“小额客诉退款严重挤占人工坐席的客服时间”。如果痛点定位错误，做出来再健壮的功能也是沉没成本。
* **隐藏能力挖掘**：为了达成该目标，系统需要配合哪些不易被察觉的基础能力？（如为了安全退款，是否必须联动网关查询商户的账户可用余额？）
* **边界雷区锁定**：系统在哪些边界场景下应当被物理阻断？（如订单状态异常、退款金额超过阈值、商户信用等级过低等）。
* **架构方案取舍**：我们备选了哪几条通路？各自在安全性、吞吐率与开发成本上的代价是什么？最终选择哪一个？

一旦上述问题有了确切的工程数值与逻辑表达，即可沉淀为 Spec 文件。若一个新加入的分支 Agent 读完 Spec 后，依然需要中断执行并向人类确认“大额退款的具体上限是多少”，则说明该 Spec 的信息密度尚未达到合格标准。

## gstack 的需求与 Spec 辅助生成设计

在 gstack 的参考实现中，Think 阶段的追问与打磨被转化为固化的快捷指令：

* **交互式需求生成**：`/office-hours` 指令会自动触发一组结构化的追问矩阵，通过多轮交互挑战你的业务假设，挖出系统边界与隐含架构要求，最终导出可作为 Plan 输入的 Spec 初稿。
* **歧义度检查与收敛**：`/spec` 命令对 Spec 文本进行结构化梳理，自动识别文档中的模糊语意（如“响应时间要快”、“确保安全”等非量化表述），强迫人类开发者将其收缩定义为精确的工程参数（如“API 响应时长不得超过 200ms”、“拦截超出 100 元的划扣”），直到文件歧义度收敛至系统安全线以下。

即使不引入 gstack，该理念依旧通用：由人类开发者在 `docs/specs/` 下手动编写 Spec 文件，坚持不放过任何含糊的形容词，将其全部转译为确定性的数值和强类型状态。

## 主线演练：Relay 退款特性的 Spec 规范设计

以 Relay 为例，我们来看一份能让 Agent 顺利冷启动的 Spec 文档（`docs/specs/auto_refund_spec.md`）应当具备哪些细节：

```markdown
# Relay 自动退款特性 Spec (docs/specs/auto_refund_spec.md)

## 1. 核心诉求与待解问题
- 背景：目前 80% 的客诉均为单笔金额小于 5,000 分（50 元）的小额退款。手动审批耗费了客服 60% 的工作时间。
- 目标：将符合特定小额场景的退款流程完全自动化。

## 2. 核心功能能力（Capabilities）
- 订单检索：在退款前，必须调用 `billing_service.queryOrder` 获取订单状态，确认状态为 `SUCCESS` 且未结清。
- 退款执行：调用网关扣减 `billing_service.executeRefund`，并将流水记录写入 `refund_records`。
- 幂等性：网关请求头中强制包含 `Idempotency-Key`。

## 3. 约束与安全边界（Boundaries）
- 金额硬卡口：单笔自动退款限额 `RefundLimit <= 5,000` 分。超出此阈值必须调用 `createApprovalRequest`，挂起至人工审批队列。
- 状态硬拦截：若订单在 90 天前创建，或订单账户被风控标记为 `SUSPICIOUS`，一律拦截并流转至人工审查。

## 4. 选定架构方案与舍弃对比（Trade-offs）
- 方案选择：AI 意图解析分类 + 硬编码风控规则引擎兜底。
- 舍弃方案：完全依靠 LLM 决策退款。
- 舍弃理由：模型输出具有非确定性，无法在工程上保证 100% 不越权划扣。
```

当开发 Agent 读取了这一份 Spec 进行编码时，由于其本地单测覆盖了‘历史可疑订单退款拦截’场景，Agent 尝试编写了一个测试用例。测试运行迹线如下：
```bash
$ npm run test:refund
✖ test_suspicious_order_should_block
  Expected: Refund blocked and ApprovalRequest created.
  Actual: Refund executed successfully (Transaction ID: ref_10023).
```
Agent 在读取测试失败后，分析代码，发现自己在编写 `shouldEscalate()` 风控拦截函数时，遗漏了订单状态标志 `OrderFlag.SUSPICIOUS` 的逻辑判断。正是由于 Spec 在 `3. 约束与安全边界` 中给出了极高精度的参数列表与状态标识说明，Agent 得以在半分钟内修复逻辑，补充了缺失的判定规则，最终使得自验证测试全绿通过。

这样的一份 Spec，彻底消除了模糊意图带来的试错成本。

## 典型反模式

* **直接以业务意图替代 Spec**：仅向 Agent 输入一行“帮我实现自动退款”便开始编码。这等于让 Agent 自行设计资金安全红线，几乎必然会导致严重的逻辑偏差和后续的大面积返工。
* **Spec 充斥愿景与口号**：在 Spec 文档中写入诸如“提供极致的用户退款体验”或“系统需要稳定可靠”等无法工程度量的词语。Agent 无法据此实现自动化门禁拦截。
* **闭门造车缺乏判据**：写完 Spec 后自己草草看过便宣布锁定，不组织跨视角评审，也不以“新 Agent 能否冷启动直接编码”作为硬性研判指标。

## 本章要点

* **Think 的核心产出**：Spec 是系统编码的起点，必须达到冷启动可执行、无歧义的工程标准。
* **速度放大了风险**：Agent 的高吞吐特质使得先想清楚的杠杆更大，错误方向的代码累积速度是极快的。
* **追问挑战意图**：运用追问矩阵，挑战产品想法中的隐藏需求、安全边界与方案取舍，将其转化为可量化指标。
* **远离虚空描述**：Spec 必须落到功能能力、安全边界与架构理由上，禁用任何不可度量的愿景词汇。
