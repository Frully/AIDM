# 并行从哪来：多开对话与两层隔离

在 AI 驱动开发的工程生命周期中，最大的效能吞吐红利来源于并发开发。大模型 Agent 可以在极短时间内同时并发处理多个互不相关的需求特性或缺陷修补。但要释放这一高吞吐红利，我们必须解决多条开发线之间物理资源冲突的瓶颈。

**并发开发能力来源于桌面端可同时开启多个独立对话会话，但要确保并发任务互不串台，必须同时落实两层隔离：依靠 Git Worktree 隔离静态代码分支，以及依靠会话钩子与编排脚本在运行时自动隔离端口和沙箱数据库等动态环境。**

如果仅仅做了代码层面的物理隔离，多个并发 Agent 在运行时仍然会因为争抢同一个网络端口或向同一个数据库灌入脏数据而导致严重的并发验证冲突。

## 多开对话：并行的能力来源

在实际协同研发中，如 `Claude Code` 这类桌面端或 CLI 客户端能够同时开启多个独立的对话会话。由于大模型的上下文具备独立的会话生存周期，每一个独立的会话即可被视为一个独立工作的虚拟 Agent 队友。

我们可以让会话 A 负责开发自动退款判定逻辑，会话 B 负责重构多渠道分类路由，会话 C 同时去定位坐席审批台的一个前端 UI 缺陷。三条线在时间线上是完全并行的。

但是，若三条并行线指向同一个物理工作目录和本地运行栈，灾难将无法避免：
* 静态代码层面，A 和 B 都在改动同一个文件，本地工作区代码会互相覆盖，导致 Git 提交冲突。
* 动态运行时，A 启动的测试服务占用了 3000 端口，B 的服务便无法启动；A 运行单元测试时向本地数据库写入了测试数据，这笔脏数据会污染 C 的端到端自验证。

因此，多线开发的前提是必须对 A、B、C 实施在静态与动态两个维度上的绝对物理隔离。

## 双层隔离屏障

### 第一层：Worktree 隔离静态代码
第一层隔离在物理文件系统上割开各个 Agent 的代码修改范围，这通过 `git worktree` 机制实现。

相比于传统的频繁切换 `git checkout`，`git worktree` 允许我们将同一个 Git 仓库的多个不同分支同时检出到不同的本地物理文件夹中。每个文件夹拥有自己独立的工作区和分支，但共享同一套底层的 Git 历史注册表。

这一层物理隔离确保了 Agent 之间“编辑文件互不踩脚”，每个 Agent 只能看到并修改自己所属 Worktree 文件夹下的代码。

### 第二层：动态环境的自动装配
静态隔离只是完成了第一步。如果我们将并发的本地栈服务依然绑定在同一个全局端口（如 3000）或同一个本地数据库上，运行时冲突依然会发生。

---

因此，第二层隔离必须由自动化脚本和生命周期钩子（如 `SessionStart`）承载。当一个新的 Worktree 开启时，脚本必须自动为该目录分配一个当前系统空闲的端口号（如 3001、3002），并在本地自动 provision 一个与该 Worktree 名称绑定的临时测试数据库（如 `relay_wt_refund_01`）。

只有当静态代码（Worktree）与动态环境（独立端口与数据库）双层防护都配置就绪，并发的 Agent 才能井水不犯河水，真正独立地在本地执行开发、测试与浏览器自验证。

## 主线演练：Relay 多线并发目录树与环境配置

在 Relay 示例项目的开发中，工程师同时分发了三个并发原子任务。在开发机上，通过 Worktree 机制生成了如下并行的物理目录树拓扑：

```bash
$ git worktree list
/path/to/project/AIDM/main-repo        main-branch [main]
/path/to/project/AIDM/wt-refund-01     RELAY_REFUND_01 [feat-refund-decision]
/path/to/project/AIDM/wt-routing-02    RELAY_ROUTE_02 [feat-classification]
```

当两个独立的 Agent 进程分别在 `wt-refund-01` 与 `wt-routing-02` 中开工并加载本地环境时，会话启动钩子自动在各自目录的根下生成了相互隔离的环境变量配置文件：

```ini
# /path/to/project/AIDM/wt-refund-01/.env
PORT=3001
DATABASE_URL="postgresql://localhost/relay_sandbox_wt_refund_01"
REDIS_URL="redis://localhost:6379/1"

# /path/to/project/AIDM/wt-routing-02/.env
PORT=3002
DATABASE_URL="postgresql://localhost/relay_sandbox_wt_routing_02"
REDIS_URL="redis://localhost:6379/2"
```

通过这一层动态环境装配，在 `wt-refund-01` 中工作的退款 Agent 能够自由地起服务、运行测试、向沙箱数据库写入特定退款状态的测试记录，并在本地开发时保证隔离度。这保证了并发吞吐量最大化的释放。

## 典型反模式

* **只隔离静态代码，共享动态资源**：使用了 `git worktree`，但没有自动化端口和数据库分配，导致多线运行时因为争抢 3000 端口或连同一个数据库而频繁报错。
* **物理目录隔离全靠手动配置**：每次开辟新分支并行开发时，需要开发人员手动复制目录、手动修改环境变量中的端口号以及手动创建临时数据库，繁琐的人工流程导致团队重新退回到单线串行模式。
* **无规则并发多开**：在未梳理任务依赖关系、任务边界存在大面积物理交叠的情况下硬性多开，导致多个 Agent 在不同的分支上修改相同的公共文件，在最后的 PR 合并阶段产生无法解决的逻辑冲突。

## 本章要点

* 并行吞吐红利来自客户端多开会话，每个独立会话扮演一个并发 Agent 的角色。
* 隔离必须做到两层：`git worktree` 物理隔离静态代码，自动化脚本动态分配独立端口、独立数据库和进程缓存。
* 动态隔离必须做到完全自动化（即一键开辟 Worktree 并完成环境装配），否则繁琐的手动过程会摧毁并行的效率优势。

> 双层隔离解决了多线开发“互不打架”的问题。但在高并发多线运行时，若缺乏严明的编排纪律，大仓的分支系统和陈旧的 Worktree 目录很快就会失控。我们将在第 15 章 `多线并行的开发纪律与环境清理` 中讨论多线并发的管理与清理策略。
