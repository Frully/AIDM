# Agent 自验证与业务验收

代码在静态层面通过了严格的 Review 与安全审计，这保障了其结构与质量的纯净。然而，静态检查的绿灯并不能等同于用户视角的“行为正确性”。尤其在复杂的 Web 系统中，API 接口类型的匹配无法拦截前端表单渲染错误、交互死锁等只能由“人眼观察”才能捕获的界面逻辑缺陷。

**交付产物必须由 Agent 亲自驱动浏览器跑通端到端自验证，并在 Spec 阶段约定的多层业务 KPI 达标后方可放行。** 未经过 Agent 亲自模拟真实用户行为进行浏览器端到端自验证的产出，决不允许声明为完成（DoD）。技术指标全绿不等于业务有效，如果不进行模拟真实交互的自验证，极易发生前端渲染死锁等静默故障。

## 完成定义（DoD）中的自验证铁律

为防止 Agent 投机取巧，项目的“完成定义”必须强制嵌入端到端自验证流程：

```
+--------------------------------------------------------+
|                 DoD (完成定义) 检查清单                  |
| 1. [x] 静态编译与 Linter 规则无警告                     |
| 2. [x] 自动化单元测试与集成测试 100% 通过                |
| 3. [x] 核心 LLM 评测套件无退化                          |
| 4. [x] 独立上下文代码评审与安全 CSO 审计通过             |
| 5. [x] Agent 运行 Playwright 模拟用户进行端到端跑通并留存截图 |
+--------------------------------------------------------+
```

Agent 不能只给出“我修改了文件”的口头声明，必须呈递物理级的证据——包含浏览器操作轨迹的终端迹线与运行截图。

## 端到端自动化运行机制

在自验证阶段，Agent 需要调用自动化测试框架（如 `Playwright` 或 `Puppeteer`）真实拉起无头（Headless）或有头浏览器，在本地沙箱数据库装配就绪后（参见第六部分），模拟客户进行真实点击与文本输入，以此检验整个链路的顺畅度。

## 调试挫败细节：消失的前端“元”与“分”之差

在重构 Relay 客服后台审批表单时，Agent 修改了后端的退款 API 结构，将原本返回的浮点数金额字段统一规范化为了数据库常用的“整数分（Cents）”。这一变动不仅让所有的后端 API 单元测试跑通，而且类型系统也顺畅对齐。

然而，当 Agent 在本地拉起 Playwright 进行端到端自验证时，截图对比工具报出了红色错误。

原来，由于前端模板中漏掉了针对新分字段的 `/100` 格式化过滤器，导致原本一笔 1200 元的退款请求，在坐席审核大屏上被赫然渲染显示为了 `120000.00 元`。

任何普通的后端逻辑检查都绝对抓不住这个显示层面的漏洞。如果直接交付上线，将会引发客服人员的巨大恐慌，甚至导致资金的误批。正是因为 DoD 中强加了“Agent 必须拉起真实浏览器并核对截图”这一物理红线，该缺陷在本地交付前最后一刻被强行按下，并由 Agent 自动追加了 `/100` 格式化器予以修复。

## AI 产品的验收困境

普通功能验收相对机械：给定输入，输出等于预期值，断言通过，合并。AI 产品的验收要复杂得多——输出”在可接受范围内”，且”整体上足够好”，这两句话藏着无数陷阱。

Relay 上线前，分类模型在测试集上的总体准确率达到了 92%，工程团队认为达标。上线后第三天，客服主管注意到退款工单队列几乎是空的，但退款相关的用户投诉量却在上涨。调查后发现：92% 的总体准确率掩盖了一个结构性问题——95% 的分类错误集中在”退款意图被错误归类为订单查询”这一子类，而退款恰好是 Relay 流量最高的操作类型。被错误分类的退款对话进了普通查询队列，坐席不知道要主动处理，对话就这样静默堆积。

紧急回滚后，团队重新翻看 Eval 报告才意识到：报告里只有一列总体准确率，没有按意图子类拆分的分层统计。高风险子类的失误被淹在了整体数字里。

这个教训说明，AI 产品的验收门槛必须分层设计：总体指标是健康度的粗筛，高风险子类的专项指标才是上线的真实门槛。

## 业务验收 KPI 的定义方式

KPI 的定义工作应当在 Spec 阶段完成，和业务方对齐，写进需求文档，而不是在上线前临时凑数。等到代码写完再问”怎么算成功”，业务方的答案会受到已有结果的影响——这是一种隐性的预期降格。

Relay v1.0 的业务验收指标模板如下。三类 KPI 性质不同，处理方式也不同：

```markdown
## Relay v1.0 业务验收指标（Spec 阶段定义，上线前确认）

### 准入 KPI（未达标不上线，硬门槛）
- 退款意图识别准确率 ≥ 98%（专项测试退款子类，不是总体准确率）
- 误触发退款（用户未明确表达退款意图，AI 自动执行退款）发生率 = 0（测试集中）
- 高风险转人工的召回率 ≥ 99.5%（不能漏掉任何需要人工审核的请求）

### 质量 KPI（上线后持续监控，低于阈值触发优化 sprint）
- 自动解决率 ≥ 60%（无需人工介入即完成的对话比例）
- 平均对话轮数 ≤ 3 轮（AI 确认一次属正常，追问 5 次说明 prompt 逻辑不清晰）
- 用户满意度评分 ≥ 4.0/5.0（上线后第 1 周随机采样）

### 成本 KPI（治理上限，超出触发架构评审）
- 每已解决对话的模型费用 ≤ $0.15
- 单次对话 token 消耗 P95 ≤ 20000
```

准入 KPI 是合同性的——没达到，不管工期多紧，不上线。质量 KPI 是监控性的——低于阈值时触发优化迭代，但不强制立刻下线（除非降幅触发了业务 SLA）。成本 KPI 防止 AI 功能陷入”技术上能跑、成本上跑不起”的困境，一旦触发上限，需要架构层面的介入，不是调几个参数能解决的。

### Relay 的教训：退款子类专项指标如何写进 Spec

上线事故发生后，团队在 Spec 文档里新增了一节”高风险子类专项指标”，强制要求每次 Eval 报告必须输出按意图子类分层的通过率矩阵，而不只是总体数字：

```
Eval 报告示例（事故修复后格式）

意图子类           测试样本数   通过率   备注
-----------        ----------   ------   ----
退款               480          98.5%    准入门槛 ≥ 98%，当前达标
订单查询            320          99.1%    
账户问题            150          97.3%    低于质量目标，下个 sprint 优化
总体               950          98.5%    （此行仅供参考，不作为上线依据）
```

这个格式变化看似简单，实际上切断了”用总体数字掩盖子类问题”的路径。CI 里的 Eval 脚本现在会显式断言每个高风险子类的通过率，任何一列低于准入阈值，流水线整体标红。

## AI 功能的 DoD 清单（扩充版）

本章开头给出的 DoD 框架覆盖了技术层面的自验证。AI 功能还需要叠加业务层面的人工确认，缺一不可：

```markdown
## AI 功能 DoD 清单（每次 AI 功能发布前执行）

### Agent 自验证（CI 自动运行，全部通过才允许进入人工验收）
- [ ] Eval 套件通过率 ≥ 准入阈值（阈值从 Spec 里取，不在代码里硬编码）
- [ ] 高风险子类专项 Eval 通过率达标（退款子类 ≥ 98%，高风险转人工召回率 ≥ 99.5%）
- [ ] 对抗性 Eval（prompt injection 场景）全部通过
- [ ] 单元测试和集成测试全部通过
- [ ] ESLint 通过，无 console.log
- [ ] 日志安全扫描通过（无 PII 字段出现在日志输出中）
- [ ] 所有 AI 决策点已挂载监控埋点（`callLLM` 调用包含 `traceId`）
- [ ] Playwright 端到端自验证通过，截图和终端迹线已归档

### 人工验收（发布前最后确认，由产品经理或业务方执行）
- [ ] 产品经理在 staging 环境验证了退款、高风险转人工两个核心场景
- [ ] staging 环境的 Eval 报告已确认准入 KPI 全部达标
- [ ] 回滚方案已验证：可在 5 分钟内回滚到上一个稳定版本
- [ ] 监控告警规则已配置（token 消耗异常、误触发退款、分类失败率告警）
```

Agent 自验证部分由 CI 在每次 PR 合并前自动执行；人工验收部分需要真人在发布前逐条确认并签字。Agent 无法替代人工验收，因为”staging 环境的核心场景是否符合业务预期”这个判断依赖的是业务经验，而不是技术指标。

### 没有 DoD 清单的代价

Relay 的第一次迭代没有这份清单。工程师在本地跑通了所有测试，把 PR 合并进了主分支。上线前，产品经理本来想在 staging 手动点一下核心退款场景，但当天另有一个紧急需求，这一步被跳过了。

上线三小时后，监控大盘显示退款队列堆积量急剧上升。排查后发现：staging 和生产环境的 LLM 模型版本不一致（staging 用的是旧版），导致分类行为差异在 staging 上根本无法暴露。这个问题如果在上线前的人工验收步骤里检查过”staging 与生产环境的模型版本是否对齐”，会在当天就被捕获。

紧急下线、回滚、重新对齐环境配置，耗时约 4 小时。事后团队在 DoD 清单里增加了一条：`[ ] staging 与生产的模型版本已确认一致（model_id 和 revision 均对齐）`。这一条后来触发了两次预防性修复。

## gstack 参考实现

在 gstack 体系中，端到端自验证通常由命令行中的 `/qa` 技能承载。该指令会在隔离的 worktree 动态沙箱中拉起预配置的 Playwright 或 Puppeteer 测试套件：

```bash
# 由 Agent 执行以触发自验证的指令
$ gstack qa --visual-check
```

该工具从 `./playwright.config.ts` 读取配置，调起无头浏览器实例，记录执行轨迹，并保存视觉快照。在自验证失败或出现视觉漂移时，生成的日志和截图将直接返回到 Agent 的上下文中以便其自愈修复。测试框架本身是可替换的实现层，核心在于必须在代码合并前强制执行端到端自验证。

## 典型反模式

* **测试通过即完成**：CI 跑通后直接宣布”完工”，不进行任何真实的端到端行为演练，导致界面白屏、CSS 漂移等显性 bug 一路绿灯滑向生产。
* **自验证缺少物证**：放任 Agent 给出”我已在本地用浏览器手动点过了”的模糊描述，缺少运行截图、网络调用 HAR 归档等不可篡改的交付物证。
* **用总体指标掩盖高风险子类**：Eval 报告只看整体通过率，高风险意图子类的失误被均值稀释，上线后才在业务数据里暴露。
* **KPI 在上线前才定义**：业务方看到初步结果后调低预期，技术指标勉强达标，上线后却无法满足真实业务需求。

## 本章要点

* 检查通过不等于行为正确。DoD 必须同时包含技术自验证（Playwright 端到端、Eval 通过率）和业务人工验收（核心场景确认、KPI 达标确认）两层。
* AI 产品的业务 KPI 必须在 Spec 阶段定义并与业务方对齐，并且要按高风险子类分层设计，总体准确率不能作为上线的唯一依据。
* 准入 KPI（如退款子类准确率 ≥ 98%）是硬门槛，质量 KPI（如自动解决率 ≥ 60%）是上线后的监控目标，成本 KPI（如每对话费用 ≤ $0.15）是架构约束，三类 KPI 的处理方式不同。
* Agent 自验证输出的终端迹线与截图是物理凭证；人工验收的签字是业务授权，两者都不能省略。

> 自验证和业务 KPI 对齐，完成了交付的最后一米。接下来，当系统真正要从本地 worktree 部署推向生产环境，我们将在下一章探讨可逆性在发布阶段的设计——数据库迁移与版本灰度的安全防线。
