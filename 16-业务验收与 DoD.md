# 业务验收与 DoD

代码在静态层面通过了评审与安全审计，表明其结构干净。但静态绿灯不等同于用户视角的“行为正确”：静态依赖分析和单元测试无法拦截前端表单渲染错位、交互死锁这类只有“拉起浏览器实际跑一遍”才能捕获的逻辑漏洞。验证阶梯走到这一阶，必须将判定权从单纯的机器二元检查，递交给能够模拟用户行为的 Agent 探索和有最终决定权的业务负责人。

**交付物必须由 Agent 亲自驱动浏览器跑通端到端自验证，并在 Spec 阶段约定的通用业务验收指标达标后才允许放行。** 未经 Agent 模拟真实用户行为进行浏览器端到端自验证的产出，严禁声明为完成（DoD）；技术指标全绿不等同于业务有效。


## 完成定义（DoD）中的自验证铁律

为了防止 Agent 在交付时投机取巧，项目“完成定义”（DoD）必须强制嵌入物理级别的端到端自验证：

```
+--------------------------------------------------------+
|                 DoD (完成定义) 检查清单                  |
| 1. [x] 静态编译与 Linter 规则无警告                     |
| 2. [x] 单元测试与集成测试 100% 通过                      |
| 3. [x] 关键路径黄金测试与对抗性安全测试通过              |
| 4. [x] 独立上下文代码评审与安全审计通过                  |
| 5. [x] Agent 运行 Playwright 模拟用户端到端跑通并留截图  |
+--------------------------------------------------------+
```

Agent 不能仅呈递“我修改了文件并测试通过”的口头声明，必须提交包含浏览器操作轨迹的终端迹线与运行截图作为客观交付凭证。


## 端到端自动化运行机制

在自验证阶段，Agent 调用自动化测试框架（如 `Playwright`）真实拉起浏览器，在隔离的沙箱数据库装配就绪后，模拟客户进行真实点击与输入，检验整个交互链路的顺畅度。

在 Relay 中，为自验证退款工单详情页的正常显示，Agent 编写并执行了端到端测试：

```bash
$ npx playwright test src/e2e/refund-workflow.spec.ts

[INFO] Navigating to: http://localhost:3001/agent/refund-queue
[ACTION] Selector "#queue-table tr[data-id='ref_98274a']" clicked.
[VERIFICATION] Expected ".refund-amount" to display "$1,200.00"
               Actual: "$1,200.00"  Status: PASSED
[INFO] Saving artifact: ./artifacts/screenshots/e2e_pass_ref_98274a.png
[SUMMARY] 1 spec passed. End-to-end self-verification succeeded.
```


## 调试挫败细节：消失的前端“元”与“分”之差

在重构 Relay 审批表单时，Agent 将后端退款 API 的浮点金额字段统一规范为“整数分（Cents）”。这一改动让所有后端 API 单元测试通过，类型系统也顺畅对齐。

但在 Agent 自动拉起 Playwright 执行端到端自验证时，截图比对抛出了像素漂移错误。由于前端模板漏掉了针对新分字段的 `/100` 格式化过滤器，导致一笔 1,200 元的退款在坐席审核界面上被直接渲染成了 `120000.00 元`。

任何后端逻辑检查都无法拦截此类纯显示层缺陷。若该代码直接合入生产，将导致客服发生误批与资金风险。正因为 DoD 强加了“Agent 必须拉起真实浏览器并核对截图”的刚性物理红线，该缺陷在交付前最后一刻被拦截，并由 Agent 自动追加格式化过滤器修复。


## 验收困境：总体指标掩盖高风险子场景

普通功能的验收相对单一，而复杂系统在业务层面的验收则极易陷入“均值稀释”的陷阱。

Relay 在一次发布前，团队按照汇总的核心场景通过率进行验收，各项指标均显示达标，予以放行。然而系统上线第三天，客服主管反馈自动退款工单队列大面积积压，而客户退款投诉量却在攀升。排查发现，业务验收只关注了全量场景的汇总通过率，而业务失败几乎 100% 集中在“自动退款”这一最高流量、最高风险的子场景中——被逻辑判定失败的退款会话被错误地分类并堆积在静默队列中。

教训是：**业务验收门槛必须按照子场景进行分层隔离。** 总体通过率仅能作为健康度的粗筛，高风险子场景（如自动划扣、转人工、计费拦截）的专项验收通过率，才是上线的真实门槛。


## 业务验收 KPI 的定义方式

业务验收指标（KPI）必须在 Plan 阶段的 Spec 规约中与业务方达成对齐并写入文档，严禁在上线前临时捏造。在编码完成后才去追问“何为正确行为”，业务方往往会受既有实现的影响而被迫降低预期。我们将指标划分为准入、质量与成本三类，进行隔离定义与追踪：

```markdown
## Relay v1.0 业务验收指标（Spec 阶段定义，上线前确认）

### 准入 KPI（未达标不上线，硬门槛，用与 AI 无关的通用验收指标）
- 核心场景（退款、高风险转人工）由产品经理在 Staging 环境确认通过。
- 误触发关键操作（用户未明确表达即自动执行退款/扣款）发生率 = 0（在对抗性测试中）。
- 高风险子场景专项验收通过，不被总体通过率稀释。
- 回滚演练通过：Staging 环境可在 5 分钟内自动回滚到上一稳定版本。

### 质量 KPI（上线后持续监控，低于阈值触发优化迭代）
- 自动解决率 ≥ 60%（无需人工坐席介入即完成的对话比例）。
- 平均对话轮数 ≤ 3 轮。
- 用户满意度评分（CSAT）≥ 4.0 / 5.0（上线后首周随机采样）。

### 成本 KPI（治理上限，超出触发架构评审）
- 每笔已解决对话的模型 token 费用 ≤ $0.15。
- 单次对话 token 消耗 P95 峰值 ≤ 20,000。
```

准入 KPI 具备刚性契约性，未达标绝对禁止上线；质量 KPI 负责在线上提供持续反馈，作为下一次外环 Reflect 的输入；成本 KPI 则作为安全阀，防止系统在技术上可行但在商业成本上破产。

**确立最低可行性 AI 输出观察集**：不进行复杂的 AI 产品级 Eval（如评测套件设计、Judge 裁判校准及精度调优，这些属于 AI 产品工程范畴）并不意味着对模型输出质量放任不管。开发方法论可接受的底线是：**针对退款和高风险转人工等核心场景，必须维护一个由人类专家标注的、包含 50-100 个典型案例的“最低可行性抽检样本库”**。每次发布前，由人工或异构评审 Agent 对生成结果进行抽样比对。这在控制工程成本的同时，能为非确定性输出拉起一条最低限度的回归防御底线。


## DoD 检查清单

除了机器自动化的检查，DoD 还必须叠加人工的业务确认：

```markdown
## DoD 检查清单（每次发布前执行）

### Agent 自验证（CI 自动运行，全部通过后方可进入人工验收）
- [ ] 单元测试和集成测试 100% 通过。
- [ ] 关键路径黄金测试与并发幂等契约测试通过。
- [ ] 对抗性安全测试（防提示词注入）通过。
- [ ] ESLint 静态校验通过，无 `console.log` 残留，无 PII 日志泄漏。
- [ ] 关键决策路径完成可观测性埋点（`callLLM` 必须包含 `traceId`）。
- [ ] Playwright 端到端浏览器自验证跑通，视觉截图已归档。

### 人工验收（Staging 环境最终确认，由产品经理或业务方执行）
- [ ] Staging 环境验证了退款、高风险转人工两个核心场景。
- [ ] 高风险子场景专项验收通过（未被均值指标稀释）。
- [ ] 线上回滚方案实证演练通过（5 分钟内完成回滚）。
- [ ] Staging 与生产的运行配置（环境变量、依赖版本）已确认完全一致。
- [ ] 生产环境告警规则配置确认（如误触发退款告警、业务指标异常告警）。
```

自验证由 CI 自动化卡口强制运行；业务验收则必须由真实人类在发布前逐条查验并签字授权。AI 无法平替最终的业务验收——“Staging 表现是否满足业务预期”依赖人类的行业经验，而非单纯的绿灯状态。

### 没有 DoD 清单的代价

在 Relay 早期迭代中，项目由于缺乏刚性的 DoD 清单管理，开发人员仅在本地验证通过单测后，便将分支合入主干。产品经理原本计划在 Staging 环境中对核心退款流进行手工核对，但因突发的紧急迭代直接绕过了该环节，导致代码直接推送上线。

系统部署后三小时，生产环境监控告警爆发：Stripe 退款队列发生大面积堆积，告警指示配置项解析异常。排查定位显示，Staging 环境与生产环境的运行时环境变量发生了版本漂移，导致 Agent 在 Staging 呈现的“正常表现”实际上是旧版本逻辑的误报。此次事故导致线上退款功能阻断达 4 小时。团队随后在 DoD 检查清单中强制追加了“运行期配置（环境变量、依赖版本）Staging 与生产完全对齐”的物理校验项。


## gstack 参考实现

端到端自验证由命令行的 `/qa` 技能承载，在隔离的 worktree 沙箱里自动拉起预配置的 Playwright 测试套件：

```bash
$ gstack qa --visual-check
```

它自动加载 `./playwright.config.ts` 中的参数，在无头浏览器中运行自验证脚本并输出快照；一旦发现渲染错误或视觉漂移，执行迹线与截图将自动流回当前 Agent 编写会话，指导其进行故障自愈。测试框架属于可替换的物理层，核心是不通过端到端行为自验证，代码绝对禁止合入。


## 典型反模式

* **静态编译通过即视为完成**：CI 跑通即盲目上线，不对最终的浏览器交互与渲染进行行为验证，将白屏与排版塌陷漏过。
* **自验证缺乏物理物证**：容许 Agent 给出“本地已手动验证”等主观描述，无运行截图、无网络接口调用迹线归档。
* **用均值通过率掩盖高危缺陷**：仅看汇总的验收通过率，导致核心资金路径的严重缺陷被高频普通咨询的绿灯均摊。
* **验收指标临时凑数**：在代码编写完成后才定义 KPI，导致技术妥协影响验收硬指标。


## 本章要点

* 机器检查通过不等于最终行为正确。DoD 必须由技术自验证（端到端 Playwright、关键路径）与人工业务验收（核心场景确认、回滚演练）双层构成。
* 业务验收指标（KPI）必须在 Plan 阶段的 Spec 规约中前置定义，且必须针对高风险子场景实施分层指标隔离，杜绝总体均值稀释。
* 准入 KPI 使用与 AI 无关的通用验收指标；虽然不跑 AI 产品级 eval，但必须保留最低限度的人工抽检样本库以防出现非确定性纯黑箱输出。
* 自验证的终端迹线与快照是物理凭证，人工最终签字是业务放行授权，两者均不可缩减。
