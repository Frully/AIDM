# 锁定方案：多视角评审与设计文档先行

Relay 项目早期有过这样一个插曲：Agent 把自动退款的方案写完之后，在同一个对话里让它"再检查一遍"。它检查完说没问题，于是直接开始编码。两周后测试发现，并发场景下同一笔退款会触发两次出款。回头看方案，漏洞一目了然——但那个"再检查一遍"的瞬间，Agent 就是看不见它，因为它坐在自己写下那行代码的位置上，脑子里装着"这段逻辑是对的"这个前提。

这不是 Agent 不够聪明，是上下文的物理位置决定了它能看见什么。同一个会话里，它对自己刚写下的东西永远带着作者惯性。方案评审如果不换人换上下文，等于没审。

**写方案与审方案必须在独立的上下文中分离执行，且评审通过后的第一件事是把架构决策写进设计文档——这两步合在一起，才算 Plan 阶段完成。**

## 为什么自审在结构上就是失败的

会话上下文不是中性的容器。一个 Agent 在写完方案的会话里，它的注意力权重分布是倾向于"这个方案能走通"的。让它在相同上下文里挑漏洞，等于让它用支持自己的逻辑来质疑自己——结果通常是找到一些表面的笔误，而忽略更深的结构性问题。

换一个新上下文（新会话、新 Agent 角色，或者换一个不同的底层模型）之后，评审者面对的是一份陌生文档，没有"我刚刚想通了这个逻辑"的记忆负担。它审视数据流时，超时路径、并发竞态、幂等缺失这类问题就会自然浮出来，因为它没有"这里肯定没问题"的预设。

人类的 Code Review 实践已经把这个道理验证了几十年。作者写完代码提 PR，必须等另一个人来审，正是同一个原因。AI 协同开发把这个原则推到方案层面：在开始编码之前，方案就要过一遍异构视角的评审，而不是等到代码合并前才发现设计层面的错误。

## 三个视角查不同维度的漏洞

一份方案的问题从来不是单一维度的。范围蔓延、工程隐患、交互断裂是三类性质不同的缺陷，需要三种不同的眼光来发现。把它们合并成"全面评审"交给一个角色，往往的结果是每个维度都蜻蜓点水，没有一个查透。

**CEO / 业务视角**专门看方案是否老老实实待在 Spec 的边界里。重点检查：这个方案有没有悄悄实现 Spec 里没要求的功能（scope creep）？引入了哪些不必要的技术复杂度，比如一个本来不需要的消息队列，或者超前三个版本的过度解耦？核心用户操作路径有没有遗漏场景？这个改动的优先级和当前最重要的业务目标是否对齐？CEO 视角不关心怎么实现，只关心"做没做该做的事"。

**工程视角**专门查数据流的每一个异常分支。每个有副作用的外部接口调用——写数据库、调第三方支付、发通知——有没有幂等键设计？高并发下，两个请求同时通过限额校验然后都执行，这种竞态条件有没有被锁机制挡住？数据库事务边界是否正确，有没有把本该在一个事务里的写操作拆散了？网络超时、下游不可用的降级路径是否明确？测试矩阵是否覆盖了所有涉及资金和权限的关键路径，而不只是正常路径？

**设计视角**只在涉及用户界面的功能时启用。检查新增交互是否与设计系统规范一致（颜色、间距、组件选择），有没有出现非标准组件，用户操作链路从触发到反馈的每一步是否都被完整考虑。

这三个视角的问题不会相互覆盖，但都可能致命。业务视角发现的 scope creep 不会出现在工程评审报告里；幂等缺失不会出现在 CEO 评审里。只有分开跑，才能分别抓住。

## 按风险裁剪视角，不是每个改动都跑全套

视角的数量要跟改动的风险规模匹配。一个只修改某个 API 参数类型的小改动，跑一遍工程视角就够了，没必要拉 CEO 视角来讨论业务优先级。一个涉及用户界面变化的功能必须加设计视角，但如果它的边界和优先级已经在 Spec 里明确了，CEO 视角可以裁掉。只有当改动涉及范围调整或优先级重排时，才有必要跑 CEO 视角。

评审投入和改动风险成正比是基本原则。把全套视角强制套在每一个小修改上，会让评审变成纯粹的开销，工程师最终会绕过它。

## 意见冲突时怎么收敛

不同视角给出冲突的建议是正常的。工程视角说"引入消息队列做异步隔离"，CEO 视角说"这会增加至少两周的复杂度，当前 MVP 不应该有这个"。这时候既不能让评审无限循环，也不能强行宣判某一方正确。

处理方法是：把冲突点本身写进设计文档，明确列出两种立场、各自的理由，以及最终的取舍决定和决策者。设计文档里有一节专门记录这些，后来接手的 Agent 或工程师看到"曾经考虑过异步方案但因为 MVP 复杂度被否决"，就不会在下一个版本里重蹈覆辙或者重新打起同样的架构辩论。收敛权归 Plan 负责人，不归某个视角。

## Relay 评审迹线：工程视角发现两个高危漏洞

Relay 自动退款方案在提交给工程视角评审时，评审 Agent 在全新上下文里读取了 `./docs/plans/auto_refund_flow_v1.md`，三分钟内产出了如下报告片段：

```markdown
### 评审对象: auto_refund_flow_v1
### 评审视角: 工程/架构评审

[WARN] 金融副作用接口缺失幂等键设计
- 缺陷定位：方案第 24 行在下游超时处理中，直接调用了第三方退款接口 `stripe.refunds.create`。
  当网络抖动发生 504 Gateway Timeout 时，重试模块（第 32 行）可能发起二次请求，
  引发重复退款事故。
- 整改要求：所有涉及资金流出的外部接口调用，必须在本地持久化创建唯一交易凭证，
  并透传基于 sha256(order_id + refund_action_sequence) 生成的幂等键。

[WARN] 高并发下存在资源竞态（Race Condition）
- 缺陷定位：缺少悲观锁机制。两个并发请求可能同时通过金额阈值校验，
  然后各自触发退款，导致重复出款。
- 整改要求：在执行金额校验前，必须用 SELECT ... FOR UPDATE 或分布式锁
  锁定对应订单状态，保证同一订单同一时刻只有一条处理链路在执行。
```

这两条在主写 Agent 的会话里都看不见——写方案时，超时重试的逻辑"显然"是对的，并发校验"显然"在前面就拦住了。换上一个没读过这段逻辑的评审上下文，问题立刻浮出来。

方案按整改要求修改后，工程视角再跑一遍，两条 WARN 消失，方案锁定。

注意：此时评审结论并没有"就这样了"，而是直接成为下一步设计文档的输入。幂等键的构造算法、锁机制的选择，都进了 `docs/designs/auto_refund_v1.md` 的对应章节。这是评审和设计文档之间的天然接续：评审找到了问题，设计文档记录了怎么解决以及为什么这么解决。

## 设计文档：让架构取舍跨越会话断裂带

评审通过之后，如果直接开始写代码，项目会进入一个隐性的信息黑洞。代码记录了最终的计算逻辑，却不记录"曾经考虑过同步方案，因为并发风险被否决"这件事。等半年后新的 Agent 接手，面对一个看起来过度复杂的多层队列结构，它没有理由不"简化"它。等它删掉异步隔离层，退款模块在峰值流量下的雪崩会重演一遍。

设计文档填的是这个空白——不是记录"代码里定义了什么"，而是记录"为什么是这个架构而不是其他的，以及我们否决了什么"。这个差别决定了它的实际价值。

在 AI 协同开发里，设计文档还有一层格外重要的作用：它是跨越会话记忆断裂带的唯一交接物。每个新 Agent 都是冷启动，没有任何关于之前对话的上下文。如果没有设计文档，它只能从代码反推设计意图，而从代码到意图的路径充满歧义。有了设计文档，它读进去，就能重建关键的心智模型，安全地继续迭代。

文档流的顺序是刚性的：Spec（说系统必须做什么）→ Design（说内部如何构建）→ 代码（实现）。Spec 是 Plan 阶段把 Think 阶段的设计精炼后的工程规范，明确接口约定、边界参数和状态定义。Design 是在 Spec 约束下回答"内部数据流怎么走、架构选型为什么这样定"的文档。两者层次不同，Spec 是 Design 的刚性输入，跳过 Spec 直接写 Design 等于在没有需求约束的情况下做架构决定。

## gstack 参考实现

`gstack` 把三个评审视角解耦为独立的原子 Skill，通过 `/autoplan` 工作流在编码启动前自动调度并合并评审产出：

- `/plan-ceo-review`：业务视角，专门砍掉 Spec 边界外的冗余功能设计。
- `/plan-eng-review`：工程视角，诊断数据流完备性、幂等缺失、竞态条件、事务边界。
- `/plan-design-review`：设计视角，扫描交互一致性和非标组件，只在涉及 UI 的功能上启用。

如果想引入更彻底的异构对立，可以通过 `/codex` 调度与方案主写 Agent 不同的底层模型——比如用推理模型写方案、用长上下文轻量模型挑漏洞——强迫第二意见真的来自不同的推理路径。

设计文档骨架模板：

```markdown
# 设计方案: [功能名称] v[版本号]
# 关联需求: [指向对应 Spec 文件的相对路径]

## 1. 背景与目标
描述此设计解决的工程痛点与业务价值。

## 2. 架构取舍
列出曾考虑的备选方案，说明采纳当前方案的权衡依据。
重点写"为什么否决了其他方案"，不只写"现在怎么做"。

## 3. 核心流程与数据流
描述核心模块间的调用关系与时序，明确标注同步接口与异步消息边界。

## 4. 副作用与幂等设计
列出所有存在外部写副作用的环节，说明幂等键构造机制与并发保护方案。

## 5. 异常处理与降级回滚
定义各环节超时、网络错误的重试规则，说明分布式事务回滚或补偿链路。

## 6. 评审结论摘录
记录评审阶段发现的关键问题和整改决策，以及意见冲突的取舍记录。
```

## Relay 设计文档片段

Relay 自动退款方案经过评审锁定后，Agent 在启动编码前输出了 `./docs/designs/auto_refund_v1.md` 的核心片段：

```markdown
# 设计方案: 异步自动退款系统 v1
# 关联需求: docs/specs/auto_refund_spec.md

## 架构取舍

### 同步 vs 异步退款
决议：否决 HTTP Webhook 同步调用 Stripe API 的方案，采用「判定 → 创建审批闸
→ 异步入队列 → 消费者执行」的解耦设计。

权衡理由：Stripe 在高并发时极易发生 504 延迟。同步调用会长时间占用数据库
连接池，导致客服接入主线程因连接数耗尽而雪崩。引入 RabbitMQ 后，将强一致性
放宽为最终一致性，外部调用隔离至独立消费者，主线程高吞吐量得以保持。

## 副作用与幂等设计
- 幂等键构造：sha256(order_id + refund_action_sequence)
- 并发保护：执行金额校验前，SELECT ... FOR UPDATE 锁定订单行
- 回滚机制：消费者捕获 RefundFailureEvent，触发 Saga 事务回滚，
  解冻对应订单的可退款额度

## 评审结论摘录
- 工程视角 WARN-1（幂等键缺失）：已在幂等设计章节中落实
- 工程视角 WARN-2（竞态条件）：已在并发保护方案中落实
- 曾有讨论是否引入分布式锁（Redis）替代数据库悲观锁；
  因当前部署未包含 Redis，决定 v1 沿用 SELECT ... FOR UPDATE，
  v2 视负载情况再迁移
```

这个片段里的"曾有讨论是否引入 Redis 锁"就是当初 CEO 视角和工程视角之间意见的产物。工程视角想要 Redis，CEO 视角说现在没有 Redis 是个运维负担，最终决策写进了文档。六个月后任何接手这个模块的 Agent 看到这一行，都不会重新发起一场关于是否引入 Redis 的辩论——因为决策过程已经在这里了。

## 典型反模式

**同一会话里自写自审。** 写完方案后在同一对话里让 Agent "再检查一遍"。这不是评审，是作者惯性的自我确认。盲区在那个上下文里是不可见的，因此就这样被保留了下来，直到测试或者线上暴露。

**评审报告挂起不落地。** 工程视角发现了幂等缺失，把报告存了一个 Markdown 文件，然后 Agent 开始编码，代码里依然是原来的逻辑。评审必须整改并重新确认，整改结论必须写进设计文档，否则评审等于走了一遍形式。

**设计文档只复述现状不记录否决。** 文档里只写"当前实现是异步队列"，不写"曾经考虑过同步方案，因为并发风险被否决"。这类文档在被接手时毫无防护作用，后来的修改者完全可能重新发明同步方案，重踩已经被前人系统地绕开的坑。

**跳过 Spec 直接写设计文档。** 在 Think 阶段产出的设计意图还没有被精炼成工程 Spec 之前，就开始写内部架构设计。结果是设计文档里的接口约定、边界参数自行定义，与 Spec 产生漂移，最终实现和业务需求对不上。

## 本章要点

- 评审必须在独立的上下文中进行，换会话、换角色、换模型都可以，但绝不能是方案主写者在同一对话里自我审核。
- 三个视角查不同维度：业务视角查边界和优先级，工程视角查数据流异常和并发安全，设计视角查交互一致性。视角数量按改动风险裁剪，不是每个改动都跑全套。
- 冲突的评审意见不靠循环讨论收敛，而是把冲突点和取舍理由写进设计文档，由 Plan 负责人拍板。
- 设计文档的核心价值是记录架构取舍和被否决的方案，而不是复述代码里已有的内容。
- 评审通过 → 整改落实 → 整改结论写进设计文档 → 方案锁定，这个顺序是 Plan 阶段完成的判断标准。
