# 锁定方案：多视角评审与设计文档先行

在 Relay 早期研发中，Agent 编写完自动退款方案后，被要求在同一对话会话内“自我评审”。Agent 确认方案无误并直接进入编码。然而，上线后在并发环境下，同一笔退款因未加锁触发了二次出款。回头审视方案，漏洞显而易见，但写方案的 Agent 在自我确认的瞬间，因无法跳出“当前设计完全正确”的推演假说，对并发竞态视而不见。

**方案评审与设计锁定必须在分离的物理上下文中独立进行，以防范编写者自身盲点导致的架构退化。** 

---

## 为什么自审在结构上必定失败

大模型会话的上下文具有极强的启动效应与逻辑偏向。当 Agent 在同一个会话里推演完方案后，其注意力权重已被强力锚定在证明“该方案能走通”的路径上。让同一个会话自审，模型只会查找拼写或接口命名的浅表瑕疵，无法挑战底层的设计漏洞。

真正的评审必须执行物理隔离。开启一个干净、无状态的新会话，或者引入与编写模型不同底座的异构模型，让评审 Agent 充当批判者。新会话没有“前文推理成功”的上下文包袱，只针对方案 Diff 文本发起质疑，才能让超时降级、并发竞态与幂等设计等缺陷自然浮出。

---

## 三视角评审的分工原则

系统设计的缺陷分为不同维度，混合评审往往导致每个层面都浅尝辄止。评审应拆分为三个独立的视角：

1. **CEO / 业务视角**：确保方案在 Spec 的刚性边界内运行。重点拦截 scope creep（范围蔓延），防止 Agent 超前引入不必要的技术复杂度（如非必须的中间件或过度解耦结构），确保开发资源聚焦于当前核心业务。
2. **工程视角**：专门审查数据流的异常边界。重点检验所有外部写操作（如数据库写入、第三方支付调用、外部通知发送）是否具备幂等键设计；并发请求下是否存在资源竞争，悲观锁或分布式锁是否配置妥当；数据库事务边界是否正确；网络超时降级策略是否明晰。
3. **设计视角**：仅在涉及用户交互的变更中启用。负责验证新增的界面元素与交互逻辑是否严格符合设计系统（Design System）规范，防止 Agent 随意发明非标准组件。

## 视角的动态裁剪与冲突收敛

评审投入应与改动风险成正比。小型的内部重构仅需通过工程视角校验；不涉及 UI 的改动应裁撤设计视角；若业务目标在 Spec 阶段已经高度明确，可免去 CEO 视角的重复确认。

当不同评审视角产生意见冲突时（例如工程视角要求引入异步队列做隔离，而业务视角因开发排期要求保持同步极简），收敛的终点不是无休止的会话争论，而是由 Plan 阶段的人类负责人进行物理裁决。裁决结论连同否决的替代方案必须被显式记录在随后的设计文档中，作为未来迭代的可审计决策资产。

## Relay 评审演练：工程视角的负向反馈

在 Relay 中，当自动退款方案提交至工程视角评审时，评审 Agent 在全新上下文中读取了方案，输出如下整改报告：

```bash
$ gstack review --plan ./docs/plans/auto_refund_flow_v1.md --view eng

[WARN] 金融副作用接口缺失幂等键
  - 缺陷：方案在超时重试分支中直接调用 `stripe.refunds.create`，若网关发生 504 超时，重试可能引发重复出款。
  - 整改：必须在本地生成交易凭证，并透传基于 sha256(order_id + refund_action_sequence) 的唯一幂等键。

[WARN] 并发下存在资源竞态 (Race Condition)
  - 缺陷：缺少锁机制。两个并发请求可能同时通过金额阈值校验，造成重复出款。
  - 整改：执行金额校验前，使用 SELECT ... FOR UPDATE 对订单行加悲观锁。
```

这两条高危 WARN 在方案主写会话中完全未被捕获。修改并重新跑通工程评审后，WARNs 消除，方案正式锁定。

## 设计文档：应对会话断裂带的交接契约

评审通过后，编码前必须将最终决策沉淀为设计文档（Design Doc）。代码仅记录当前的最终实现，无法表达“为什么不采用同步方案”等被否决的方案，这会在后续迭代中留下隐性漏洞——后来接手的 Agent 可能会因为看不懂复杂的异步结构，而轻易将其重构成有缺陷的同步逻辑。

设计文档是应对 Agent “会话记忆断裂”的交接物。Agent 每次开工都是冷启动，设计文档是重建其架构心智模型的唯一事实源。

文档的流动链条必须保持刚性方向：**Spec（系统做什么）→ Design（如何构建）→ Code（实现）**。Spec 是 Design 的刚性输入，严禁跳过 Spec 直接做架构设计。

## gstack 参考实现

在 gstack 体系中，多视角评审被封装为独立的命令工具（如 `/plan-review` 技能），以确保评审上下文的干净与隔离：

```bash
# 调起独立的工程与业务评审，结果输出为独立的 Markdown 报告
$ gstack plan-review --plan ./docs/plans/auto_refund.md --views eng,ceo
```

如果需要引入更彻底的逻辑对抗，gstack 能够通过配置调起与编写模型不同的推理模型（如在 `/codex` 中配置异构大模型作为评审裁判），使第二意见不携带相同的注意力偏置。设计文档推荐存放在 `./docs/designs/` 目录下，并以统一的骨架管理：

* **背景与架构取舍**：交代方案的核心目标与已否决的方案。
* **副作用与幂等设计**：写明悲观锁与分布式锁的加锁字段及幂等键格式。
* **异常流与评审结论**：记录评审整改意见及最终裁决。

## 典型反模式

* **同一会话里自写自审**：这导致 Agent 被作者偏向所误导，直接放过架构层面的关键漏洞。
* **评审报告挂起不落地**：评审发现的问题只在会话聊天里口头讨论，没有被写回设计文档，后续 Agent 冷启动时无法继承这些整改要求。
* **设计文档只复述现状**：仅记录“怎么做”，漏掉了“否决了什么”。导致新一轮的 Agent 尝试重构时，会把曾经被否决的坏方案重新写一遍。

---

## 本章要点

* 方案与评审必须彻底隔离上下文，利用异构模型或干净新会话查找潜在盲点。
* 三视角评审（业务、工程、设计）分层查错，按改动风险动态裁剪，收敛权归人类负责人。
* 设计文档是冷启动 Agent 继承架构意图的物理凭证。Spec → Design → Code 顺序不可跳过。
