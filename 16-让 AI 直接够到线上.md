# 让 AI 直接够到线上

有了完善的可观测性数据与外部告警系统，一旦线上发生故障，我们的目标是尽可能压缩故障定位与解决（MTTR）的时间窗口。在传统的研发模式中，线上环境的排障完全依赖于人类开发者登录跳板机去分析日志。在 AI 驱动开发的模式下，最高效的排障手段是为 Agent 提供一套受控的安全通道，使其可以直接查阅线上反馈数据，自主完成定位与代码修复。

**反馈链路必须向 Agent 提供闭环访问通路。通过封装最小只读权限的包装命令行工具，允许 Agent 在被授予操作令牌时，自主调用命令拉取线上对应 Trace ID 的上下文日志。严禁以"手工拦截并人工复制日志片段给 Agent"的低效沟通模式维持排障工作，这会使人类开发者沦为昂贵且迟钝的"人肉管道"。**

## 安全防线与最小只读权限设计

让 Agent 能够拉取线上反馈，绝不等于向其敞开未经限制的安全后门，这会给生产系统带来灾难性的逻辑风险。我们必须采用严格的"开窗而非拆墙"的安全架构原则：

* **完全的只读权限限制**：给 Agent 的命令行通道或 API 客户端只读权限。Agent 绝不应拥有在生产数据库上执行 `UPDATE`、`DELETE` 或对容器执行物理重启的操作权。
* **防注入的 API 包装**：不直接向 Agent 暴露原始的数据库连接器或 SSH 接口，而是向其提供一层进行了防参数注入（SQL Injection/Shell Injection）封装的专用只读查询脚本（如 `relay-ops`）。
* **基于双签的危险操作护栏**：对于确实需要执行紧急变更的操作（如刷新特定内存缓存以恢复被阻塞的订单流），命令执行时必须产生确认机制，必须要求人类开发者在终端显式输入 `y/n` 以授予物理授权许可。
* **全量审计日志归档**：Agent 发起的所有线上查询命令，必须在审计中心留下不可篡改的日志，以便于合规性回溯。

## AI-native 可观测性：日志是给机器读的

把日志设计成给人看的，这个前提在 Agent 排障时会造成实际损失。

Relay 早期的退款日志长这样：

```
[2024-01-15 10:23:45] Refund processed for order #12345, amount: $25.00 ✓
```

人读没问题——时间、订单、金额一目了然。Agent 读就麻烦了：它要从这个自由格式字符串里提取 `order_id`，要判断 `✓` 代表成功还是其它含义，要把 `$25.00` 转成分单位数字才能和系统里其他字段做比较。当 Runbook 里写 `if amount_cents > 50000 then escalate`，Agent 却在日志里读到 `$25.00`，它就得先做单位换算再做比较——这个转换步骤不在任何地方有文档，Agent 可能换对，也可能不换。

同一条信息，AI-native 的结构是：

```json
{
  "ts": "2024-01-15T10:23:45Z",
  "event": "payment.refund.completed",
  "conversation_id": "conv_abc123",
  "order_id": "order_12345",
  "amount_cents": 2500,
  "duration_ms": 342,
  "status": "success"
}
```

每个字段名是合同。`conversation_id` 出现在这里、在数据库里、在 Runbook 的查询条件里，三处必须拼写完全一致。`amount_cents` 永远是整数分，不存在 `$` 符号，不存在小数。`event` 字段的值 `payment.refund.completed` 一旦被 Runbook 或 Eval 引用，就和代码里的函数名一样——你改掉它，引用它的地方全部静默失效。

失败路径的记录同样有硬约束：成功记一条 INFO，失败不能只写 `"status": "error"`，必须有 `reason` 字段，值是具体的枚举字符串：

```json
{
  "ts": "2024-01-15T10:24:12Z",
  "event": "payment.refund.failed",
  "conversation_id": "conv_abc123",
  "order_id": "order_12345",
  "amount_cents": 2500,
  "duration_ms": 5103,
  "status": "error",
  "reason": "kb_timeout",
  "retry_count": 2
}
```

Agent 在 Runbook 里按 `reason` 做路由决策，`"reason": "kb_timeout"` 走重试分支，`"reason": "refund_limit_exceeded"` 走升级分支。如果这个字段缺失或变成 `"reason": "timed out connecting to kb"`（自由文字），Agent 的条件判断就失效了，它不得不去猜这句话是不是超时，这种猜测的准确率无法被 Eval 稳定衡量。

事件名称 `payment.refund.failed` 也是接口。它不是注释，不能随手改成 `refund.payment.error` 或 `payment_refund_failed`——哪怕语义完全相同，Runbook 的 trigger 条件和 Eval 的断言都会同时失效，且没有任何编译器会报错。

## AI-readable Runbook：条件可执行，而非步骤清单

人工 Runbook 的标准格式是步骤序列：「1. SSH 到生产服务器；2. 查看 `/var/log/relay/refund.log`；3. 搜索含 TIMEOUT 字样的行；4. 根据搜索结果判断……」这对人有效，因为人能灵活解读每一步的意图，遇到偏差可以自行调整。Agent 执行这套流程时，SSH 在哪、日志路径是什么、`TIMEOUT` 字样匹配哪种实际错误——每一步都需要额外的上下文，而这些上下文原本是工程师脑子里隐含的知识。

能被 Agent 实际执行的 Runbook，需要是声明式规则：trigger 是可查询的事件条件，diagnosis 是可直接执行的查询命令，action 是带条件守卫的操作指令。Relay 退款超时的 Runbook 片段如下：

```yaml
# relay-runbook.yml
incidents:
  - name: refund_timeout
    trigger: "event=payment.refund.initiated AND duration_ms > 5000"
    diagnosis:
      - query: >
          SELECT conversation_id, amount_cents, retry_count, created_at
          FROM refund_queue
          WHERE conversation_id = :conversation_id
            AND created_at > NOW() - INTERVAL '1 hour'
      - check: "curl -s https://payment-gateway.internal/health"
    auto_actions:
      - condition: "retry_count < 3 AND amount_cents < 50000"
        action: "retry_refund"
      - condition: "retry_count >= 3 OR amount_cents >= 50000"
        action: "escalate_to_human"
        reason: "超出自动处理阈值，需人工审批"
```

Agent 读到 `trigger: "event=payment.refund.initiated AND duration_ms > 5000"` 时，它知道去日志里查这个条件，而不是去猜哪个事件算"退款超时"。`diagnosis` 里的 SQL 里有 `:conversation_id` 占位符，Agent 用当前会话的 `conversation_id` 填入后可以直接执行，不需要猜表名。`auto_actions` 里两个 `condition` 是互斥的穷举，Agent 不需要在没有覆盖到的分支上自行判断——遇到两个条件都不满足的情况，它就等待，不乱动。

Relay 在没有这套结构之前，曾经有一次人工写了一份四页的 Runbook 文档，让 Agent 自行阅读后执行。Agent 读完后把 `escalate_to_human` 理解成了"给用户发通知邮件"，因为文档里有一句「将此问题上报给相关方」——这是合理的字面解读，但完全不是期望的行为。结果一笔 ¥12,000 的退款在等待人工审批期间，Agent 先给客户发了一封措辞模糊的"我们正在处理您的问题"邮件，客户转而找了另一个渠道投诉，产生了双重工单。事后复盘发现问题根源是 `escalate_to_human` 这个 action 没有配置具体的执行目标，只是一段自然语言描述。

## 渐进式访问展开：权限跟着信任走

给 AI 访问线上的权限，不是开个账号然后附上一份免责声明就结束了。权限应该分层开放，每一层对应一个可验证的信任里程碑，没有通过验证就不往下走。

**第 1 层：只读日志查询。** Agent 可以按 `conversation_id` 过滤日志，拉取指定时间窗口内的事件序列。`relay-ops logs search` 命令在这一层有效。通过标准：连续 5 次以上排障，Agent 能准确定位问题根因，没有执行过全表扫描，没有尝试越权访问其他 `conversation_id` 的数据。

**第 2 层：只读数据库查询。** Agent 可以执行 `SELECT`，连接串配置了只读副本，ORM 层屏蔽了 `UPDATE`/`DELETE`/`INSERT` 语句——不是靠 Agent 自觉，而是连接串所用的数据库账户在数据库层面没有写权限，执行写语句直接返回权限拒绝错误。通过标准：查询模式稳定两周，没有出现过全表扫描（通过慢查询日志监控），没有访问过 PII 相关表（`user_payment_info`、`customer_contact` 等）。

**第 3 层：无副作用诊断操作。** Agent 可以触发 `retry_refund`（重新入队，不修改已有记录）、刷新状态缓存这类操作。这些操作不改变核心业务数据，最坏情况是幂等地重试一次。通过标准：3 次以上成功的诊断案例，没有误触发（触发了但问题条件不满足的情况）。每次触发都有完整的审计日志留存。

**第 4 层：有副作用操作。** Agent 可以触发退款、发通知邮件这类写入外部系统的操作。每个操作都要经过参数守卫——`relay-ops` 的退款接口定义如下：

```typescript
// src/ops/refund.ts
export async function executeRefund(params: {
  conversation_id: string;
  amount_cents: number;
  operator: "agent" | "human";
}): Promise<RefundResult> {
  if (params.amount_cents > 50000) {
    throw new Error(
      `RefundGuardError: amount_cents ${params.amount_cents} exceeds agent limit 50000`
    );
  }
  if (!params.conversation_id.startsWith("conv_")) {
    throw new Error(
      `RefundGuardError: invalid conversation_id format`
    );
  }
  // ... 实际退款逻辑
}
```

当 Agent 尝试用 `amount_cents: 75000` 调用退款接口时，CI 里的集成测试会捕获这个行为，生产环境里的守卫会直接抛出 `RefundGuardError`，操作不会执行。这个守卫不是写在 `AGENTS.md` 里的"请不要退款超过 ¥500"——那是提醒，可以被忽略。守卫在代码里，绕不过去。

这四层权限的当前状态，每一层都在 `AGENTS.md` 里有对应的权限描述：

```
## Agent 当前线上权限（2024-01-15 更新）
- 第 1 层（只读日志）：已开放
- 第 2 层（只读 DB）：已开放，限只读副本
- 第 3 层（无副作用诊断）：已开放，retry_refund 和 cache_flush
- 第 4 层（有副作用操作）：已开放，amount_cents < 50000，需审计日志
```

`AGENTS.md` 里的这段描述和代码守卫是同步更新的。权限变化时，两处同时改——`AGENTS.md` 让 Agent 知道自己能做什么，代码守卫确保它做不了超出范围的事。

## 主线演练：Relay 安全排障 CLI 调用迹线

在 Relay 示例项目中，当监控系统抛出 `RelayRefundFailed` 告警时，排障 Agent 通过获取的安全只读 CLI，在本地执行了查询尝试。

以下是 Agent 运行的排障命令与终端交互的完整迹线：

```bash
$ relay-ops logs search --conversation-id conv_abc123 --limit 5

[INFO] Authenticating agent session... Done (Session ID: s_agt_91823)
[INFO] Executing read-only query against log index [logs-relay-prod-*]
[INFO] Query filter: { "conversation_id": "conv_abc123" }

--- Log Record #1 ---
{
  "ts": "2024-01-15T10:24:12Z",
  "event": "payment.refund.failed",
  "conversation_id": "conv_abc123",
  "order_id": "order_12345",
  "amount_cents": 2500,
  "duration_ms": 5103,
  "status": "error",
  "reason": "kb_timeout",
  "retry_count": 2
}

--- Log Record #2 ---
{
  "ts": "2024-01-15T10:23:45Z",
  "event": "payment.refund.initiated",
  "conversation_id": "conv_abc123",
  "order_id": "order_12345",
  "amount_cents": 2500,
  "duration_ms": 342,
  "status": "success"
}

```

Agent 读到 `"reason": "kb_timeout"` 和 `"retry_count": 2` 后，对照 `relay-runbook.yml` 里的条件：`retry_count < 3 AND amount_cents < 50000` 满足，执行 `retry_refund`。这个判断是直接的字段比较，不需要解读自由文字。

整个排障过程没有人介入。5 分钟内退款重试成功，`payment.refund.completed` 事件写入日志。

## 调试挫败细节：巨大模糊查询导致的 OOM 灾难

在早期设计中，我们给 Agent 暴露了一个宽泛且无硬性护栏的日志查询工具。在一例"Stripe Webhook 延迟处理"的线上异常排障中，Agent 为了定位故障范围，在没有配置时间范围与 Trace ID 过滤器的情况下，直接执行了全表扫描指令：

```bash
# Agent 在不设防的 CLI 中发起了全量模糊检索
$ relay-logs-admin query "Stripe Webhook" --limit 500000
```

该模糊查询瞬间击穿了测试网关，并导致远端 Elasticsearch 的单节点因为扫描海量 JSON 文档而内存暴涨，最终触发系统 OOM 宕机。同时，由于拉取了近 1.2 GB 的冗余 JSON 日志，瞬间撑爆了 Agent 自身的 LLM 传输上下文（Context Window），导致其由于 Token 溢出在终端直接崩溃。

这次由于权限不设防带来的生产停机事故痛击了团队。此后，我们重新设计并封装了排障工具链，对所有对外导出的查询命令在 CLI 入口强加了 `--limit 100` 的物理拦截护栏，并彻底禁止了未带 `conversation_id` 或精确时间分片的全局正则搜索，防止此类人为逻辑故障再次破坏系统。

## 典型反模式

* **人肉日志管道**：因为数据合规名义封锁了全部日志查看端点，开发人员必须在微信中手动将 Kibana 日志截图或拼接的文字粘给 Agent，导致团队效率退回到石器时代。
* **过度暴露读写特权**：为图一劳永逸的安全设置，直接将生产环境的 AWS IAM / GCP 管理员密钥赋予 Agent 进程，使得 Agent 遇到编译逻辑漏洞时可能会物理清理掉生产数据库表。
* **查询缺失硬性护栏**：对于检索工具未作多级限流与 Limit 长度剪枝配置，导致 Agent 频繁由于上下文溢出或触发大额 Token 账单而无法连续作业。
* **用自然语言 Runbook 代替声明式规则**：把四页操作文档丢给 Agent 自行解读，Agent 会从语义上推断每一步的意图，推断结果和工程师的预期往往不一致，且这种偏差在测试环境里不会暴露，只在生产故障时出现。
* **日志结构随意演化**：`event` 字段从 `payment.refund.failed` 改成 `refund.payment.error` 只需要一行代码改动，但 Runbook 的 trigger 条件和 Eval 套件会同时静默失效，且没有任何 CI 门禁会报错。

## 本章要点

* 反馈只有被 Agent 够到才具备工程现实意义，必须为 Agent 提供直接的线上可观测数据访问能力。
* 日志字段名是接口：`event`、`reason`、`conversation_id` 一旦被 Runbook 或 Eval 引用，就和函数签名一样，不能随意改名。失败路径的 `reason` 字段必须是可枚举的具体字符串，供 Agent 按字段做条件判断。
* AI-readable Runbook 的核心是 `trigger → diagnosis → condition → action` 的机器可执行结构，而不是给 Agent 一份人工步骤文档让它自行解读。
* 给 AI 的线上权限分四层渐进开放，每一层对应可验证的信任里程碑；代码守卫（参数校验、数据库账户权限）是硬约束，`AGENTS.md` 里的权限描述是说明，两者同步更新。
* 通道包装要遵循最小特权、只读优先、基于令牌的安全审计策略，杜绝直接暴露写操作权限。

> 可观测性的反馈回路已在 Build 阶段闭环就绪。接下来的问题是：对于大模型驱动的功能，传统的精确断言测试已经失效——我们将在下一章讨论如何通过 Eval 测试大模型的行为档位。
