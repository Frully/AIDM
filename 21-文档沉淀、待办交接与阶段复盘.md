# 文档沉淀、待办交接与阶段复盘

Relay 某次支付重构合并之后，Agent 修改了 `stripe-client.ts` 中 `createRefund` 的签名，把原本的两参数接口升成了三参数必传。PR 顺利过了 Review，代码合入主干。

问题在半小时后出现。另一个冷启动的 Agent 分支按照 `docs/references/stripe-api.md` 里的旧版说明开发"客服拒绝退款自动通知"功能，写出了这行代码：

```typescript
await stripeClient.createRefund(chargeId, amountCents);
// 编译报错：Expected 3 arguments, but got 2.
```

该 Agent 由于冷启动时只读到了过期文档，反复认为是本地包依赖发生了漂移，不断运行 `npm install` 并尝试重写导入路径，在死循环里自我排障了近两个小时，烧掉了大量 Token 额度。

这不是模型能力问题，是输入端的问题。文档过期就等于给 Agent 喂了错误的约束。

**文档是冷启动 Agent 每次工作的输入源，必须在构建中通过同步门禁强力阻止文档与代码的签名漂移。** 代码仅记录当前实现，文档则补充了设计意图、背景与否决方案，这才是新接入的 Agent 摆脱低效试错的真正武器。

## 文档是怎么变成复利的

传统开发里，文档是写完代码之后顺手补的附属品，维护不维护全靠自觉。AI 驱动开发里，这个关系倒过来了：Agent 每次冷启动，第一件事是读 `AGENTS.md`、架构文档、API 参考，然后才动手。文档的质量直接决定 Agent 在新会话里能从哪个高度起跑。

一次踩坑，如果只留在当事人脑子里，下次另一个 Agent 启动时还会从零踩一遍。如果把踩坑结论写进 `AGENTS.md` 的约束条目，或者写进对应的参考文档，下次 Agent 读到约束就直接绕过去了。这是复利：沉淀一次，后续每个会话都省掉重复成本。

Harness 三件套（文档 + 检查 + 反馈）里，文档管的是输入侧。检查在代码离开仓库时拦截越轨行为，反馈在运行时提供校准信号，文档则在 Agent 进入任何工作之前就把约束注入上下文。把约束写成文档，比把约束写成 CI 脚本要低阻抗得多——不需要维护额外的检查逻辑，Agent 读到即执行。

但这个机制有一个致命前提：文档必须是活的，不能过期。

## 沉淀什么、怎么防过期

不是所有东西都值得写进文档，堆文字没有意义。要沉淀的是那些一旦丢失、后续 Agent 就会重复犯错的知识。

**Diátaxis 框架**把技术文档切成四类：教程（Tutorial）帮新人上手，操作指南（How-to）解决具体问题，参考信息（Reference）提供 API 定义和配置契约，解释性说明（Explanation）交代架构设计背后的取舍。这四类对应的读者不同、更新频率也不同。混在一起写是最常见的失控来源：把解释性文字塞进 API 参考里，或者把操作步骤写进解释文档，都会导致 Agent 查文档时拉到一堆无关内容，甚至读出相互矛盾的指令。

对 AI 驱动的项目来说，最关键的两类是参考信息（Reference）和解释性说明（Explanation）。Reference 文档记录接口签名、字段含义、配置格式——这是 Agent 生成代码时的直接依据，任何变更都必须同步更新。Explanation 文档记录架构决策：为什么选这个方案、否决了哪些替代方案、有哪些不能越过的红线——这是 `AGENTS.md` 背后的论据，也是唯一能让后来者不重复踩同一个坑的地方。

文档过期的根本原因是维护周期和代码变更周期脱节。解决方法是把同步更新变成必须过的门禁。Relay 在 CI 里引入了 `doc-hygiene-linter`，扫描 TypeScript 的 JSDoc 注解与对应 Markdown 文档块之间的签名一致性：

```javascript
// doc-linter.config.js
module.exports = {
  targets: [
    {
      codePath: 'src/payment/stripe-client.ts',
      docPath: 'docs/references/stripe-api.md',
      validators: ['method-signature-match', 'jsdoc-sync']
    }
  ]
};
```

一旦 Agent 改了接口签名但没更新文档，PR 阶段的 CI 检查会直接报错并阻断合并：

```bash
$ npm run lint:docs

[INFO] Scanning document contract configurations...
[ERROR] Doc Drift Detected: File src/payment/stripe-client.ts, Method "createRefund"
        The code signature has changed:
        Code: "createRefund(chargeId: string, amountCents: number, reason: string)"
        Doc: "createRefund(chargeId: string, amountCents: number)" (docs/references/stripe-api.md:L42)

[ERROR] Linter Failed: Documentation is out of sync with code signature.
[ERROR] Please update "docs/references/stripe-api.md" to match code changes.
```

这个报错在 Relay 的支付重构事故之后第二天就挂上去了。此后类似的文档漂移在进入主干之前就被拦住，再没有发生过"冷启动 Agent 按过期文档调接口"的循环。

## 待办交接：跨会话的延迟记忆

文档解决了已经固化的知识的传递问题。还有一类知识需要单独处理：推迟到下个迭代的工作。

Agent 会话没有记忆。一个会话里发现了问题、决定推迟处理，如果只是在代码里留一行 `// TODO: Fix notification`，下一个冷启动的 Agent 读到这行注释时，完全不知道这是什么问题、为什么推迟、有没有依赖前置条件、能不能直接删掉。

Relay 曾经因此付出过一次代价。开发人员在重构账单结算模块时，发现 Stripe 异步回调成功后发送 Slack 内部通知的逻辑在本地联调时总是超时。为了不耽误退款核心功能的 PR 合并，Agent 在代码里写了一行 `// TODO: Fix notification` 并把通知模块屏蔽，PR 合入了主干。

下个迭代，另一个被派发"重构订单回流"任务的 Agent 启动了独立 Worktree。它在做冗余清理时，看到这行被注释掉的通知模块，认定这是历史遗留的死代码，直接物理删除并合入了线上。Slack 通知链路彻底断开，引发了长达三天的财务同步断层——退款成功了，但内部没人收到通知，账单对不上。

这次事故确立了一条硬规则：**代码里严禁出现行内 `TODO` 注释。所有推迟工作必须提炼到根目录的 `TODOS.md`，并遵循五要素格式。** CI 静态扫描器对行内 `TODO` 注释视同语法报错直接拦截。

五要素是：做什么（Action）、为什么（Rationale）、技术边界（Scope）、前置依赖（Dependencies）、推迟缘由（Deferred Reason）。

```markdown
<!-- TODOS.md -->
## [DEFER] 自动退款拦截器向 VIP 策略倾斜

* **做什么**：将 `src/refund/decision-engine.ts` 中的额度检查改为按客户历史消费动态派生。
* **为什么**：VIP 用户的退款限额应放宽到 2000 元以改善大客户满意度，目前固定 1000 元过死。
* **技术边界**：修改 `decision-engine.ts` 并更新 `refund-decision.eval.ts` 里的 VIP 样本评分。
* **前置依赖**：需要先等 CRM 服务导出 `getUserTotalSpent` 的只读 API（`TODOS.md#L45`）。
* **推迟理由**：Stripe 自动退款 v1 刚上线，需要先用固定阈值运行两周以确认稳定性。
```

这五个字段让后续 Agent 在冷启动时能直接从 `TODOS.md` 派生出实施计划，不需要任何额外上下文：

```bash
$ npx gstack task-pickup --todo-id VIP_REFUND_UPGRADE

[INFO] Scanning TODOS.md for ID: VIP_REFUND_UPGRADE... Found.
[INFO] Checking dependencies: CRM service API. Status: READY.
[INFO] Synthesizing Implementation Plan...

--- Generated Plan Preview ---
1. Read file: src/refund/decision-engine.ts
2. Import client: src/clients/crm-client.ts
3. Update threshold evaluation logic from static 1000 to user-class derived
4. Update VIP eval sample in refund-decision.eval.ts
5. Verify E2E Playwright.

[INFO] Plan successfully generated. Proceeding to execution phase...
```

`TODOS.md` 是跨会话的延迟记忆。没有它，推迟的工作要么被遗忘，要么被误删，要么被新 Agent 以错误的方式处理。

## 阶段复盘：把教训变成约束

阶段复盘的目标是把反复出现的问题转化为下次绕不过去的约束，而不是写总结报告。

Relay 每个迭代结束后，团队会扫描当周发生的问题：哪些类型的错误出现了两次以上？哪些 Agent 行为是事后回头看本来可以被自动拦截的？然后做两件事：一是把能自动化检查的条件补进 CI；二是把不能自动化、但需要 Agent 在每次会话开始时就记住的约束，写进 `AGENTS.md`。

Relay 支付重构事故之后，CI 里多了 `doc-hygiene-linter`。`// TODO` 注释事故之后，CI 里多了行内 TODO 扫描器，`AGENTS.md` 里多了一条红线："所有推迟工作必须写入 `TODOS.md` 五要素格式，严禁使用行内 TODO 注释"。

复盘不等于事后总结。"我们下次要更谨慎"写进总结没有意义，因为 Agent 不会读总结报告。把"下次要更谨慎"具体化成一条 CI 检查规则或一条 `AGENTS.md` 约束条目，才会在下次 Agent 启动时直接生效。

Harness 就是这样建起来的：每次事故打一个补丁，补丁是 CI 规则或文档约束，下次绕不过去。积累足够多的补丁，系统就不容易在同一个地方摔倒两次。

## gstack 参考实现

在 gstack 体系中，文档生命周期管理与阶段复盘可以通过一系列自动化命令行工具来支撑：

- `/document-release`：在合并新特性时，通过静态分析提取 API 与 schema 变更，自动将最新的接口签名同步更新到相关的 Reference 文档中。
- `/retro` 与 `/learn`：在迭代结束的复盘会议后，工程师通过输入问题复盘情况，自动生成或更新本地 `AGENTS.md` 里的负向约束规则，完成模式的固化和归纳。

```bash
# 在发布阶段自动执行的文档变更检测与规范提取
$ gstack doc-release --diff-file ./src/payment/stripe-client.ts

# 本地沉淀与模式沉淀
$ gstack learn "Avoid float representation for stripe charge operations"
```

文档工具和流程是具体的实现载荷，可以根据实际使用的文档服务自由替换。其不变的实质是以 `AGENTS.md`（输入侧约束）、CI Linter（拦截卡口）与诊断迹线（反馈）拼合而成的 Harness 体系，在每次事故的迭代修复中持续提升 Agent 交付的安全基线。

## 典型反模式

行内 `TODO` 注释是最常见也最危险的反模式。这种注释会使冷启动 Agent 产生严重误读：当它看到被注释掉的代码时，往往无法识别这是推迟处理的业务逻辑，还是应当被清理的无用废弃代码，从而陷入混乱的排查中。

另一个高频反模式是文档更新和 API 变更脱钩，没有门禁保证同步。这让文档成为定时炸弹：越是稳定的老文档，越可能在某次重构后悄悄过期，等到下一个 Agent 冷启动读到它，问题才暴露。

还有一种更隐蔽的失误：复盘只停留在语言层面，没有转化为可执行的 CI 条目或 `AGENTS.md` 条目。团队讨论了问题、达成了共识，但没有写进任何 Agent 会读到的地方。两个月后团队换人，问题重新出现。

## 本章要点

* 文档管的是输入侧：Agent 冷启动时读到的约束质量，决定了它能从哪个高度起跑。一次踩坑沉淀成文档约束，后续每个会话都省掉重复成本。
* Diátaxis 框架把文档切成四类（Tutorial、How-to、Reference、Explanation），混类是文档失控的主因。对 AI 驱动项目，Reference 和 Explanation 优先级最高，且必须在 CI 里做同步门禁。
* `TODOS.md` 是跨会话的延迟记忆。推迟工作必须用五要素（做什么/为什么/技术边界/前置依赖/推迟缘由）显式登记，代码里禁止行内 `TODO` 注释。
* 阶段复盘的产出是 CI 检查项和 `AGENTS.md` 约束条目，不是总结报告。只有写进 Agent 每次启动时会读到的地方，教训才算真正固化。
* 文档 + 检查 + 反馈，三件套缺一不可。文档约束输入，检查拦截越轨，反馈校准运行时。这个闭环稳定下来之后，Harness 本身就在迭代中持续变强。
