# 设计文档先行

代码仅仅记录了最终的计算逻辑，却无法还原当初为什么做出如此设计、考虑过哪些备选方案、以及为什么否决它们。如果缺乏将这些决策和架构取舍显性化沉淀的实体设计文档，所有的设计动机将只留存在主写 Agent 某次临时会话的脑子里，随会话关闭而化为乌有。

**在写代码前，必须先写一份设计文档，把架构决策与设计取舍显性地记录下来。它让系统的演化意图变得可交接、可审计，保证后续冷启动的 Agent 能够安全迭代系统。**

这就形成了一条刚性的文档流：Spec（说什么）→ Design（怎么搭）→ 实现（代码）。每一环都是下一环的刚性输入，且在物理上留下清晰的可追溯链路。

其中，**Spec** 是 Plan 阶段将 Think 设计文档精炼后的工程规范，用于明确接口约定、边界参数与状态定义，回答“系统必须做什么、在何处拦截”；而 **Design（设计文档）** 是本章的主角，回答“系统如何在内部构建以达成 Spec 的约束”。两者层次不同，不可跳过前者直接写后者。

以 Relay 自动退款为例，Plan 阶段的工程评审将 Think 设计文档精炼为如下工程 Spec（ `docs/specs/auto_refund_spec.md` ）：

```markdown
## 核心能力
- 退款前必须调用 `billing_service.queryOrder`，确认订单状态为 `SUCCESS` 且未结清
- 调用 `billing_service.executeRefund` 执行扣减，流水写入 `refund_records`
- 所有网关请求头必须包含 `Idempotency-Key`

## 安全边界（硬卡口）
- 金额：单笔自动退款限额 `RefundLimit <= 5,000` 分；超出必须调用 `createApprovalRequest`，挂起至人工审批
- 状态：订单创建超过 90 天，或账户标记为 `SUSPICIOUS`，一律拦截至人工审查

## 架构选型
- 采纳：AI 意图分类 + 硬编码规则引擎兜底
- 否决：LLM 全权决策（原因：非确定性输出无法保证资金安全）
```

这份 Spec 是技术设计文档的直接输入：工程师据此设计内部模块的数据流、幂等机制与回滚链路。若一个冷启动 Agent 读完 Spec 后依然需要向人类确认“大额退款的具体上限是多少”，则说明 Spec 的信息密度尚未达到合格标准。

## 文档是冷启动 Agent 的唯一交接点

在 AI 协同研发中，设计文档具备一层额外的战略价值：它是跨越 Agent 记忆断裂带的交接棒。

大模型协同在本质上是无记忆的冷启动（Cold Start）过程。当一个新特性启动或一个新的 Agent 接手自动退款功能的后续更新时，它没有任何关于此前对话的心智上下文。此时，设计文档就是它读入并重建心智模型的输入端工件（Harness 中的“文档”支柱）。

如果缺乏设计文档，新接入的 Agent 只能采取两种低效的做法：要么从数千行现存代码中盲目反推设计意图（极易因理解偏差而猜错），要么退化为不断询问人类工程师，让团队重新陷入“人类解释意图”的效能瓶颈中。设计文档配合项目宪法，构成了项目可随时被任何冷启动 Agent 或新加入工程师接手的前提。

## gstack 设计文档模板参考

在 `gstack` 规范中，设计文档是连接 Spec 与代码实现的核心桥梁。一份规范的设计文档必须包含背景、架构取舍、核心逻辑定义、副作用与幂等设计、以及异常处理与回滚这五个核心板块。其标准骨架结构如下：

```markdown
# 设计方案: [功能名称] v[版本号]
# 关联需求: [指向关联的 Spec 文件路径]

## 1. 背景与目标
- 描述此设计所解决的工程痛点与业务价值。

## 2. 架构取舍 (Design Trade-offs)
- 列出曾考虑过的备选方案 A、B。
- 说明最终采纳当前方案的权衡决策依据（包括并发度、延迟、开发成本、系统复杂度等维度的损益评估）。

## 3. 核心流程与数据流
- 详细描述核心模块间的调用关系与时序。
- 明确标注同步接口与异步消息边界。

## 4. 副作用与幂等设计
- 梳理存在外部写副作用（如第三方支付、发送通知、数据库写）的所有环节。
- 详细阐述幂等键构造机制与在并发下的锁竞争防护。

## 5. 异常处理与降级回滚
- 定义各环节超时、网络错误等故障状态下的幂等重试规则。
- 阐述 Saga 分布式事务回滚或手动补偿链路设计。
```

## 主线演练：Relay 自动退款设计文档片段

在 Relay 示例项目中，当自动退款方案锁定后，Agent 在启动编码前，首先在 `./docs/designs/auto_refund_v1.md` 中产出并锁定了如下架构设计文档：

```markdown
# 设计方案: 异步自动退款系统 v1
# 关联需求: `Spec_Refund_v1`

## 1. 架构取舍 (Design Trade-offs)

### 同步退款 vs 异步消息退款
- **决议**：否决通过 HTTP Webhook 同步调用 `Stripe API` 退款的方案，采用「判定 -> 创建审批闸 -> 异步入队列 -> 异步消费者执行」的解耦设计。
- **权衡理由**：`Stripe` 外部退款接口在高并发时极易发生 504 延迟或超时。若在 Webhook 主线程同步调用，将长时间占用数据库连接池，导致整个客户支持渠道的接入进程因连接数爆满而雪崩。引入消息队列（ `RabbitMQ` ）后，我们将强一致性放宽为最终一致性，将外部调用隔离至独立消费者模块，确保接入主线程的高吞吐量与可用性。

## 2. 核心副作用边界与幂等设计
- **幂等规则**：下游退款执行器必须对每笔退款调度强验证 `stripe_refund_idempotency_key`。
- **构造算法**：`idempotency_key = sha256(order_id + refund_action_sequence)`。
- **回滚机制**：若执行器捕获到退款金额超支或账号受限，立即抛出 `RefundFailureEvent`，触发 `Saga` 事务回滚，解冻对应订单的可退款金额额度。
```

半年后，即使有全新的 Agent 想要重构退款逻辑，当它看到看似繁琐的多层队列和独立的 `Saga` 回滚引擎时，只要通读这份设计文档，就能瞬间打消“为了简化代码而误删异步隔离层”的愚蠢念头，保证了架构的安全演进。

## 典型反模式

* **边编码边拼凑架构**：完全不留实体设计文档，架构意图仅在临时会话中存活，随着对话会话的生命周期结束，项目架构随之失控并逐渐腐化。
* **事后补写“代码中文复述说明”**：代码写完后，为了完成任务而补写一份毫无技术细节的文档。这类文档往往只复述“定义了什么变量/类”，却没有记录“为什么不选其他方案”的取舍逻辑，毫无参考价值且极易因代码变更而过期。
* **文档缺乏负向权衡**：只记录“现在怎么做”，不记录“曾经考虑过什么、为什么在评估后否决了它”，导致后来的修改者无意中重踩前人已经避开的坑。

## 本章要点

* 必须遵循先写设计文档、再启动编码实现的严格顺序。
* 设计文档的战略重心在于记录架构取舍（为什么这么做、为什么否决其他方案）。
* 设计文档是冷启动 Agent 进行自助交接的首要载体，是整个项目具备“可无缝接手性”的技术底座。
