# 写出来就可诊断

当代码交付到线上环境运行后，传统的本地 Debugger 单步跟踪调试手段随之失效。在分布式系统和云原生架构中，线上问题的排查完全依赖于系统对外输出的指标与日志。可观测性并非系统上线后的被动运维补丁，而是必须在 Build 阶段与核心业务逻辑一同实现的刚性研发规避。

**新功能的研发必须将“写出来就可诊断”作为准入条件。每一次状态转移、网络 I/O 交互与失败分支的捕获，都必须在编写代码的同时埋设好对应的结构化日志。事后追加日志本质上是一种极具风险的排障负债——因为发生故障那一瞬间的上下文和状态已经在寂静中永远流失。**

## 结构化日志优于拼接文本

在 AI 参与的排障闭环中，严禁在代码中直接使用 `console.log` 或 `print` 等非结构化文本输出日志。非结构化文本不仅极难通过自动化脚本实施二次分析与字段过滤，也会使得排障 Agent 无法建立稳定的对象匹配逻辑。

### 1. Pino 敏感信息拦截与日志序列化配置

在 Relay 系统中，我们使用 `pino` 作为底层的统一结构化日志收集器。为了防止 Agent 在记录日志时无意中将用户的 PII 数据（个人隐私数据）或支付密钥打印出来，必须在初始化 Logger 时声明 `redact`（敏感字段擦除）拦截器：

```javascript
// src/common/logger.ts
import pino from 'pino';

export const logger = pino({
  level: process.env.LOG_LEVEL || 'info',
  // 敏感字段擦除配置，支持嵌套对象路径的静态与动态拦截
  redact: {
    paths: [
      'req.headers.authorization',
      'payload.card_number',
      'payload.cvv',
      'payload.stripe_secret',
      'payload.customer_message_raw' // 严格过滤用户原始聊天文本，防止泄露 PII
    ],
    censor: '[REDACTED]',
    remove: false // 将敏感内容替换为特定遮蔽占位符
  },
  serializers: {
    // 确保 Error 对象在序列化时不丢失调用栈（stack trace）
    err: pino.stdSerializers.err
  }
});
```

### 2. 精细的结构化日志输出迹线

依靠上述配置，当 Relay 在调用 Stripe API 扣款失败时，日志收集器会将带有分布式追踪上下文（`trace_id`）的事件完整归档为单行 JSON：

```json
{"level":50,"time":1782384102941,"pid":48201,"hostname":"relay-worker-01","trace_id":"t_stripe_refund_98274a","session_id":"sess_98127","event":"stripe_api_refund_failed","duration_ms":312,"payload":{"refund_id":"ref_82a3f","card_number":"[REDACTED]","stripe_secret":"[REDACTED]"},"err":{"type":"StripeInvalidRequestError","message":"The bank account has insufficient funds.","stack":"StripeInvalidRequestError: The bank account has insufficient funds.\n    at Stripe.makeRequest (/workspace/node_modules/stripe/lib/Stripe.js:143:12)\n    at processTicksAndRejections (node:internal/process/task_queues:95:5)"},"msg":"Stripe refund call returned non-200 status"}
```

## 日志设计的硬性规则

为确保每一次线上异常都具备可定位的追踪痕迹，代码编写必须贯彻如下规范：

* **关联 ID 全链路贯穿**：每一个 HTTP 请求、定时任务或队列消息在进入系统时，必须通过中间件在上下文（AsyncLocalStorage）中绑定唯一的 `trace_id`。所有派生的下游子日志必须默认携带该 ID，以确保能利用单一标识拼凑出完整的事务行进路线。
* **状态迁移和外部调用必记**：对第三方支付接口、向量数据库以及 LLM API 的网络请求发起前后，必须各自记录一条日志，并附加耗时指标。
* **保留 Error 原始对象**：严禁在 `catch` 块中通过 `err.message` 将异常转换为普通字符串记录。这会彻底丢失底层的调用堆栈，给排障带来灾难性的黑盒困境。

## 深度健康检查与存活心跳

可观测性不仅仅是从内部向外吐日志，还包含外部健康检查机制的链路深度。

在提供 REST 服务时，健康检查端点（如 `/health`）切忌只返回一个静态的 `{"status": "ok"}`。它必须实现对依赖存储（Database）、消息队列（Redis）的真实心跳探测。任何一个外部依赖连接断开，该端点必须返回 `503 Service Unavailable`，以便容器编排器（如 Kubernetes）或告警平台能及时发现“僵尸服务”并进行流量切除。

## 主线演练：Relay 退款防线与安全擦除挫败

在开发 Relay 自动退款判定的功能时，Agent 在 `stripe-client.ts` 模块中首版编写的代码包含如下逻辑：

```typescript
// 错误示范：Agent 贪图排障方便，直接将完整的错误对象与传参 payload 进行了混搭打印
try {
  const result = await stripe.refunds.create({ charge: payload.charge_id });
} catch (error) {
  logger.error({ payload, error }, 'Refund failed'); // 导致 payload.stripe_secret 直接打印
}
```

在测试阶段，大仓内置的 CI 日志合规性静态扫描工具（`log-hygiene-linter`）拦截了这次提交，并输出了如下错误：

```bash
$ npm run lint:log
[ERROR] Static Log Scanner: File src/payment/stripe-client.ts, Line 15.
Detected un-redacted payload logging. 
Variable "payload" may leak Stripe private credentials to production stdout.
Build Failed.
```

这次调试挫败促使开发人员不得不增加统一的 `pino` 拦截路径，并强制要求 Agent 仅解构需要记录的非敏感字段（如 `charge_id`），从底层彻底切断了机密信息泄露的可能。

## 典型反模式

* **事后补偿型日志**：当线上系统发生未捕获的内存溢出或退款漂移时，由于当时没有任何上下文打印，排障人员只能在代码中盲目插入几行 `console.log`，然后无奈地等待线上故障的下一次发生。
* **吞没异常堆栈**：在 `catch` 捕获异常后，仅打印了简短的 `System error occurred`，彻底将关键报错调用栈抹去，给后续定位造成人为的障碍。
* **健康检查“伪存活”**：健康检查端点只检测本进程 Node.js 是否存活，而对底层的 PostgreSQL 掉线、Redis 满载不闻不问，导致监控大盘绿意盎然，而线上服务早已因无法连接数据库陷入彻底崩溃。

## 本章要点

* 可观测性属于 Build 阶段的刚性纪律，代码必须在编写的当下就包含诊断路径。
* 严禁使用无格式拼接文本日志，全量日志必须通过 Logger 进行结构化 JSON 输出，并配置敏感字段擦除（Redaction）规则。
* 分布式追踪关联 ID（`trace_id`）必须贯穿所有并发及异步调用链，以作为还原事务链路的骨架。
* 深度健康检查端点必须探照底层依赖的物理连接状态，不得伪造存活指标。
