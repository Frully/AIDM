# 分层架构与依赖守卫

单体仓库（Monorepo）抹去了上下文屏蔽墙，但随之而来的风险是：如果没有物理红线约束，各个模块之间很容易发生杂乱的互相引用与循环依赖，导致项目架构迅速退化为高耦合的“意大利面”代码。

**必须在单仓内建立清晰的物理单向分层架构，并使用静态依赖分析工具在本地 Hook 与 CI 门禁中进行刚性校验，让任何越层或反向引用在编译/提交阶段直接被机器阻断。**

架构规则如果仅记录在文档中，只是一纸空文式的口头提醒，极易被赶进度的工程师或追求捷径的 Agent 绕过；将其写入编译和依赖检测流程中，化为机器可强制执行的物理拦截卡口，才是刚性约束。

## 物理单向分层契约

在 AI 驱动开发的体系下，大仓内部需要定立清晰的单向依赖路径。以下为一种经典的从低（底座）到高（消费端）的层级定义：

* **契约层（Contracts）**：仅包含纯数据结构（Types/Schemas），零运行时副作用与外部依赖。该层拥有最高级别的可引用性，可被大仓内任意其他包直接依赖。
* **基础设施层（Infrastructure）**：封装数据库操作、缓存客户端、消息队列、外部网关包装等底层通用能力。
* **业务编排层（Domain/Services）**：核心业务逻辑编排主场，可依赖基础设施层与契约层。
* **进程入口层（Apps/Gateways）**：各个独立运行的外部应用入口（如 Webhook 接入网关、坐席前台服务等），仅允许依赖业务编排层和契约层。

这套规则强制约束了依赖的演进方向：允许高层依赖低层，严禁低层依赖高层。同层的包之间如果存在共享逻辑，必须将共享部分剥离并下沉至公共契约或基础设施包中，严禁发生同层之间的随意循环导入。

## 守卫工具的负向测试

一个极易被研发团队忽略的安全隐患是：负责强制依赖规则的静态拦截守卫，自身可能会因为配置项写错或规则解析路径失效而沦为无意义的“哑门禁”。

如果我们没有为依赖守卫编写对应的“负向测试（Negative Testing）”，我们就会盲目信任这道防线，以为它正在正常保护架构，从而埋下严重的安全隐患。

所谓负向测试，是指在测试套件中主动构造一个被依赖守卫严禁的“越层引用用例”，并断言该检测工具必须抛出特定的错误退出码与拦截日志。只有当这个负向测试在本地或 CI 中被验证“确实拦截成功”时，我们才能确信这道安全护栏是实实在在起作用的。

## 主线演练：Relay 依赖拦截配置与负向拦截迹线

在 Relay 示例项目中，系统设计规定：“跨越渠道接入层与业务执行层的安全防线 ── 凡是直接操作 Stripe 网关进行出款的高风险业务包 `packages/workers/src/refund`，严禁被多渠道接入网关 `packages/gateway` 直接引用，必须经过中间的 Saga 审批编排层。”

为了让这一规则被机器强行约束，大仓引入了 `dependency-cruiser` 依赖扫描工具。以下为其具体的配置文件片段：

```javascript
// .dependency-cruiser.js
module.exports = {
  forbidden: [
    {
      name: "gate-no-direct-refund",
      comment: "多渠道网关接入进程不得绕过编排层直接调用自动退款执行模块",
      severity: "error",
      from: { path: "^packages/gateway" },
      to: { path: "^packages/workers/src/refund" }
    }
  ]
};
```

当负责渠道接入的 Agent 试图通过直接 `import { executeStripeRefund } from 'packages/workers/src/refund'` 绕过审批流时，在本地 `pre-push` 或者 CI 的执行阶段，静态检测工具会立刻给出如下拦截报错并终止提交流程：

```bash
$ npx depcruise --validate .dependency-cruiser.js packages/gateway

  ✖ gate-no-direct-refund: packages/gateway/src/webhook.ts -> packages/workers/src/refund/execute.ts
    多渠道网关接入进程不得绕过编排层直接调用自动退款执行模块

Error: Dependency cruiser detected 1 architecture violation. Build failed.
```

为了确保该守卫有效，测试团队在测试用例中故意向 `packages/gateway/__tests__/fixtures/invalid_import.ts` 写入越层引用，并运行校验测试。当测试断言输出退出码为非零且包含 `gate-no-direct-refund` 错误码时，该测试才宣告通过。

## 典型反模式

* **依赖约束流于文档与口头宣贯**：由于缺乏流水线物理拦截，Agent 在自动编写代码时极易为了少写几行桥接代码，而跨模块随意调用，导致架构在几个 PR 之后彻底腐化。
* **孤立使用依赖守卫但无负向验证**：依赖扫描规则的匹配正则表达式写错，导致其没有匹配到任何真实路径，门禁形同虚设，团队却在虚假的安全感中继续迭代。
* **直接依赖兄弟进程**：两个独立部署的网关进程为了复用一小段逻辑直接互相 import，不仅破坏了进程边界，也使得 Worktree 并行开发时经常发生因改动一侧而导致另一侧编译中断的冲突。

## 本章要点

* 大仓必须通过 `dependency-cruiser` 等静态分析工具锁定依赖演进方向，严禁低层代码或入口层反向依赖核心层。
* 规则只有写成“构建不过”的卡口门禁，对 Agent 而言才是机器可强制执行的刚性硬约束。
* 守卫工具必须配有负向测试，故意制造越层以确保这道架构防线处于灵敏状态。

> 依赖扫描守卫是刚性检查在架构层面的缩影。在大仓中，类型编译、单元测试、SLA 校验等其它多维度检查同样需要被整合成一体。我们将在第 13 章 `检查即门禁` 中讨论如何构建一道无法被绕过的自动化出厂大闸。
