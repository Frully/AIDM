# 写出来就可诊断

当代码交付到线上环境运行后，传统的本地 Debugger 单步跟踪调试手段随之失效。在分布式系统和云原生架构中，线上问题的排查完全依赖于系统对外输出的指标与日志。可观测性并非系统上线后的被动运维补丁，而是必须在 Build 阶段与核心业务逻辑一同实现的刚性研发规范。

**新特性的交付必须以结构化可观测性埋点为前置条件，确保在代码编写的同时完备记录交互链路上的 Trace 上下文。**

## 约束写在 AGENTS.md，实现交给 Agent

日志规范的最大问题不是"没想到"，而是"没写死"。把"要用结构化日志"放在团队 Wiki 里，两周后新的 Agent 上下文不带那份文档，`console.log` 就回来了。把约束写进 `AGENTS.md`，Agent 每个会话都会读到，违规时 ESLint 会直接卡住构建——约束才算落地。

Relay 的 `AGENTS.md` 里有这样一段日志约束：

```markdown
## 日志约束（硬性红线）
- 禁止在任何地方使用 console.log / console.error / console.warn
- 所有日志必须通过 `@relay/logger` 封装（统一输出格式）
- 每条日志必须包含 conversation_id 字段（关联业务对话上下文）
- 生产环境日志级别不低于 INFO；调试信息用 DEBUG，不能在生产留存
- 永远不能在日志里输出：用户消息原文、手机号、邮箱、支付卡号
```

这段文字是人类写的，讲的是业务决策，不是技术实现。Agent 拿到它之后做两件事：生成对应的 ESLint 规则，并实现 `@relay/logger` 这个封装包。

ESLint 规则是 harness 检查层里最轻量的一环，Agent 根据约束生成，人 review 后合入：

```json
// .eslintrc.json 片段（AI 根据 AGENTS.md 日志约束生成）
{
  "rules": {
    "no-console": "error"
  }
}
```

`@relay/logger` 的实现同样由 Agent 生成。接口设计（child logger 模式、必须携带 `conversation_id`）是在 `AGENTS.md` 约束里隐含的，Agent 把它落成 pino 封装：

```typescript
// packages/logger/src/index.ts — AI 实现，人类 review
import pino from 'pino';

const base = pino({
  level: process.env.NODE_ENV === 'production' ? 'info' : 'debug',
  formatters: {
    level: (label) => ({ level: label }),
  },
  redact: {
    paths: [
      'req.headers.authorization',
      'payload.card_number',
      'payload.stripe_secret',
      'payload.customer_message_raw',
    ],
    censor: '[REDACTED]',
  },
});

export function createLogger(context: { conversation_id: string }) {
  return base.child(context);
}
```

业务代码里的用法是：Agent 在每个 Service 里自动引入 `createLogger`，传入当次请求的 `conversation_id`，之后所有派生日志都自动携带这个字段：

```typescript
// packages/payment/src/RefundService.ts
import { createLogger } from '@relay/logger';

export async function initiateRefund(req: RefundRequest) {
  const log = createLogger({ conversation_id: req.conversationId });
  log.info({ event: 'refund.initiated', amount_cents: 2500 }, '退款流程启动');
  // ...
}
```

当 Agent 在其它地方尝试绕过封装，直接写 `console.log` 时，ESLint 会在构建时报错：

```
packages/payment/src/RefundService.ts
  47:5  error  Unexpected console statement  no-console
✖ 1 problem (1 error, 0 warnings)
```

这一行报错就是 harness 在发挥作用：人类在 `AGENTS.md` 里写了"禁止 console.log"，ESLint 强制执行，Agent 无法绕过。读者的职责是在 `AGENTS.md` 里写清楚"禁止 console.log"和"每条日志必须有 conversation_id"，然后让 Agent 把这些约束落地成代码和配置——执行是自动的。

## 没有 conversation_id 之前是什么样的

Relay 早期版本里，日志是这样的：

```
INFO  退款流程启动 amount_cents=2500
INFO  调用 Stripe API charge=ch_1a2b3c
ERROR Stripe refund call returned non-200 status
```

某天线上出现退款失败率异常。排障 Agent 拿到错误日志，只知道"有退款失败了"，不知道哪条对话触发的、是哪个用户、当时分类结果是什么。跨服务的调用链（分类服务 → 退款服务 → Stripe 回调）完全断开，Agent 只能在三个服务的日志文件里逐条比对时间戳，反复尝试对齐，耗了将近两个小时也没能确认根因。

加入 child logger 模式之后，同一条链路的输出变成：

```json
{"level":"info","time":1782384100000,"conversation_id":"conv_9f2a1b","event":"refund.initiated","amount_cents":2500}
{"level":"info","time":1782384101230,"conversation_id":"conv_9f2a1b","event":"stripe.request","charge":"ch_1a2b3c","duration_ms":null}
{"level":"error","time":1782384102941,"conversation_id":"conv_9f2a1b","event":"stripe.refund_failed","duration_ms":312,"err":{"type":"StripeInvalidRequestError","message":"The bank account has insufficient funds."}}
```

按 `conversation_id=conv_9f2a1b` 过滤，三分钟内就能把"分类→退款→Stripe 回调"的完整链路串出来，定位到银行账户余额不足这一根因。两者的差距不在排障能力，在于日志里有没有贯穿始终的业务关联 ID。

## PII 日志红线

Relay 处理用户的对话消息。分类服务、起草服务、工具调用服务都会接触到用户原文——打一条 `log.debug({ message }, '收到用户消息')` 太自然了，Agent 写起来毫无障碍。但用户消息原文一旦进了日志，就意味着它可能出现在日志采集平台、告警平台、甚至第三方监控工具里。

`AGENTS.md` 里的红线是这样写的：

```markdown
## PII 日志红线（绝对禁止）
- 禁止在日志里输出用户消息原文（conversation.messages）
- 禁止记录手机号、邮箱、卡号等用户个人信息
- 可以记录 user_id（内部 ID），不能记录用户的真实姓名
```

光靠约束文字不够，还需要两层机制来兜底。一是 pino 的 `redact` 配置（见前文），在序列化阶段把 `payload.customer_message_raw`、`payload.card_number` 等字段替换为 `[REDACTED]`——即使 Agent 把整个 payload 对象传进去，敏感字段也出不来。二是 Eval 测试里有专门的"日志安全"案例，验证 AI 生成的代码没有把用户原文写进日志：把一条含有用户消息的请求打进测试环境，断言日志输出里不能出现 `customer_message_raw` 的实际值。

这两层是互补的：`redact` 防御已知的字段路径，Eval 发现 Agent 用了未登记路径绕过 redact 的情况。

## 主线演练：退款 payload 泄露与拦截

开发 Relay 自动退款判定时，Agent 在 `stripe-client.ts` 里写了第一版：

```typescript
// 错误示范：Agent 把完整 payload 对象传进了日志
try {
  const result = await stripe.refunds.create({ charge: payload.charge_id });
} catch (error) {
  logger.error({ payload, error }, 'Refund failed');
  // payload 里含有 stripe_secret，直接进日志
}
```

CI 里的日志合规性扫描工具拦截了这次提交：

```bash
$ npm run lint:log
[ERROR] Static Log Scanner: File src/payment/stripe-client.ts, Line 15.
Detected un-redacted payload logging.
Variable "payload" may leak Stripe private credentials to production stdout.
Build Failed.
```

Agent 被打回修改。修复的方向不是"把 stripe_secret 从 payload 里摘掉"，而是只解构需要记录的非敏感字段：

```typescript
} catch (error) {
  log.error(
    { event: 'stripe.refund_failed', charge_id: payload.charge_id },
    'Refund failed',
  );
}
```

这条 lint 错误改变了 Agent 的编码习惯：从"传整个对象图省事"变成"只记录明确需要的字段"。后者是合规的，前者在 Relay 这种处理用户消息的场景里是不可接受的风险。

## 结构化字段与文本输出的区别

严禁在代码中直接使用 `console.log` 或 `print` 等非结构化文本输出日志。非结构化文本无法通过自动化脚本实施字段过滤，也使排障 Agent 无法建立稳定的对象匹配逻辑。

结构化日志里的每个字段都有名字、有类型、有约定：`event` 是事件名（用点号分隔的命名空间），`duration_ms` 是耗时（整数毫秒），`amount_cents` 是金额（整数分，不是浮点元）。这些字段名就是服务间的契约，告警规则、仪表盘 query、排障脚本都依赖它们。字段名改了，下游全断。把字段名写进 `AGENTS.md` 的约束里，Agent 生成代码时会遵守；字段名只在口头约定，Agent 迟早会用不同的命名。

光靠约束文字还不够——需要测试来保护字段名不被重构改掉。Relay 用 Ajv JSON Schema 测试锁定告警依赖的核心字段：

```typescript
// test/observability/log-contract.test.ts
const refundAlertLogSchema = {
  type: 'object',
  required: ['event', 'trace_id', 'error_code', 'refund_amount_cents'],
  properties: {
    event: { type: 'string', const: 'stripe_api_refund_failed' },
    error_code: { type: 'string' },
    refund_amount_cents: { type: 'integer' }
  }
};

test('告警日志必须包含外部监控依赖的字段', () => {
  const isValid = ajv.compile(refundAlertLogSchema)(dummyLogEvent);
  expect(isValid).toBe(true);
});
```

Relay 曾经有 Agent 在重构时把 `error_code` 改成了 `err_code`，外部告警查询式 `error_code="STRIPE_ERROR"` 立刻返回零条，整夜的支付失败完全没有触发告警。这个测试落地后，字段名变动会在 CI 阶段直接被拦截。

## 深度健康检查

可观测性不只是从内部向外吐日志，还包含外部健康检查的链路深度。健康检查端点（如 `/health`）切忌只返回一个静态的 `{"status":"ok"}`——它必须对依赖的数据库、消息队列做真实的心跳探测。任何外部依赖连接断开，该端点必须返回 `503 Service Unavailable`，让容器编排器或告警平台能及时发现"僵尸服务"并切除流量。Relay 的健康检查会同时探测 PostgreSQL 连接、Redis 延迟和 LLM 接口可达性——三者任意一个超出阈值，端点返回 503，不会假装自己活着。

## gstack 参考实现

在 gstack 体系中，结构化日志的规范由内置的 `@gstack/logger` 依赖包提供支持，其底层封装了高性能日志库 `pino`。

开发人员通过在 `AGENTS.md` 中约束统一的日志工厂函数，Agent 会在每个新建模块中直接引入此方法并派生 child logger。例如，定义包含业务关联 ID 的日志初始化逻辑：

```typescript
// src/payment/handlers.ts
import { getLogger } from '@gstack/logger';

export async function handlePayment(req: PaymentRequest) {
  // 派生出绑定唯一 conversation_id 的子日志实例
  const logger = getLogger({ conversation_id: req.conversationId });
  logger.info({ event: 'payment.started', amount: req.amount }, 'Processing stripe transaction');
}
```

这一套以 `@gstack/logger` 为核心的输出配置构成了具体的物理实现层。你可以按需替换为其它支持结构化 JSON 输出的日志工具（例如 Winston 或 Bunyan），其关键不变原则是输出格式必须保持字段结构化，且必须在每次状态转换和失败分支上透出一致的关联 ID。

## 典型反模式

事后补偿型日志：线上发生未捕获的内存溢出时，由于当时没有任何上下文输出，排障人员只能在代码里盲目插入 `console.log`，然后等待故障再次发生。没有 `conversation_id` 的日志更糟——即使有日志，也无法串联跨服务调用链，只能靠时间戳猜测关联关系。

吞没异常堆栈：在 `catch` 捕获后仅打印 `System error occurred`，把关键调用栈彻底抹去。正确做法是把原始 Error 对象传进 `log.error`，pino 会用 `pino.stdSerializers.err` 完整序列化类型、消息和堆栈。

健康检查伪存活：健康检查只检测 Node.js 进程是否存活，对 PostgreSQL 掉线、Redis 满载不闻不问，监控大盘绿意盎然，线上早已因无法连接数据库彻底崩溃。

## 本章要点

- 可观测性约束写进 `AGENTS.md`，用 ESLint 和日志合规扫描强制执行——Agent 绕不过去，比口头约定可靠得多。
- `conversation_id` 是 Relay 日志里的业务骨架：所有跨服务调用必须用它串联，排障时用单一字段过滤出完整链路。
- PII 防护两层兜底：pino `redact` 在序列化层拦截已知字段，Eval 日志安全案例发现未登记路径的泄漏。
- 结构化字段名是服务间的契约，字段名要写进约束，不能只靠口头约定。
- 深度健康检查端点必须探测底层依赖的真实连接状态，不能伪造存活指标。

> 可观测性数据已经就绪——结构化日志、关联 ID、字段契约测试一并到位。接下来的问题是：排障时如何让 Agent 直接够到这些线上数据？
