# 文档即复利

在 AI 驱动开发的模式下，文档不仅是写给人类阅读的参考文本，它更是喂给大模型的高频核心输入端。大仓的质量防线由三大支柱组成：**检查**、**反馈**与**文档**。当代码上线完成一次功能演进后，开发过程中积累的所有一次性工程结论（如架构设计考量、接口重构约束、线上排障 Runbook）必须被持久化沉淀，否则这些智力资产将随着会话窗口的关闭而在大模型语境中永远蒸发。

**文档沉淀本质上是大仓建设的复利资产。一次性的踩坑与排障结论一旦形成标准文档，将在后续成百上千次相互独立的 Agent 会话中高频复用，避免团队重复陷入“同一个工程陷阱，被不同 Agent 反复踩过”的低效消耗。**

## 为什么文档对 AI 而言是复利

在传统瀑布开发流中，写文档常被开发人员视为可有可无的额外负担。但在 AI 驱动的研发环境下，这一知识流向被彻底逆转：

```
+-------------------------------------------------------+
|  开发人员/Agent -> 踩坑/故障 -> 总结 Runbook/AGENTS.md   |
+-------------------------------------------------------+
                           |
                           v
+-------------------------------------------------------+
| 新 Agent 会话启动 -> 自动拉取 README/宪法并注入上下文    |
+-------------------------------------------------------+
                           |
                           v
+-------------------------------------------------------+
|         新 Agent 规避了相同问题，完成零成本防错          |
+-------------------------------------------------------+
```

检查负责拦截越轨行为（What can leave），反馈负责提供运行时校准（How it runs），文档负责对输入侧进行约束规范（What enters）。如果不在输入侧沉淀知识，我们将不得不把原本可以在提示词之外拦截的逻辑限制全部写成繁复的 CI 代码。这不符合软件工程的低阻抗原则。

## Diátaxis 文档架构与防过期守卫

文档老化的本质是其维护周期与代码变更周期产生了逻辑断层。如果让 Agent 参照过期的 README 去调用接口，它会在错误的 API 命名上反复重试直到耗尽 Token。

为此，大仓的文档结构必须采用 **Diátaxis 架构规范**，将文档精准切割为四种形态：
1. **教程（Tutorials）**：面向新人的上手项目。
2. **操作指南（How-to Guides）**：针对特定任务的步骤式说明。
3. **参考信息（Reference）**：API 定义、参数契约、配置文件描述。
4. **解释性说明（Explanation）**：系统架构设计权衡的背景说明。

### 1. 自动化文档契约校验配置

为了防范文档与代码不一致引起的退化，我们在大仓中部署了 API-to-Doc 差异扫描器 `doc-hygiene-linter.js`，通过提取 TS 代码中的 JSDoc 注解与 README 中对应的 Markdown 代码块进行静态对比：

```javascript
// doc-linter.config.js
module.exports = {
  // 定义需要严格同步的物理文件映射
  targets: [
    {
      codePath: 'src/payment/stripe-client.ts',
      docPath: 'docs/references/stripe-api.md',
      // 检测核心导出类或方法的签名与 markdown API 声明的一致性
      validators: ['method-signature-match', 'jsdoc-sync']
    }
  ]
};
```

### 2. 静态文档验证器在 PR 阶段的报错迹线

依靠该扫描器，一旦 Agent 在重构代码时遗漏了文档更新，CI 将当场阻断 PR：

```bash
$ npm run lint:docs

> relay-app@1.0.0 lint:docs
> node scripts/doc-hygiene-linter.js
```

## 调试挫败细节：卡在“老接口”上挣扎的两小时

在 Relay 系统支付重构期间，Agent 在 `stripe-client.ts` 中规范化了退款接口，将第三个可选参数 `reason` 升级为了必传项。

然而，由于当时未引入文档契约检查，Agent 修改了代码后，并未同步更新 `docs/references/stripe-api.md`。

半小时后，另一个独立启动的 Agent 分支拉起任务去开发“客服拒绝退款自动通知”的后台 Worker。该 Agent 严格按照 `stripe-api.md` 中的旧版调用说明，在 Worker 逻辑中编写了如下代码：

```typescript
// Agent 照着过期文档写出的错误调用
await stripeClient.createRefund(chargeId, amountCents); // 编译报错：Expected 3 arguments, but got 2.
```

该 Agent 由于上下文在冷启动时只读到了过期的文档，反复认为一定是本地包依赖发生了漂移，不断运行 `npm install` 并尝试重写导入路径，在死循环中自我排障了近两个小时，烧掉了大批额度。

这次调试挫败使得团队深受教训，并在当天的发布流程中强制挂载了 `doc-hygiene-linter` 门禁，确保任何 API 改动如果不随手更新对应的 Diátaxis 参考文档，便绝对无法合入主干。

## 典型反模式

* **垃圾堆型文档**：在根目录下随意乱放过期的 `.docx` 或临时 txt 记录，缺少 Diátaxis 架构规范，导致 Agent 启动时读入了一堆互相矛盾的过期知识。
* **孤岛式排障**：线上遇到大模型故障并解决后，不将排障 Runbook 沉淀回项目文档，导致后续发生同类事故时，新 Agent 又得从零人肉探索。
* **发布与文档脱轨**：API 发生了破坏性变更（Breaking Change），但发布合并时无需提供文档更新，导致大仓的文档库在发布第二周便沦为毒药。

## 本章要点

* 文档是 AI 驱动开发下的高效生产要素，与检查、反馈共同承载大仓的质量防线。
* 大仓文档应严格按照 Diátaxis 规范分类，防止知识混杂并便于自动化校验。
* 必须在 CI 中挂载文档与代码同步校验工具，防止 API 签名与接口文档发生逻辑偏离导致 Agent 产生开发死循环。

> 文档固化了已经达成共识的代码设计与线上 Runbook。但在项目快速前进的过程中，我们依然会有大量推迟处理的技术债（TODOS）以及每一迭代的改进决策。我们将在第 28 章 `遗留待办交接与阶段复盘` 中探讨如何让系统在迭代中具有连续的延迟记忆。
