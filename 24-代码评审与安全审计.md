# 代码评审与安全审计

CI 门禁与自动化 Eval 套件是拦截机械性代码缺陷与算法逻辑退化的钢筋锁链。然而，自动化门禁无法捕捉到架构层面的逻辑漏洞，例如在极端并发环境下的资源竞争、或因重构引发的间接性越权风险。因此，在交付链条中，使用人类专家（或干净上下文中的独立 Agent）进行人工级代码评审（Code Review）与安全威胁建模审计（Security Audit）是上线前的终极质量哨卡。

**代码评审必须在彻底物理隔离的干净上下文（Clean Context）中由独立的第二意见（Second Opinion）角色运行。严禁让主写代码的 Agent 在同一会话上下文里进行“自我评审与盖章通过”，因为模型无法自行跳出自身逻辑推演的盲区。**

## 分离设计：编写与评审双向隔离

大模型在自我纠错（Self-Correction）时极易产生盲目自信偏见。当一个 Agent 完成了代码改写，它的上下文被强力偏向于证明“当前修改完全正确”。如果我们在同一对话上下文中输入“请帮我评审刚才的代码”，模型只会以极高概率生成“逻辑完美，没有问题”的敷衍回复。

真正的评审必须严格执行双向隔离：
1. **上下文隔离**：开启全新的会话。评审 Agent 的系统提示词被灌注为“资深技术专家/恶意安全攻击者”，只接收被修改的 Git Diff 文本。
2. **模型隔离（推荐）**：使用与编写代码不同底座的模型。例如，若由 GPT-4o 编写代码，则必须使用 Claude 3.5 Sonnet 进行评审。不同模型的注意力机制与盲区相互补充，能提高逻辑漏洞的查出率。

## 安全审计框架：OWASP 与 STRIDE 对抗

在常规逻辑审查之外，必须独立加挂一道安全哨卡。我们结合两个经典框架进行系统防线加固：

* **OWASP Top 10 拦截**：针对 Web 应用的注入、越权、敏感信息泄露进行排查。
* **STRIDE 威胁建模**：从仿冒（Spoofing）、篡改（Tampering）、抵赖（Repudiation）、信息泄露（Information Disclosure）、拒绝服务（Denial of Service）、特权提升（Elevation of Privilege）六个维度对代码结构发起攻击视角审视。

针对 AI 驱动开发的产品，必须增加 AI 专有的第三个审计维度：**防注入绕过与越界工具调用**。

## 主线演练：Relay 自动化安全评阅迹线

在 Relay 项目中，开发团队部署了名为 `/review` 与 `/cso` 的自动化审查机器人。以下是机器人 `/cso` 在扫描退款执行模块时输出的安全警告迹线：

```bash
$ relay-ops audit pr-901

[INFO] Initializing CSO (Chief Security Officer) Audit Engine...
[INFO] Scanning Diff for Vulnerabilities (OWASP + STRIDE + LLM-Injection)

[WARNING] STRIDE - Tampering Detected:
          File: src/payment/stripe-client.ts
          Line: 24
          Detail: Method "executeDirectRefund(amount)" lacks permission validation. 
          A malicious user bypasses LLM controller and triggers raw refunds.
          
[WARNING] LLM-Injection / Privilege Elevation:

[SUMMARY] Audit Failed. 2 security warnings blocking merge.
[ERROR] Please resolve security threats before triggering production deployment.
```

## 调试挫败细节：漏洞百出的“一键退款”后门

在 Relay 自动退款链路的重构中，Agent 为了简化“用户投诉后直接触发补偿”的内部实现，在后台 API 拦截层直接留下了一个免授权端点：`/api/internal/refund`，打算只让大模型在决策判定通过后通过底层网关网络发起调用。

该 PR 的类型定义与自动化集成测试全部绿灯通过。然而，当安全审计机器人 `/cso` 在全新的上下文里对其进行 STRIDE 建模审查时，瞬间指出了致命漏洞：**该端点缺乏物理层面的 Session / JWT Token 签名鉴权。任何攻击者只要知道了退款 API 格式，就可以完全绕过前置的大模型判定引擎，直接构造原始 HTTP POST 请求，在线上刷空企业资金账户。**

这起严重的重构安全缺陷当场被红牌打回。开发团队随后在大宪章（`AGENTS.md`）中强制写入安全铁律：**所有外部调用接口必须有显式的 JWT 网关拦截层测试保护，决不允许存在任何形式的“内部免签直通车”。**

## 典型反模式

* **对话内盖章过审**：在原本写代码的同一个 Chat 窗口输入“你看这段代码没问题吧”，直接让带着满载逻辑偏向的 Agent 给自己放行。
* **重功能轻安全**：代码评审只关心“页面能不能加载出来”，而对并发下的重放攻击（Replay Attack）、越权漏洞以及大模型被诱导诱骗的逻辑缺陷视而不见。
* **忽略 CI 外的安全检查**：完全指望 ESLint 来抓逻辑安全漏洞，导致业务层面的金融安全和限额越界直接裸奔。

## 本章要点

* 代码评审是抓取并发竞争与复杂逻辑漏洞的终极防线，必须采用双向隔离原则（新上下文、新模型）。
* 安全审计必须结合 OWASP 与 STRIDE 建模，且必须针对 AI 特有的提示词注入、权限越界调用进行对抗审查。
* 严禁越过鉴权网关建立任何形式的内部通道，所有风险暴露接口必须具备硬性的物理测试防护。

> 评审与安全审计为代码本身提供了逻辑锁。但程序在真实环境中的动态运行表现，依然需要经过物理演练。我们将在第 25 章 `Agent 自验证与业务验收` 中探讨如何让 Agent 代替人类亲自跑起浏览器进行端到端验证，以及如何用业务 KPI 进行最终验收。
