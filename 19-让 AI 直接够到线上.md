# 让 AI 直接够到线上

有了完善的可观测性数据与外部告警系统，一旦线上发生故障，我们的目标是尽可能压缩故障定位与解决（MTTR）的时间窗口。在传统的研发模式中，线上环境的排障完全依赖于人类开发者登录跳板机去分析日志。在 AI 驱动开发的模式下，最高效的排障手段是为 Agent 提供一套受控的安全通道，使其可以直接查阅线上反馈数据，自主完成定位与代码修复。

**反馈链路必须向 Agent 提供闭环访问通路。通过封装最小只读权限的包装命令行工具，允许 Agent 在被授予操作令牌时，自主调用命令拉取线上对应 Trace ID 的上下文日志。严禁以“手工拦截并人工复制日志片段给 Agent”的低效沟通模式维持排障工作，这会使人类开发者沦为昂贵且迟钝的“人肉管道”。**

## 安全防线与最小只读权限设计

让 Agent 能够拉取线上反馈，绝不等于向其敞开未经限制的安全后门，这会给生产系统带来灾难性的逻辑风险。我们必须采用严格的“开窗而非拆墙”的安全架构原则：

* **完全的只读权限限制**：给 Agent 的命令行通道或 API 客户端只读权限。Agent 绝不应拥有在生产数据库上执行 `UPDATE`、`DELETE` 或对容器执行物理重启的操作权。
* **防注入的 API 包装**：不直接向 Agent 暴露原始的数据库连接器或 SSH 接口，而是向其提供一层进行了防参数注入（SQL Injection/Shell Injection）封装的专用只读查询脚本（如 `relay-ops`）。
* **基于双签的危险操作护栏**：对于确实需要执行紧急变更的操作（如刷新特定内存缓存以恢复被阻塞的订单流），命令执行时必须产生确认机制，必须要求人类开发者在终端显式输入 `y/n` 以授予物理授权许可。
* **全量审计日志归档**：Agent 发起的所有线上查询命令，必须在审计中心留下不可篡改的日志，以便于合规性回溯。

## 主线演练：Relay 安全排障 CLI 调用迹线

在 Relay 示例项目中，当监控系统抛出 `RelayRefundFailed` 告警时，排障 Agent 通过获取的安全只读 CLI，在本地执行了查询尝试。

以下是 Agent 运行的排障命令与终端交互的完整迹线：

```bash
$ relay-ops logs search --trace-id t_stripe_refund_98274a --limit 5

[INFO] Authenticating agent session... Done (Session ID: s_agt_91823)
[INFO] Executing read-only query against log index [logs-relay-prod-*]
[INFO] Query filter: { "trace_id": "t_stripe_refund_98274a" }

--- Log Record #1 ---
Timestamp: 2026-06-25T09:41:02.004Z
Logger: src/payment/stripe-client.ts
Message: Stripe API refund request failed. Status Code: 400
Payload: { "charge_id": "ch_981273", "error_code": "PARAMETER_MISSING" }

--- Log Record #2 ---
Timestamp: 2026-06-25T09:41:01.890Z
Logger: src/refund/decision-engine.ts
Message: Decision rules applied. Recommendation: AUTO_REFUND
Payload: { "refund_amount_cents": 12000, "user_flag": "VIP_USER" }

[INFO] Query complete. Total 2 matching logs retrieved.
```

通过这一层只读的 CLI，Agent 能够瞬间分析出故障的核心原因：决策引擎做出了 `AUTO_REFUND` 决定并调用了支付端，但支付端在向 Stripe 发起扣款请求时由于参数缺失（`PARAMETER_MISSING`）导致 400 报错。这一排障完全不需要人类开发者介入去当“日志搬运工”。

## 调试挫败细节：巨大模糊查询导致的 OOM 灾难

在早期设计中，我们给 Agent 暴露了一个宽泛且无硬性护栏的日志查询工具。在一例“Stripe Webhook 延迟处理”的线上异常排障中，Agent 为了定位故障范围，在没有配置时间范围与 Trace ID 过滤器的情况下，直接执行了全表扫描指令：

```bash
# Agent 在不设防的 CLI 中发起了全量模糊检索
$ relay-logs-admin query "Stripe Webhook" --limit 500000
```

该模糊查询瞬间击穿了测试网关，并导致远端 Elasticsearch 的单节点因为扫描海量 JSON 文档而内存暴涨，最终触发系统 OOM 宕机。同时，由于拉取了近 1.2 GB 的冗余 JSON 日志，瞬间撑爆了 Agent 自身的 LLM 传输上下文（Context Window），导致其由于 Token 溢出在终端直接崩溃。

这次由于权限不设防带来的生产停机事故痛击了团队。此后，我们重新设计并封装了排障工具链，对所有对外导出的查询命令在 CLI 入口强加了 `--limit 100` 的物理拦截护栏，并彻底禁止了未带 `trace_id` 或精确时间分片的全局正则搜索，防止此类人为逻辑故障再次破坏系统。

## 典型反模式

* **人肉日志管道**：因为数据合规名义封锁了全部日志查看端点，开发人员必须在微信中手动将 Kibana 日志截图或拼接的文字粘给 Agent，导致团队效率退回到石器时代。
* **过度暴露读写特权**：为图一劳永逸的安全设置，直接将生产环境的 AWS IAM / GCP 管理员密钥赋予 Agent 进程，使得 Agent 遇到编译逻辑漏洞时可能会物理清理掉生产数据库表。
* **查询缺失硬性护栏**：对于检索工具未作多级限流与 Limit 长度剪枝配置，导致 Agent 频繁由于上下文溢出或触发大额 Token 账单而无法连续作业。

## 本章要点

* 反馈只有被 Agent 够到才具备工程现实意义，必须为 Agent 提供直接的线上可观测数据访问能力。
* 通道包装要遵循最小特权、只读优先、基于令牌的安全审计策略，杜绝直接暴露写操作权限。
* 必须在专用 CLI（如 `relay-ops`）中强制实施分页、检索限制（`--limit`）与多重拦截护栏，规避因为模糊全局查询把大盘跑垮的“大模型拒绝服务”灾难。

> 可观测性的反馈回路已在 Build 阶段闭环就绪。这使得我们的系统对线上波动具有了极强的透明度。然而，对于复杂的 AI 自动化客服系统而言，规则和测试并不能保障模型输出文本的质量与稳定性。我们将在第七部分中，讨论如何为系统引入专业且刚性的 Eval 评测体系。
