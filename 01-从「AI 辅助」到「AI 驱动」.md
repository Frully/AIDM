# 从「AI 辅助」到「AI 驱动」

用 AI 编写代码，最常见的场景是：你在编辑器中输入，它补全下一行代码；遇到阻碍时，切换到聊天窗口询问，再将代码粘回编辑器。这确实节省了敲击键盘的时间，但主导代码库前进的依旧是你。研发的吞吐量与推进速度，完全受限于你个人的认知带宽。

这种模式被称为 **AI 辅助**。而在 **AI 驱动** 模式下，研发的心智模型将发生根本性的转变：你不再逐行编写代码，而是将完整的特性开发或重构任务交付给 AI 独立执行。你的核心职责转换为：为它划定明确的代码边界、构建能够自我纠正的反馈环境，并在终点进行严格的方案与工件评审。

**AI 驱动研发的本质是角色的转换——你从直接产出代码的作者，变成了设计约束、提供反馈和评审产出的环境设计者。可靠的交付不再依赖精雕细琢的提示词，而依赖你为 Agent 搭建的研发脚手架。**

关于环境脚手架的具体搭建与配置原则，全书在后文将逐一展开。在此之前，我们必须先理解并接纳这一角色的本质转变。

## 为什么：从作者变成环境设计者

模式的转变在于研发瓶颈的转移。

在辅助模式下，你依然是系统的瓶颈。即使 AI 补全速度再快，你仍需逐行检查生成的每一行代码。你扮演着“作者”的角色，而 AI 仅是一支更快的画笔。

在驱动模式下，研发瓶颈转移为两项核心能力：**环境脚手架的设计质量**，以及**方案与工件的评审效率**。当 AI Agent 在三个独立的 `worktree` 中并发推进不同特性时，逐行盯代码不仅低效，而且不可行。你需要做的是在它开工前铺设好坚实的轨道：明确哪些是不可逾越的红线、配置好必须通过的代码门禁，并定义清晰的“完成标准”（DoD）。你仅需在关键节点阅读它的技术提案、核对验证报告，并决定是否允许合并。

心智模型的转换伴随着阵痛。你必须放弃对每次键盘敲击的绝对控制权，转而刻意练习两项新技能：一是将模糊的业务边界转化为可由机器自动化强制拦截的刚性约束；二是培养出在不关注具体编码细节的前提下，快速研判代码合规性与边界正确性的评审直觉。部分编码经验极其丰富的开发者在此处往往会感到严重不适，因为惯用的编码肌肉记忆被迫失效，而新的工程习惯尚未建立。

前期成本也是必须考量的因素。构建这套约束、卡口与 DoD 规范需要极大的前期投入。这套方法论并不适用于“写个一次性脚本”这类短期任务，其红利主要体现在中大型或长期维护的项目中，前期的脚手架投入将分摊在未来的数千次迭代中。

## 约束胜过提醒

让 AI 产出可靠的杠杆并非精雕细琢的提示词，而是刚性的环境约束。

提示词本质上仅是**提醒**。随着对话上下文的拉长，模型极易产生漂移；一旦重开会话，之前的口头叮嘱就会被彻底清空。你在系统 Prompt 中写入“大额退款必须转人工审批”，可能在前两次测试中表现正常，但在第三次就可能因为上下文拥挤而被模型忽略。

相比之下，**约束**是无法绕过的物理栅栏。将“金额采用整数分（Cents）、禁用浮点数”定义为一条 Lint 静态规则或强类型约束，Agent 一旦写出浮点数，构建流水线就会报错中断。将“超出交易阈值必须路由至人工队列”写成框架底层的一道核心路由闸门，Agent 便绝无可能绕过审批直接调用扣款接口。

每当 Agent 出现交付偏差时，第一反应不应是“我该如何把提示词修改得更详尽”，而是“我该如何将这条规则转化为它无法绕过的物理约束”。

## 主线演练：Relay 退款特性的两种开发路径

以虚构的 AI 客户支持自动化 SaaS —— Relay 为例。该产品有一项核心特性：允许 AI 自动根据客户对话历史发放小额退款。

在**辅助模式**下开发该特性，你是代码的实际编写者。AI 帮你补全 `calculateRefund()` 函数、回答第三方支付的 API 参数格式，而你必须集中全部精力，人肉盯防退款金额计算是否溢出、高风险审批是否被漏掉。你成了软件交付的最后且唯一一道防线。

在**驱动模式**下，你不会直接编写退款逻辑，而是集中精力为 Agent 搭建如下的研发环境：
1. **定制约束配置**：在项目根目录的 `./AGENTS.md`（Agent 宪法）中明确写入业务红线：
   ```markdown
   ## 资金安全约束
   - 所有退款金额必须使用整数分（Cents），严禁出现浮点数计算。
   - 自动退款限额为 10,000 分（100 元）。超出此金额必须调用 `createApprovalRequest`，严禁直接调用自动退款服务。
   ```
2. **集成刚性检查门禁**：在 CI 中配置两项静态拦截：
   * **依赖守卫（Dependency Guard）**：禁止低特权渠道模块（如 `src/channels/webhook.ts`）直接 `import` 高特权的底层扣款模块 `src/services/refund.ts`，强迫所有退款操作必须通过风控网关层。
   * **黄金数据集测试（Golden Evaluators）**：包含重试逻辑测试。当网络波动、网关超时重试时，验证退款接口是否存在重复退款。
3. **制定自动化 DoD（完成标准）**：配置测试脚本，要求 Agent 启动无头浏览器模拟完整的沙箱退款流，核实退款金额计算与拦截行为，并输出测试报告。

在此环境就绪后，你启动 Agent 开始执行任务。在开发过程中，Agent 尝试编写的退款逻辑出现了两次逻辑偏差：
* **偏差一**：它在分摊优惠券后使用了浮点除法，导致金额计算产生小数。在提交代码时，CI 中的 Lint 卡口当场抛出：
  ```bash
  [Lint Error] float_usage_detected: float operations are forbidden in src/services/refund.ts (Line 42)
  ```
  Agent 捕获这一报错迹线，自动重构为整数除法并处理了余数。
* **偏差二**：它在下游接口超时重试的逻辑中漏掉了幂等键（Idempotency Key）。在运行 CI 黄金测试时，超时重试用例失败：
  ```bash
  [Test Failure] expected 1 refund transaction, but found 2. Idempotency violation.
  ```
  这一反馈让 Agent 意识到需要从会话 ID 生成唯一的幂等标记，最终修正了漏洞。

最终，Agent 完成了开发并通过了所有本地和 CI 的刚性门禁。你审阅了它的改动方案与自动化验证报告，确认合规后将其一键合并上线。

在这种范式下，真正拦截住超额退款或重复交易隐患的是你设计的**环境门禁与物理闸口**，而非你在评审代码时转瞬即逝的个人警觉。你所交付的内容也发生了升维：你没有直接编写退款模块，而是编写了一套能让 Agent 安全构建退款功能的“高压跑道”。当未来需要接入发券、发放积分等其他高风险特性时，这套跑道依旧可以稳定服役。

## 典型反模式

* **盲信 Agent 的魔法效应**：对项目不设任何物理边界与校验卡口，仅输入一句“帮我生成一个客服系统”便等待完美交付。缺乏约束与 DoD 验证的产出，其代码往往外表光鲜，内部千疮百孔，根本无法在生产环境中稳定运行。
* **伪「AI 驱动」**：仅将 Agent 作为高级版的代码自动补全笔与技术问答机，自己依然在逐行审查并修改代码，沉溺于“作者”角色而未能释放研发吞吐量。
* **无限膨胀的提示词泥潭**：试图通过堆砌数千字且不断增长的 Prompt 来规避错误，而不是将业务红线提炼成代码层面的 Lint 规则、类型签名与黄金测试。模型很快会在冗长的上下文中丢失记忆，导致同一个漏洞反复出现。

## 本章要点

* **心智范式升级**：AI 辅助是以人为主体、AI 协助编写；AI 驱动是以 AI 为执行主体、人扮演环境设计者与评审者。研发瓶颈从编码带宽转移为环境搭建质量与方案评审直觉。
* **环境大于提示词**：让交付质量可靠的杠杆是机器可强制执行的物理环境，而非复杂的提示词设计。
* **约束胜于提醒**：文字提醒极易在长上下文中产生记忆漂移，而刚性的 Lint 规则、类型守卫与自动化卡口不可绕过。当 Agent 犯错时，第一反应应是“如何将要求固化为约束”。
* **合理的队友心智**：将 Agent 视为一个工具链完备、具备短期记忆但上下文有限、且高度需要底线约束的“数字协作者”，而非神仙法宝。
