# 写作规范（写每一章前必读）

目标：28 章风格统一、好学易读。本规范是硬约束，写正文时逐条对照。

## 一、体裁定位：这是「解释型」文档，不是操作手册

按 Diátaxis，技术文档分四类：tutorial（学技能）、how-to（解决具体问题）、reference（查事实）、explanation（懂为什么）。**本书主体是 explanation——讲清原理、取舍与心智模型**，辅以少量 how-to（gstack 怎么落地）。

- **最大忌讳：一页里混类型，但不要走向「纯理论说教」。** 别把「手把手第 1 步第 2 步」塞进讲原理的正文里，但**原理必须用硬核的工程细节来承载**。讲「为什么」时，多用具体的报错信息、配置片段来证明，而不是空谈概念。操作细节归到「gstack 参考实现」小节或附录。
- 我们要读者**读完能理解并迁移**，不是照着敲一遍就完事。所以重在动机、模型、对比、反例，而非流程清单。反例和动机必须具备「程序员的粗砺感」，不能是教科书式的理想情况。

## 二、统一章节模板（每章都按这个骨架写）

**骨架是逻辑顺序，不是排版枷锁**。每章的逻辑应该包含这些关键要素以降低读者的认知负担，但**禁止千篇一律地使用相同的 H2 标题套用**。

1. **钩子（1–2 段）** — 从一个真实痛点或反例切入，点明「不这么做会怎样」，回答「我为什么要读这章」。不要用「随着 AI 的发展……」这种空开头。
2. **核心论点（1 句，加粗）** — 把本章原则一句话讲死，且**可迁移**（不绑定工具）。一章只讲一个核心论点，别塞两个。
3. **为什么 / 取舍** — 本章主体。讲原理、讲反直觉的点、和替代方案对比（为什么不那样做）。这是「解释型」的价值所在。
4. **Relay 佐证（worked example）** — 用主线示例把原则演成一个**完整、具体**的场景。尽早给、给完整：worked example 对入门最有效。
5. **gstack 参考实现** — 落到工具，并明说「这是实现层、可替换」。保持简短，别变成操作手册。
6. **反模式** — 不这么做的典型翻车（1–3 条）。
7. **本章要点（3–5 条）** — 可带走的结论，便于回顾。

- **允许有机融合**：比如，Relay 的具体代码演练可以直接穿插在「为什么」的对比中，不一定非要单列 `## Relay 佐证`。
- **篇幅不强求对称**：简单的概念用 1200 字讲透即可，复杂的实践可以写到 3500 字，不要为了字数对称而强行拉长或缩短。

## 三、句子与段落

- **先给结论（BLUF）**，再展开。别让读者读到段尾才知道你要说什么。
- **具体 > 抽象**：点名文件、命令、数字、真实的失败现象。「评分数字用 16px」胜过「字号要克制」。
- **主动语态、短句、一段一意**。一段只讲一件事。
- **每个新术语第一次出现给一句话解释**（gloss），即使读者可能已知。
- 善用**对比表格 / before-after**；拓扑、流程、依赖关系**用图**，别用大段文字描述结构。

## 四、术语与排版（简体中文）

- **中英文之间加空格**：`AI 驱动`、`16px`、`gstack 技能`。
- 文件名 / 命令 / 代码标识符用反引号：`AGENTS.md`、`pnpm depcruise`。
- **术语全书统一**（用同一个词指同一个概念）。关键词：harness（脚手架）、文档 / 检查 / 反馈、承重柱、原则层 / 实现层、worktree、Eval、信任校准。建立并维护一份术语表，新术语先入表。
- 标题层级：章标题已是文件 H1；正文用 `##` / `###`，不超过三级。

## 五、避免 AI 味（重点）

「AI 味」主要不是用词，是**句式套路**：一种读着顺、对仗工整、但没增加信息的腔调。下面每条都是高频 AI 腔，写完逐条自查、逐条删。

1. **「不是 A，是 B」对仗滥用。** 用反义对照制造节奏，是最明显的 AI 腔。一章最多用一次。
   - ✗ 这不是补全代码，是驱动整件事；瓶颈不在手速，在评审。
   - ✓ 你不再逐行写，而是把一整块活交给它完成。（直接陈述，不摆对仗）

2. **三连排比强迫症。** 习惯把任何东西都凑成三件套。该几条就几条，常常两条或四条更真。
   - ✗ 你的注意力、你的手速、你能同时装下的东西。
   - ✓ 瓶颈是你：得读完每一行才放心。

3. **破折号上瘾。** `——` 一页最多一两个。大多数能用逗号、句号或括号替掉，或干脆把句子重组。

4. **加粗满天飞。** 一章只给那一句核心论点加粗。每段都标重点，等于没有重点。

5. **戏剧性单句成段。** 「这叫 AI 辅助。」「杠杆在环境。」偶尔可以，连用就成了腔调。

6. **元叙述、自我指涉。** 别向读者解说自己的结构。删掉「这一句是全书的地基」「后面 27 章都在回答这个问题」「接下来我们来看」。

7. **信号词清嗓子。** 「值得注意的是」「最反直觉的一点是」「说白了」「关键在于」「事实上」——删掉，直接说那件事。

8. **复述自己。** 讲完一段不要再「一句话：……」总结一遍。读者读得懂。

9. **空泛强化词。** 真正的 / 本质上 / 归根结底 / 某种程度上 / 所谓 / 其实。删了句子更硬。

10. **工整的权衡对偶。** 「你交出 X，换来 Y」这种平衡句，偶尔有力，成串就假。

11. **「大厂 PPT 腔」与「虚空导师说教」。**
   - ✗ 你的新角色是三件事：设计约束、建反馈回路、评审产出。这叫监督式自治。
   - ✓ 实际上，你不再是写代码的作者，而是那个制定 CI 门禁和审批规则的人。
   - **正向做法**：采用「资深工程师写事故报告 (Incident Report)」的语感——冷静、客观、充满技术细节。不要走向另一个极端：切忌使用「技术网红/YouTuber」式的轻浮口吻（如“想象我们在咖啡厅聊天”、“那么问题来了”）。保持技术文档的硬度，用事实而非口号服人。

12. **缺乏技术细节的「悬浮案例」。**
   - ✗ 驱动模式下你一行退款代码都不写，先搭环境：红线写进 AGENTS.md，检查接进 CI。
   - ✓ 驱动模式下，你先在 `AGENTS.md` 里加一行 `- 金额使用整数分`。当 Agent 尝试使用 `amount: 12.5` 时，CI 中的自定义 Linter 会直接抛出 `Error: Float amounts are forbidden` 强行拦截。

13. **项目符号（Bullet Points）泛滥与信息稀释。**
   - AI 极度依赖列表来组织信息，常常把原本可以用一句高密度长句讲完的话，稀释成三个凑数的无聊条目。
   - **正向做法**：能用一段话（Paragraph）讲清楚的逻辑，绝不拆成列表。列表只用于真正并列的、需要快速检索的硬性规则或枚举。

14. **虚假的「情绪过渡」与「逻辑路标」。**
   - AI 喜欢用毫无信息量的废话来串联段落。例如：「那么问题来了，我们该如何...」、「这背后反映了一个深刻的道理...」、「正如我们在前文所看到的...」。
   - **正向做法**：直接进入下一段的主题。好的逻辑过渡靠的是上下文意义的自然咬合，而不是靠「清嗓子」的路标词。

**词级限制与合理使用规则**：
- **纯空话与大厂 PPT 腔（必须禁止）**：`赋能`、`抓手`、`打法`、`对齐`（仅限滥用如“对齐认知”；物理或格式对齐保留）、`拉通`。
- **AI 标志性无意义词/清嗓子词（必须禁止）**：`众所周知`、`不难发现`、`综上所述`、`显而易见`、`值得注意的是`。
- **营销/夸大词汇（必须禁止）**：`强大的`、`革命性的`、`完美的`。
- **技术术语语境下的合理放宽（切忌机械抹杀）**：
  - **闭环 (Feedback Loop)**：禁止空泛的“形成闭环”等口癖。但**允许且推荐**在控制工程、反馈系统和排障流程中使用“反馈闭环”、“闭环自愈回路”、“线上排障闭环”、“Ingress 灰度回滚闭环”等具体物理和工程概念。
  - **健壮性/鲁棒性 (Robust)**：禁止“非常 Robust/健壮的系统”这种空洞的形容词。但**允许**在架构设计中讨论“系统鲁棒性”、“健壮的异常处理”等关于错误容忍度的严谨表述。
  - **关键 (Crucial) & 全面 (Comprehensive)**：禁止“这至关重要 (crucial)”或“全面地 (comprehensive) 展示了”这种拉长句子的废话。但**允许**使用“关键路径”、“关键门禁拦截点”、“全面的回归测试套件”等明确修饰技术客体的语境。


**正向做法**：句子长短错落（别一个节奏到底）；把话说死，少对冲；让内容自己承重，少信号词和加粗。写完通读一遍，凡是「读着顺但没多给信息」的修辞，删。

## 五之二、从成稿诊断出的高频模式（2026-06-25）

以下 5 条在全书 28 章里反复出现，润色时优先清掉：

**1. 章节引用型元叙述。** 正文里写「这是第 X 章的重演 / 主场 / 落地」「这呼应第 X 章」「这和第 X 章是同一个形状」。读者读当前章时不需要知道这是哪章的重演，直接说原则就行。章节编号只留在「本章要点」之后的过渡句里，或者附录的查阅表里。
- ✗ 这是第 8 章「作者与评审分离」在代码层面的重演。
- ✓ 代码评审和计划评审同理：换干净的上下文，不让作者给自己盖章。

**2. `工具可换，原则不变` 公式。** gstack 参考实现节几乎每章以这句结尾，已变成空话。改成说明哪几条核心行为不变，或直接省略。
- ✗ 工具可换，原则不变：结构化、带关联 ID、有意义的事件就记。
- ✓ 换什么工具都行，结构化日志、关联 ID、失败分支必记，这三条保住。

**3. 正文里预告接下来多少章讲什么。** 「这一章先立住 X，接下来三章讲：Y（第 21 章）、Z（第 22 章）、W（第 23 章）」是目录式元叙述。直接进内容；尾部只保留一句话「下一章讲…」。
- ✗ 这一章先立住「为什么必须有」。接下来三章讲怎么做：套件怎么设计（第 21 章）、怎么接进 CI（第 22 章）、上线后怎么盯（第 23 章）。
- ✓ [直接展开内容] … 下一章讲怎么设计一套有用的 Eval 套件。

**4. `不是 A，而是 B` 超过一次。** 第五节第 1 条已规定一章最多一次。成稿部分章节出现两次以上。第二次改成直接陈述。
- ✗ [同章内第二次] 不是「输出等于某字符串」，是「输出落在哪个档」。
- ✓ 断言的是行为，不是字符串。

**5. `这正是 / 这也` 引出的交叉引用。** 「这正是第 2 章说的『约束 > 提醒』」「这也呼应信任校准」——联系已经在内容里了，说破反而弱。把联系移掉，直接表达逻辑即可。
- ✗ 这正是第 2 章「约束 > 提醒」落到最实的地方。
- ✓ 约束 > 提醒在这里最直接：你加一条 lint 规则，绕不过去。

## 五之三、其它禁忌

- **翻译腔**：「让我们来看看」「在……的情况下」。换成人话。
- **泄露案例业务**：任何具体业务、品牌、计费逻辑、厂商、表名，一律中性化或用 Relay（见 [AGENTS.md](AGENTS.md)）。
- **混用文档类型**：见第一节。
- **泄露本地绝对路径**：严禁在正文、代码片段、执行迹线和报错信息中出现开发者的本地绝对路径（如 `/Users/username/...` 或 `file:///Users/...` 等）。所有文件引用一律使用相对项目根目录的路径（如 `./AGENTS.md` 或 `src/types/money.ts`）或纯文件名。

## 五之四、Relay 案例硬核化硬约束

为了防止 Relay 案例变成无意义的概念重述，写 Relay 案例时必须遵守以下「三有」原则：
1. **有代码或配置**：不能只口头说「在配置里写了红线」，必须给出具体的 `AGENTS.md` 规则片段，或者 `json/yaml` 配置文件片段。
2. **有错误输出/执行迹线**：必须包含具体的报错信息（如编译报错、Lint 报错、黄金测试 Diff 报错），或者展示 Agent 的 `Thought -> Action -> Observation` 的具体执行迹线。
3. **有调试挫败过程**：写出 Agent 遇到挫败、陷入死循环、或者误读上下文的具体细节，以及人是如何介入并引导它走出来的。


## 六、长度与节奏

- 每章目标 **1500–3000 字**。讲不满说明论点太小，可并章；超太多说明塞了两个论点，该拆。
- 宁可一章讲透一个点，不要一章蜻蜓点水五个点。

## 七、好 / 坏对照

**坏**：「随着 AI 编程的发展，可观测性变得越来越重要。一个健壮的系统应该具备完善的日志能力，这对于排查问题至关重要。」
**好**：「线上白屏时，你不能 attach debugger——只能读日志。所以日志要在写代码时就埋好，且让 AI 自己就能拉到。Relay 某条对话发错回复，Agent 按 `conversation_id` 串出『分类→起草→工具调用』全链路，三分钟定位到工具返回了过期订单状态。」
