# 附录 B · 典型反模式速查

本附录整理了全书各章节中提及的、因引入 AI Agent 协同与非确定性大模型而产生的典型研发反模式（Anti-Patterns）与工程陷阱，作为自查清单。

## 一、研发心智与角色错位

* **将 Agent 视为黑盒魔法生成器**：仅给出模糊的自然语言需求便期待完美交付，缺乏明确的输入约束、反馈环路以及严格的完成定义（DoD）。（第 1 章）
* **停留在辅助模式的“逐行作者”**：仅将 Agent 作为代码补全工具或答疑终端，工程师仍陷入手写代码的低带宽泥潭，未能实现向“环境设计者”的角色转变。（第 1 章）
* **以冗长脆弱的提示词代替刚性环境**：试图通过不断堆叠、叮嘱 Prompt 来修正 Agent 的行为，而非将规则沉淀为 Lint 静态扫描、强类型守卫等物理约束。（第 1 与 5 章）
* **全自动盲目授权与过度人工阻断**：未能根据变更风险等级校准信任边界。高风险节点（如资金计算、数据库迁移）盲目给予 Agent 全自动权限，或在低风险节点过度设置人工拦截阻碍效率。（第 3 章）

## 二、规划断层与方案锁定失效

* **跳过规划（Plan）直接编码**：在模糊的想法下直接驱使 Agent 编写代码，导致 Agent 凭借高吞吐量快速产出大量方向错误的代码，造成高昂的重构与纠偏成本。（第 4 与 7 章）
* **将 Spec 编写为无约束的愿景描述**：Spec 文档中充斥着“打造极致体验”等无法被机器量化和执行的词汇，未能将其翻译为 Agent 可强制执行的能力边界与验收矩阵。（第 7 章）
* **方案与评审上下文未隔离（自批自改）**：在生成方案的同一个会话上下文内直接让 Agent 进行自我方案评审，导致思维盲区被原样继承，缺乏 Fresh Context 的独立审视。（第 7 与 15 章）
* **向单个 Agent 分配过度耦合的大粒度任务**：大块任务导致 Agent 上下文窗口迅速溢满、记忆漂移，且无法进行多线并行开发与精细粒度的阶段验收。（第 8 与 11 章）

## 三、工程基座与架构漂移

* **架构约定未接进静态校验拦截**：仅在文档中提及分层架构，未写入依赖校验工具中。Agent 极度倾向于“抄近路”，一旦失去刚性校验，会迅速写出越层引用的杂乱代码。（第 10 章）
* **反馈信号（错误日志）模糊**：静态拦截或测试失败时，仅抛出笼统的错误状态而不输出具体的报错行号或上下文迹线，导致 Agent 失去反馈信号，陷入盲目试错死循环。（第 14 章）

## 四、并行开发与环境抢占

* **仅隔离静态代码，共享动态资源**：使用 `git worktree` 隔离了目录，但多条并行线仍指向同一个本地数据库或本地网络端口，导致并行 Agent 运行时发生资源冲突。（第 11 章）
* **Worktree 目录与分支陈旧未清理**：会话结束后未及时回收临时的 Worktree 物理目录，导致磁盘空间碎片化以及本地 Git 注册表混乱，成为并行开发的最大劝退点。（第 11 章）
* **沙箱环境分配依赖手动装配**：每次并行开发新特性时，需要手动配置数据库实例、分配端口和修改环境变量，繁琐的流程使团队重新退回到单线开发模式。（第 11 章）

## 五、结构化可观测性盲区

* **日志物理隔离，Agent 无法自助访问**：开发人员充当“人肉贴日志管道”，未给 Agent 提供受控的只读 CLI 日志拉取权限，阻断了 Agent 的自助排障闭环。（第 13 章）
* **授予 Agent 过高的线上执行权限**：未遵循最小特权原则，使得 Agent 在排障时具备对生产环境的写操作权限，极易因一次误指令导致灾难性后果。（第 13 章）

## 六、安全与关键路径验证缺位

* **对抗/注入样本缺位**：安全测试集内全部为理想的顺路情况，无法测出模型在提示词注入（Prompt Injection）、恶意诱导和极端输入下的真实抵抗力。（第 15 章）
* **工具调用只控“能否调用”、不控参数范围**：把退款等高危工具加入工具列表却不在实现层做参数级守卫，模型一旦生成越界参数（超额金额、越权订单）即被照单全收。（第 15 章）
* **资金关键路径缺黄金/幂等测试**：对退款、扣款等发钱通道不设黄金数字定额测试与基于幂等键的并发契约测试，靠模型临场发挥，网络抖动下产生重复扣款。（第 14 章）

## 七、自验证与发布防线

* **测试通过即完成，未经 Agent 亲自自验证便交付**：Agent 编写完代码后，未驱动无头浏览器在沙箱中亲自模拟用户交互，导致页面布局错乱、网络请求阻断等显性 UI 问题漏过。（第 16 章）
* **用总体指标掩盖高风险子场景**：验收只看汇总通过率，高风险子场景（退款、转人工）的失误被均值稀释，上线后才在业务数据里暴露。（第 16 章）
* **将不可逆的系统操作交由 Agent 独立执行**：让 Agent 独立触发不可回滚的数据迁移、结构变更或全量物理删除，一旦出错将造成灾难性数据丢失。（第 17 章）
* **一刀切物理迁移与全量一把梭部署**：发布脚本含 `DROP COLUMN` 等不可逆变更，或新版本 100% 覆盖老实例而无灰度与自动回退，逻辑崩溃时无法快速回滚。（第 17 章）

## 八、文档复利与记忆丢失

* **技术与决策知识仅遗留在临时会话中**：排障结论与架构决策仅在单次会话里以对话形式存在，未沉淀为 `AGENTS.md` 或设计文档，导致后续开启新会话的 Agent 重复踩坑。（第 18 章）
* **任由系统文档与设计资产失效过期**：未在 PR 合并流程中同步更新对应的架构设计与 Agent 协同原语说明，导致 Agent 依据过期的 README 编写出废弃的调用代码。（第 18 章）
* **延迟工作任务记录流于形式**：在 `TODOS.md` 中记录的任务仅有一句简短的指令（如“重构退款逻辑”），缺乏背景、约束边界与依赖关系，新接入的 Agent 无法接手。（第 18 章）
* **复盘只写感想、不回写 harness**：阶段复盘产出一篇总结却不转化为 `AGENTS.md` 条目、CI 检查项或红线，下一轮照样踩同一个坑。（第 18 章）
