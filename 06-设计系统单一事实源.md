# 设计系统单一事实源

工程宪法规范了代码的内在纪律，而用户界面的外在呈现同样需要刚性的边界约束。如果缺乏统一的界面设计规范，Agent 在开发新页面时就会陷入凭直觉临时声明按钮、间距与配色的困境，使最终产出的产品显得杂乱且拼凑感严重。

**项目必须建立一份具体到 Design Token 级别的设计系统文档作为界面单一事实源，让 Agent 在动手开发前以此作为冷启动输入，并在最终验收阶段拿其充当刚性比对门禁。**

这两者的控制逻辑完全同构：工程宪法维持代码的架构一致性，设计系统则维系界面的体验一致性。二者均是将“主观的审美和直觉”降维固化为机器可读取、可搬运、可拦截的客观约束，共同构成 Harness 研发脚手架的底座。

## 设计系统作为界面一致性 SSOT 的动机

在多 Agent 并行研发的场景下，依赖人类肉眼进行逐像素的比对完全不可行。通过在设计系统文档中定义唯一的色彩代码、字号梯度与布局比例，Agent 便拥有了标准参考。此时，偏离此标准的任何视觉差异都将被直接判定为系统漏洞。

要让设计系统对 Agent 产生实质的约束力，文档内容必须被高度量化。诸如“界面要保持现代、简洁与专业”等宽泛的描述性词汇对于 Agent 而言无法指导编码决策。设计系统必须细化为具体可引用的工程常量（Design Tokens），例如：
* **主色与状态色**：声明每一种颜色的精确 Hex/RGBA 数值（如主色、成功色、异常警告色）。
* **字体与排版**：定义段落、标题与说明文字的字号（如 12px、14px、20px）与行高。
* **边框与圆角**：明确卡口与卡片等组件的圆角参数（如卡片 8px、小按钮 4px）。
* **间距刻度**：规定界面元素间距只能采用 4 的倍数进行等比阶梯缩放（如 4px、8px、16px、24px），严禁使用非标数值。

在代码层面，这些 Token 会被固化为底层的 CSS 变量或前端状态常量，且在 CI 卡口处配置静态拦截，一旦检测到组件内硬编码的“裸色值”或“非标间距”，构建流水线便会红线阻断。

## 设计系统的两个控制卡口：启动与比对

设计系统作为 SSOT，必须在 Agent 开发生命周期的两端强力介入：

* **冷启动加载（开发端）**：Agent 在启动任何与界面交互相关的任务前，必须以物理读取的方式加载 `./docs/specs/design_tokens.json`，将其作为后续生成 HTML、CSS 与组件代码的上下文约束。
* **静态与视觉比对（验收端）**：在验收代码时，使用界面静态检测工具（Linter）与视觉回归测试（Visual Regression Testing）进行校验。任何组件若偏离了设计系统定义的 Token 值，将被自动打回。

两端卡口的就位，才能确保设计系统从一份摆着好看的规范，转化为研发中真实生效的控制系统。

## gstack 在设计验收层面的工具链封装

在 gstack 的参考实现中，设计的一致性被内置于工具流程中：

* **设计系统生成指令**：通过 `/design-consultation`，系统根据产品的风格调性自动提炼并产出一份格式化、机器可读的 `design_tokens.json` 规范。
* **自动化视觉评审**：通过 `/design-review` 调起视觉比对工具链，利用快照对比技术扫描生成的页面与设计系统的偏差，自动将像素漂移、非标字体及布局错乱以 Bug 日志形式反馈给 Agent。
* **规则配置化**：通过脚手架配置将 `design_tokens.json` 转化为构建流中的环境变量，使前端框架底层组件强制与 Token 绑定。

即使脱离 gstack，该理念依旧可以手工实践：将 Token 写入根目录的 `./design_tokens.json`，在前端代码库中定义全局 CSS 变量（如 `--color-primary`），并用 Linter 插件禁止在页面中直接编写硬编码样式（如禁止直接使用 `color: #333`）。

## 主线演练：Relay 界面 Token 设计与视觉门禁

以 Relay 坐席后台的审批卡片开发为例，展示设计系统如何通过 Harness 强制生效：

在项目开发前，设计系统已经以结构化文档形式固定在 `./docs/specs/design_tokens.json` 中：
```json
{
  "color": {
    "primary": "#2563EB",
    "status": {
      "pending": "#F59E0B",
      "processed": "#10B981",
      "escalated": "#EF4444"
    }
  },
  "radius": {
    "card": "8px",
    "button": "4px"
  },
  "spacing": {
    "md": "16px"
  }
}
```

在 Build 阶段，Agent 负责编写“退款审批列表”页面 `src/components/RefundApproval.tsx`。在开发中，发生了以下两次设计漂移的拦截：

* **拦截一：硬编码色值。** Agent 为了图快，直接在待处理状态标签上写入了硬编码的十六进制色值：`style={{ color: '#F59E0B' }}`。在提交代码时，挂接的 Pre-commit 静态扫描工具检测并报错：
  ```bash
  $ npm run lint:style
  ✖ Style Lint Error: Hardcoded hex color value '#F59E0B' found in src/components/RefundApproval.tsx (Line 34).
    Rule ID: enforce-design-tokens
    Suggestion: Replace with design token variable var(--color-status-pending).
  ```
  这一报错作为反馈强迫 Agent 将其重构为引用 CSS 变量：`style={{ color: 'var(--color-status-pending)' }}`。
* **拦截二：圆角漂移。** Agent 在重写列表容器时，误将圆角设为了 `border-radius: 6px`。CI 中配置的视觉回归测试在运行沙箱无头浏览器截图时，检测到了 0.12% 的像素比对不一致，输出以下测试报告：
  ```bash
  $ npm run test:visual
  ✖ Visual Regression Failed: RefundApprovalPage
    Diff ratio: 0.12% mismatch against golden snapshot.
    Expected: border-radius 8px (.approval-card).
    Actual: border-radius 6px in src/components/RefundApproval.tsx (Line 72).
  ```
  Agent 捕获这一具体的像素偏差日志，发现自己偏离了设计系统中的 `radius.card` 规范，随即修正为 `border-radius: var(--radius-card)`，视觉测试最终全绿通过。

通过这种强制拦截，Relay 坐席后台维持了像素级的视觉一致性，而这一切完全不依赖人类评审在屏幕前的肉眼比对。

## 典型反模式

* **设计系统游离于构建流之外**：仅将设计系统写在独立的设计软件（如 Figma）或纯文本文档里，不与代码中的变量绑定，也不接入 CI 卡口拦截。这导致 Agent 经常产出大量“野色值”与自定义字体。
* **虚无空泛的界面规范**：在文档中大量使用诸如“注重用户体验”、“排版应错落有致”等主观词眼。Agent 无法理解此类无工程量化标准的词句，仍然会写出杂乱的视觉排布。
* **缺乏像素回归检测**：仅相信 Linter，而不跑视觉截图测试。Linter 无法覆盖由于复杂的 Flex 布局或定位（Positioning）引起的布局塌陷与像素漂移，必须引入无头浏览器截图比对。

## 本章要点

* **设计系统是 SSOT**：设计系统在视觉一致性上扮演的角色，与工程宪法在代码健壮性上扮演的角色完全同构。
* **规则必须高度工程化**：规范必须以具体的 Design Tokens（Hex值、精确像素行高、间距刻度）呈现，确保可加载与可验证。
* **构建两端拦截**：动手开发前加载 Token 作为上下文，QA 时通过视觉回归测试与 Style Linter 强制比对，完成回路。
* **用工具代替直觉**：通过截图比对和自动化门禁来锁定视觉质量，杜绝依赖人类肉眼进行低效的像素级找错。
