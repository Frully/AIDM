# 多线并行的开发纪律与环境清理

双层物理隔离解决了并行开发的技术前提，但并行的高效运转并不取决于并发数量的盲目累加。缺乏严明编排纪律的多开会话只会让代码库和环境资源陷入混乱，并在最后的合并审核时形成严重的堰塞湖。

**并行的研发效能取决于刚性的编排纪律而非会话数量：每条开发分支只承载单一职责的原子任务，Agent 运行受限于严密的超时与重试护栏，且在特性合并后，必须自动执行残留物理资源的销毁与回收。**

并行的极限并不是大模型的处理吞吐，而是人类工程师的评审带宽。如果并行的产出超出了人类评审者的消化速度，多线开发就只能堆出一堆无法交付的半成品。

## 单一任务契约：防范上下文污染

要保障多线并行不发生混乱，首要纪律是建立“单一会话 ─ 原子任务”的绑定契约。

大模型 Agent 的上下文窗口容量是其有限的核心资源。如果我们试图让同一个 Agent 会话同时处理“重构退款逻辑、顺便修改路由分类、顺带修补坐席前端缺陷”这三个彼此无关的事务，它的上下文会被不相干的代码片段污染，产生记忆漂移并导致编码失控。

正确的做法是为每个 Agent 圈定明确的职责红线。当一个 Agent 进入 `wt-refund-01` 工作区时，它所加载的 Spec、设计文档及 `AGENTS.md` 应当仅声明退款相关的业务实体。

这使得 Agent 能够在上下文被冗余代码污染前，干净利落地完成原子特性的编写与自验证。同时，这也让评审（Review）阶段的粒度大大降低，评审人能够集中焦点进行单项验收，规避了跨切面审查带来的逻辑遗漏。

## 行为边界与卡死护栏

并行开发中的另一个常见瓶颈是 Agent 的“行为失控”：
* **无谓的越界修改**：在完成指定任务后，Agent 倾向于去顺手修改相邻的其他代码文件，这破坏了原子提交的清晰度。
* **死循环重试**：当 Agent 在某处代码因编译或类型报错卡住时，它会不断进行自我纠错尝试，在没有任何人肉干预的情况下白白烧掉大量的 Token。

为此，编排系统必须为 Agent 设定刚性的生存周期边界与拦截限制。我们需要在配置文件中声明最大的自我修正迭代次数（如 `max_iterations: 15`）。一旦 Agent 达到此限额仍未通过编译，必须强制挂起会话并向人类开发者发出求助信号。明确规定“做完即止、卡住即报”，是防止并行任务失控和账单飙升的纪律红线。

## 物理资源的例行清理

多线并行研发最常见的“阻力”来自于开发机上残留 Worktree 环境的无限堆积。

每当一个 Agent 完成任务并开启 PR 后，如果在合并后不及时清理其 Worktree，这些残留目录会持续占用网络端口、本地 Redis 数据库实例以及物理存储空间。过不了几天，本地环境就会因为资源占满而无法启动新的 Worktree 任务。

清理残留必须被固化为研发日常的最后一环。一旦代码分支合并入主干，或者该特性被中途废弃，编排工具必须立刻自动触发资源的撤销动作：注销对应的 Git 注册表、回收端口、彻底删除沙箱数据库。不留任何技术残骸。

## 主线演练：Relay 特性合并与环境销毁迹线

在 Relay 示例项目中，大仓引入了并发限制，将最大并行 Agent 会话数锁死在 3 个（匹配主程的日常 Review 带宽）。同时，为防止 Agent 自愈死循环，在 `.agents/config.yaml` 中配置了迭代护栏：

```yaml
# .agents/config.yaml
max_iterations: 12        # 单次会话最多执行 12 轮 Code-Action 循环
session_timeout_ms: 300000 # 5 分钟超时强杀
```

当 `RELAY_REFUND_01` 分支（开发自动退款判定）的 PR 在远端 CI 跑通并通过 Review 合并入主干后，编排工具自动在本地执行了环境的清理动作：

```bash
$ npx wt-manage clean RELAY_REFUND_01
[INFO] Initiating resource cleanup for session: RELAY_REFUND_01
[INFO] Deleting git worktree path: /path/to/project/AIDM/wt-refund-01
[INFO] Dropping local database: relay_sandbox_wt_refund_01
[INFO] Freeing socket port: 3001
[INFO] Success. Worktree and dynamic resources removed.
```

通过这一层刚性的清理机制，系统确保了每次开启新的 Worktree时，本地资源状态都是全新、干净的，从根本上避免了因为环境残留导致的随机端口抢占报错。

## 典型反模式

* **多重任务交叉派发**：把不相干的任务交给同一个会话处理，导致 Agent 上下文过载，最终产出无法独立验收的大片脏提交。
* **无护栏自动重试**：未配置最大迭代步数和超时上限，导致 Agent 在编译报错时陷入长达半小时的死循环，白白消耗大量资金。
* **陈旧 Worktree 长期堆积**：合并分支后只管开发新特性而不清理旧目录，最终因端口资源和数据库实例占满导致开发环境彻底瘫痪。

## 本章要点

* 必须遵循“单一会话 ─ 原子任务”的绑定契约，防止跨领域代码片段造成 Agent 上下文污染与记忆漂移。
* 设定最大迭代轮数与超时强杀边界，构建起卡死护栏，拦截 Agent 陷入自我纠错死循环。
* 分支合并或废弃后，必须立刻自动回收物理工作区及动态关联的数据库、端口资源，确保本地环境零残留。

> 限制会话迭代次数和清理物理资源是多线并行的硬性纪律。为了让这套并发流程真正跑起来，我们需要能够自动完成 Worktree 创建与端口、数据库装配的一键式自动化脚本。我们将在第 16 章 `多线开发环境的自动隔离与装配` 中探讨这一具体实现。
