# AI 驱动开发方法论 · 教程大纲

> 体裁：**方法论 + 案例佐证**。每章讲清原则、取舍与「为什么」，用一个贯穿全书的中性示例项目作主线佐证，而非逐步跟做的操作手册。
> 编排原则：**先把理论原理讲清楚，再讲怎么实践**——第一部分是纯理论/心智模型（不碰工具），从第二部分起才落到工具与工程。
> 边界：方法论提炼自真实生产项目，但**只教模式、不暴露其业务**；与 AI 驱动开发无关的纯业务、纯 SRE 细节（通用备份/灾备演练/SLA 定义/容量规划）一律剔除。
> 语言：全文简体中文。

## 阅读约定：原则可迁移，gstack 是参考实现

每章按两层组织，并显式区分：

- **原则层（可迁移的「为什么」）**：不绑定任何工具，换个技术栈也成立。
- **实现层（gstack 参考实现）**：[gstack](https://github.com/garrytan/gstack) 是本书推荐的**最佳实践实现**。我们用它演示「怎么做」，但**每章一定先把可迁移原则讲透，再落到 gstack 实现**，让工具耦合显性、可替换。

## 适用场景（先说清楚边界）

- **教的是：用 AI 从 0 驱动一个完整项目**——立项、规划、开发、评审、测试、上线部署、运维更新，全生命周期。
- **不是为小改动准备的**：修一个 bug、改一行文案不需要这整套。本书不做「按改动大小裁剪」的妥协版。
- **样本诚实声明**：方法论提炼自少量但**真实上线、持续运维**的生产项目（不是玩具 demo，也非社区级千人验证）。在小样本上被证明可用；读者应带着「实战过、但仍需结合自身场景判断」的心态使用。

## 主线示例项目（中性、虚构）

**Relay —— 一个 AI 客户支持自动化 SaaS。** 把多渠道（邮件 / 网页聊天 / 工单）的客户对话接进来，AI 给对话分类、定优先级与路由，起草回复，并能**调用工具**（检索知识库、查订单状态、发退款或补偿积分）；低风险自动回复，高风险（超额退款、敏感操作）转**人工坐席审批**；按「已解决的对话」计量计费。线上跑在云上，AI Agent 经 CLI（AWS CLI）够到线上拉对话日志排障。**纯教学虚构，与任何真实业务无关**，只用它佐证方法、不展开业务逻辑。

> 选它当主线：它是 **agentic 产品**（LLM + 工具调用 + 有副作用动作），能把全部承重柱点亮——AI 发起退款 = 可逆性 + 黄金测试 + 信任校准；自动 vs 转人工 = 人在回路；工具选择 / 安全拒答 / 对抗诱导 = 更现代的 Eval；多渠道异步 + 流式 = 可观测性；多进程 = monorepo。而且产品里「AI 带护栏地执行动作」与本书「开发 Agent 带护栏地驱动生产」同构，主题呼应。

# 第一部分 · 理论与心智模型（先把原理讲清楚，不碰工具）

> 这一部分只讲「为什么」和思维框架，不出现任何具体工具命令。读完它，你应该理解 AI 驱动开发为什么这样组织；从第二部分起才动手搭工具。

### 1. 从「AI 辅助」到「AI 驱动」
- 心智模型：从「写代码的人」变成「设计约束 + 评审产出的人」；把模型当需要明确边界与反馈回路的队友，而非搜索引擎或魔法生成器。
- 这本书要建立的世界观：可靠性不来自更聪明的提示词，来自你给模型搭的环境。

### 2. 核心模型：Harness = 文档 + 检查 + 反馈
- 全书的统一母题，三件套缺一不可：
  - **文档（约束输入边界）**：约束、设计、决策、Runbook 沉淀成 AI 每次都读的资产，形成复利（作为 Agent 每次冷启动开工时的刚性输入）。
  - **检查（拦截输出偏差）**：lint、风格、编译、单测、集成测试、架构与依赖检查——产出后自动拦截，不过不放行（由构建流水线卡口与门禁承载）。
  - **反馈（驱动闭环自愈）**：本地是检查/测试输出，线上是可观测性；反馈要**让 AI 直接够得到**，且 Agent 要**像用户一样亲自验证**（通过可观测性日志与端到端自验证回流）。
- 关键原则：**约束 > 提醒**——模型会忘记提醒，但绕不过被强制的约束。

### 3. 人在回路：监督式自治与信任校准
- AI 驱动 ≠ 无人驾驶：人始终在评审席。
- 可逆性是分级的底层依据：操作风险不取决于金额大小，而取决于出错后能否在几分钟内回到原状——可逆则放手让 Agent 试错，不可逆则必须人审。
- 信任校准的动态过程：随着 Agent 在特定模块积累记录，自治权可逐步放开；模型版本变更或出现越界行为时收紧。
- 人在哪些节点拍板：方案锁定、关键路径改动、上线放行。

### 4. 全局地图：开发生命周期的七个阶段
- 用一张概念地图统领全书：**Think → Plan → Build → Review → Test → Ship → Reflect**（源自 [gstack](https://github.com/garrytan/gstack)）。
- 每阶段「换一顶角色帽子」、有产出、有反馈；阶段衔接，上一阶段产出自动喂给下一阶段、不漏活。
- 后续各部分逐一展开这七个阶段；横切的「承重柱」（harness / monorepo / 并行 worktree / 可观测性 / Eval）跨越多个阶段做基础设施。
- **完整闭环 vs 右尺寸**：这七个阶段是「一个完整项目至少要走全一遍」的关注点清单，不是每次提交都机械跑满的仪式。Review 与 Test 基本不可省（不验证就不算完成）；Plan 的三视角评审、并行规模随改动风险伸缩。判断「这次该走哪几阶段」本身就是一种信任校准。

| 阶段 | 干什么 | 对应本书 |
|---|---|---|
| Think | 重构问题、挑战假设、产出设计文档 | 第三部分 |
| Plan | CEO/工程/设计三视角评审，锁定方案、拆任务 | 第三部分 |
| Build | 在锁定计划上实现（可并行多线） | 第四、五部分 |
| Review | 资深工程视角抓 CI 漏掉的生产 bug + 安全审计 | 第七部分 |
| Test | 真实浏览器 QA、Agent 像用户一样验证、业务验收 | 第七部分 |
| Ship | 同步主干、跑测试、开 PR、合并并验证线上健康（可回滚） | 第七部分 |
| Reflect | 复盘、沉淀可复用模式与文档 | 第八部分 |

# 第二部分 · 搭建脚手架（理论落地，实践开始）

> 第一部分的「为什么」在这里变成「怎么搭」。这是动手的起点。

### 5. 项目宪法与 Agent 协同原语
- `AGENTS.md` = 项目宪法：把不变量、分层规则、红线集中一处，Agent 每次会话必读。用通用的 `AGENTS.md`（而非某家专有文件名）做单一事实源；在系统层用一个软链接把 `CLAUDE.md` → `AGENTS.md`，让 Claude Code 与其它 Agent 读同一份、零重复维护。
- `AGENTS.md` 骨架模板（七节：项目描述、语言约束、主线示例、硬约束、架构约束、保留惯例、工作约定）可直接复用；硬约束节须标注执行工具名，使约束机器可强制执行。
- Agent 协同原语（是手段，不是要凑齐的「套」）：skills / subagents / hooks / slash commands / settings。重点不是逐个罗列，而是它们如何拼成第 2 章那条回路。SessionStart Hook 让 AGENTS.md 在每次会话开始时自动输出给 Agent，不依赖人工粘贴。

### 6. 设计系统单一事实源
- 视觉/UX 决策集中一份文档，动 UI 前必读、QA 时查漂移。
- 用 gstack 设计规范做底座（design-consultation 立规范、brand guidelines 定调性、design-review 当一致性检查关），让设计一致性也变成可检查的约束。

# 第三部分 · 立项与规划（Think + Plan）

### 7. 从模糊想法到可执行 Spec
- 六问重构问题：挑战假设、挖未言明的能力、给多方案 + 工作量估计；产出设计文档供下游消费。
- 一份好 Spec 的标志：冷启动的人或 Agent 据此直接动手。

### 8. 锁定方案：多视角评审与设计文档先行
- 铁律：绝不在同一上下文里自批；Plan 阶段要锁定两样东西——**方案**（多视角评审通过）和**设计**（写成文档）——两者合在一起才算 Plan done。
- 三视角各查什么（每个视角有具体检查项）：CEO 查范围蔓延与 ROI；工程查数据流异常分支、幂等键、并发安全、测试矩阵；设计查交互一致性与非标组件。
- 按风险裁剪视角（不是每个改动都跑全三视角）；意见冲突时写进设计文档留可审计决策。
- 先写设计文档把架构取舍显性化、留可审计的「为什么」；文档即交接，冷启动可续。Spec → Design 是强制顺序，不可跳过。

### 9. 把方案拆成 Agent 能领取的任务与交付物追踪
- 把锁定的方案拆成**有明确边界、单一职责**的任务，让 Agent（尤其并行多 Agent）能各自领取、互不踩——这是第五部分并行的前置：边界不清，并行就是制造冲突。
- 任务粒度三个判断标准：能独立验证、不跨越多个架构层、Agent 不需要反复猜测意图。
- 跨任务依赖：接口契约先行——把两个任务共用的类型定义或接口声明作为独立的先行交付物锁定。
- 中途发现边界划错了：暂停 Agent、更新 TODOS.md、重启任务，不在当前会话里扩大范围。
- 轻量交付物追踪：Spec → 设计 → 任务 → 代码/测试的链路，让 cold-start Agent 与评审能追到「为什么」。

# 第四部分 · 工程基座（Build · 检查闭环主场）

### 10. Monorepo：给 AI 一个没有上下文墙的代码库
- 仓库边界 = 上下文墙，降低 AI 产出质量。红利：全局上下文、原子提交、跨切面改动一次到位。代价与边界：何时不该硬塞进一个仓。
- 工具选型：pnpm workspaces（新项目起点）→ Turborepo（CI 超 10 分钟时加构建缓存）→ Nx（需要微前端或代码生成插件时）。
- CI 受影响包检测：`--filter=...[HEAD^1]` 只跑受当前 commit 影响的包，避免全量 CI 时间线性增长。
- 大型 Monorepo 的上下文窗口管理：AGENTS.md 标注任务相关包，每包 README 说明职责边界，TODOS.md 任务条目标"涉及包"字段。

### 11. 分层架构与依赖守卫
- 划层原则：按变更频率和对基础设施的依赖程度划定层边界。AI 专属划层：LLM 调用放基础设施层，Prompt 模板放域层或专属模块，Tool schema 定义放域层。
- 定义层次并**强制方向**；「谁不能依赖谁」写进 CI/hook，越层被构建直接拦；守卫本身要有「证明它会拦」的负向测试。
- 多语言工具：dependency-cruiser（JS/TS）、import-linter（Python）、ArchUnit（Java）、go-arch-lint（Go）。工具配置可由 AI 生成（给定 AGENTS.md 约束），人 review 后挂进 CI。

### 12. 检查即门禁
- 检查全家桶：lint、代码风格、编译、单元测试、集成测试、架构与依赖检查；前移到 pre-commit/pre-push/CI。
- 本地 hook（lefthook）快速拦截低成本错误；远程 CI 不可绕过，拦截高成本错误——两层阶梯式门禁缺一不可。
- 检查配置（lefthook.yml、GitHub Actions workflow）由 AI 根据 AGENTS.md 约束生成，人 review 后提交，之后自动执行。
- 检查输出也是反馈：失败信息要可读、可定位，AI 据此自我修正。

# 第五部分 · 并行 Worktree 开发（Build 并行）

> 骨架：**并行能力从哪来 + 怎么互不串台**。能力来源 = 桌面端可同时开多个对话/会话，一条对话一条并行线。两层隔离缺一不可：**worktree 隔离静态代码**，**脚本 + hook 隔离动态环境**。

### 13. 并行 Worktree：双层隔离与开发纪律
- 能力来源：桌面端可同时启用多个对话，多线并行；多线指向同一目录会互相覆盖，必须两层隔离。
- 第一层（静态代码）：worktree 给每个 Agent 独立目录、独立分支，共享一份 git 历史。
- 第二层（动态环境）：仅隔静态不够，运行期端口/库/服务进程也要隔离，否则多线抢同一套运行环境而打架。
- 纪律：单一职责任务契约、上下文护栏（max_iterations、session_timeout）、用完立即清理（合并后销毁 Worktree 和沙箱数据库）。陈旧 Worktree 堆积是最大劝退点。

### 14. 多线开发环境的自动隔离与装配
- 会话启动 hook 自动 provision；装配脚本（wt-setup.sh / wt-clean.sh）由 AI 根据需求规范生成，人 review 后提交代码库，之后每个 Worktree 用同一套脚本装配。
- SessionStart Hook 在每次打开该 Worktree 的 Agent 会话时自动输出 AGENTS.md、当前分支和未完成 TODO，Agent 不需要提醒就知道自己在哪个任务线上。
- 并发安全的 slot 分配（共享 `.git` 注册表 + 目录锁）、幂等重入。

# 第六部分 · 可观测性（反馈闭环主场）

### 15. 写出来就可诊断
- 线上靠日志/指标查问题，不是 attach debugger；新代码写时就带可观测性。
- 约束→实现链条：AGENTS.md 写"禁止 console.log、必须有 conversation_id"；ESLint no-console 强制执行；`@relay/logger` 封装由 AI 实现给定约束。字段名（event、conversation_id、reason）是服务间契约，不能随意改名——Ajv JSON Schema 契约测试在 CI 里防止重构改掉这些字段名。
- PII 日志红线（不记录用户消息原文、手机号、邮箱）也写进 AGENTS.md，靠 ESLint 自定义规则和 Eval 安全案例双重兜底。

### 16. 让 AI 直接够到线上
- 反馈要让 AI 访问得到：给 Agent 受控的线上访问能力（如 AWS CLI）。
- AI-native 可观测性：结构化 JSON 日志、固定字段名是合同、失败路径显式记录 reason 枚举。AI-readable Runbook 是声明式条件规则（`if condition → action`），而非给人看的步骤清单。
- 渐进式访问展开（与信任校准挂钩）：只读日志 → 只读 DB 查询 → 无副作用诊断 → 有副作用操作，每层需通过对应信任里程碑。
- 权限与护栏：最小权限、只读优先、危险操作要参数级守卫、访问留痕。

# 第七部分 · 评审、验证与交付（Review + Test + Ship）

### 17. AI 功能的 Eval 测试
- 大模型输出非确定性，精确字符串断言失效；必须断言行为档位——「断言字符串是单测，断言档位是 Eval」。
- Eval 覆盖率策略：两维判断（输出是否确定 × 是否有副作用）；意图分类/工具调用决策/知识库检索 → Eval；纯函数计算/日志格式化 → 单元测试。
- 临界样本（999元 vs 1001元阈值测试）与对抗样本（Prompt 注入攻击）是 Eval 的刀刃。
- Judge 模型用于评估主观维度（是否礼貌），必须先以人工真值校准（≥95% 一致率）才能接 CI。
- CI 精准触发：只在 prompts/模型配置/检索配置变更时运行；发钱路径 100% 拦截，分类路径 ≥95%。
- 上线后采样 5-10% 真实流量打分，回流 Golden Dataset 形成闭环；所有 LLM 调用必须走统一 Client 封装（Langfuse/Helicone/LangSmith 可热切换）。

### 18. 代码评审与安全审计
- 资深工程视角抓 CI 漏掉的生产 bug；author/review 分离、fresh context 评审避免偏袒自己刚写的代码。
- 安全审计当一关：OWASP Top 10 + STRIDE。
- AI 特有安全威胁：间接提示注入（用户输入永远在 user 角色，RAG 内容用 XML 标签包裹隔离）；参数级工具授权（不只允许工具调用，还限制参数范围）；Prompt 安全审查清单（5 项，含 Eval 对抗性测试覆盖）。

### 19. Agent 自验证与业务验收
- 硬纪律：**产出未经 Agent 亲自像用户一样验证，就不算完成**——「检查通过」≠「行为正确」。
- 开真实浏览器跑端到端：实际点一遍、发现 bug、固化成回归测试。
- DoD 清单：Agent 自验证层（Eval 通过率、对抗性 Eval、PII 日志安全扫描、监控埋点覆盖）+ 人工验收层（staging 验证、KPI 确认、回滚方案验证）。
- **业务验收 KPI**：准入 KPI（不达到不上线，含退款意图识别准确率、误触发率 = 0、高风险转人工召回率）+ 质量 KPI（上线后监控）+ 成本 KPI（单对话 token 上限）。

### 20. 上线与交付
- ship 流程：同步主干、跑测试、审覆盖、开 PR；land-and-deploy 合并并验证线上健康。
- **数据库迁移安全**：AGENTS.md 约束"禁止直接 DROP COLUMN，必须用 expand-and-contract 两步策略"；CI migration-linter 自动检查；迁移 SQL 由 AI 根据约束生成，人 review，两步之间有观察期。
- **可逆性是 AI 自治的前提**：你越让 Agent 驱动部署/迁移，就越要求每一步可逆——部署可回退，迁移做成可回滚。不可逆的操作不交给 Agent 独立执行。
- Agent 驱动的发布更新：经 CLI 受控部署（最小权限 + 确认护栏 + 操作留痕）、灰度 + 部署后 Agent 自验证线上健康、健康不过自动回滚。
- 关键正确性路径纪律：用黄金数字定额测试 + 幂等保证（以 AI 发起的退款为例，验金额正确、重试不重复退款、错退可冲正回滚）。

# 第八部分 · 文档沉淀与延续性（Reflect）

### 21. 文档沉淀、待办交接与阶段复盘
- 文档是喂给 AI 的输入端，与检查、反馈共同构成 Harness 三件套（呼应全书核心模型），在 Reflect 阶段回流为下一轮的约束输入。
- 沉淀什么（按 Diátaxis 分类）：约束（`AGENTS.md`）、设计文档（说明「为什么」）、决策日志（记录否决方案）、Runbook（机器可执行）、QA/Eval 报告。doc-hygiene-linter 在 CI 里检测文档漂移，文档更新与代码变更同步合并。
- `TODOS.md`：每条延迟工作带五要素（做什么/为什么/技术边界/前置依赖/推迟理由），冷启动 Agent 直接接手，不依赖 Slack 记忆。
- 阶段复盘：把反复出现的问题固化成 CI 检查项和 AGENTS.md 条目——复盘不是总结，是把教训转化为下次绕不过的约束，完成 Harness 闭环。
