# 可逆交付与灰度上线

部署上线并非验证的终点，而是验证阶段向线上环境的物理延伸。由于生产环境具有最复杂的网络拓扑和最真实的并发流量，代码即便通过了本地和 Staging 的所有检查，上线后依然可能爆发不可预测的退化（如内存泄露、死锁以及未捕获的边界异常）。

**可逆性是系统能够授权给 Agent 自主发布与运维的前提；对于无法保障秒级回滚的可回撤变更，必须禁止 Agent 独立执行。**

本章聚焦于“自治授权边界”在生产阶段的闭环落地。根据 [人在回路：监督式自治与信任校准](./03-人在回路：监督式自治与信任校准.md) 所确立的核心原则：操作越可逆，越可以放手让 Agent 独立执行；而不可逆的动作，则必须通过严格的人工审批。数据库的可逆迁移（Expand-and-Contract）与灰度部署自愈，是保证 Agent 自主发布的底线护栏。


## 数据库迁移安全：双向可逆设计

在涉及数据库 Schema 变更的发布中，最容易发生由于新版代码故障导致回滚时，数据库结构已被破坏且无法兼容老版代码的灾难。这在 Agent 自动部署的场景下是不可接受的。

为了实现零停机部署与秒级回撤，我们强制引入 **Expand-and-Contract（扩展与收缩）两步策略**：

1. **第一步（Expand 阶段）**：只做向后兼容的新增修改（如新增列并允许为空、保留旧列）。此时新老代码能同时在数据库上安全运行。
2. **观察期**：灰度发布新版代码。如果新版运行异常，可立即将代码回滚至旧版本，旧版代码依然可以无缝读写当前数据库。
3. **第二步（Contract 阶段）**：当新代码在线上稳定运行数天后，发起第二次物理迁移，删除旧列或废弃冗余结构。

在 `./AGENTS.md` 中必须强制写入此规范，强迫 Agent 生成两步可逆迁移脚本：

```markdown
## 数据库迁移红线
- 严禁在单次部署中直接执行 `DROP COLUMN`、`RENAME COLUMN` 或修改列属性。
- 必须使用 Expand-and-Contract 策略分两次发布。
- 迁移脚本必须在本地 Staging 环境跑通物理回滚（down 脚本）校验。
```

同时，CI 流程中必须集成 `migration-linter`。当 Agent 提交的 SQL 脚本包含 `DROP` 或 `ALTER TABLE ... MODIFY` 等不可逆变更时，构建卡口一票否决。


## 灰度发布与健康度自验证

有了可逆的数据库设计，发布就变成了可以由 Agent 自动化编排的“流量划分与自校验过程”：

```
+-------------------------------------------------------------+
| 1. 部署 1% 新版本实例 ----> 2. 灌注灰度流量                 |
|                                |                            |
|                                v                            |
| 4. 指标回滚 <---- 3. Agent 实时调取 Ingress 日志与 HTTP 探测 |
+-------------------------------------------------------------+
```

部署脚本在每次发布时首先划分 1% 流量至新版本。随后，Agent 会以 10 秒为周期，持续向新实例发送 HTTP 健康探测请求，并拉取线上出站网关的异常日志。

在 Relay 部署 v2.1.0 期间，Stripe 接口变更引发了底层的签名校验失效。在流量划分至 1% 的瞬间，监控探针捕获了大量的 500 报错，Agent 调取了部署迹线：

```bash
$ relay-ops deploy --image relay-app:v2.1.0 --canary 1

[INFO] Deploying canary instance for v2.1.0 (1% traffic)...
[INFO] Monitoring health state (Window: 60s, Threshold: ErrorRate < 1%)
[WARNING] Health Probe failed: GET /health returned 500.
[ERROR] Error rate spike: 12% > 1%.
[ALERT] Health check FAILED. Initiating Automated Rollback...
[SUCCESS] System restored to stable state. Rollback complete. MTTR: 14s.
```

健康不过自动回滚，14 秒回到稳定态——正因为每一步可逆，才敢把这套发布交给 Agent 编排。

涉及资金的关键路径，上线前还要过检查最底层那两道刚性测试（黄金数字定额与幂等并发，见 [检查即门禁](./14-检查即门禁.md)）；幂等契约在这里有它的生产侧验证：灰度时模拟断网重放，确认重试不重复发钱。


## gstack 参考实现

自动化部署与迁移集成在发布技能 `/ship` 与合并控制中：

```bash
$ gstack ship --image relay-app:v2.1.0 --canary-rate 10
```

它触发迁移文件的 lint 扫描与灰度流量划分，与云厂商的交付网关对接。部署脚本是可替换的物理运行层，核心原则是：不论对接何种基础设施，都必须用健康探测（如 `/health`）实时检验新实例，报错超标时自动回撤流量。


## 典型反模式

* **一刀切物理迁移**：发布脚本直接包含不可逆的 Schema 变更。新代码一旦发生故障回滚，老版代码由于结构不兼容直接瘫痪，导致系统发生级联故障。
* **全量直接替换**：新发布直接 100% 覆盖线上运行实例，缺乏灰度流量划分与指标观测，导致逻辑崩溃时只能执行漫长的人工紧急重建。
* **强行下发不可逆变更**：将没有回滚路径的底层数据库操作、或含有物理删除的变更逻辑交由 Agent 独立执行，在出现非预期异常时使系统处于无法收敛的瘫痪状态。


## 本章要点

* 可逆性是把上线授权给 Agent 的核心前提；不可逆的步骤必须人审，不交给 Agent 独立执行。
* 迁移用双向可逆设计（Expand-and-Contract），约束写进 `./AGENTS.md`、靠 CI migration-linter 强制，而不是靠 Agent 记忆。
* 部署必须配灰度注入 + 健康度自验证 + 自动回滚，把验证延伸到生产；核心指标超标时数十秒内回撤。
* 资金关键路径的黄金数字与幂等测试在检查门禁（见 [检查即门禁](./14-检查即门禁.md)）就拦，这一章只做它们的生产侧灰度验证。
