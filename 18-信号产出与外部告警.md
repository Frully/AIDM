# 信号产出与外部告警

应用已经能够稳定地产出结构化日志与性能快照。接下来的架构设计重点是：如何基于这些原始信号建立高可用的线上告警体系。一个常见但存在严重设计缺陷的做法是在应用内部编写定时任务，自行分析状态并通过邮件或 Webhook 发送报警。

**应用进程只负责高效率、无状态地输出原始信号（如日志和 Metric 暴露端点），决不参与任何关于“该不该告警”的业务逻辑决策与推送发送。告警的状态评估、去重过滤、冷却机制与分级升级策略，必须全部解耦给独立的外部监控系统（如 Prometheus、Loki、PagerDuty）。**

## “同生共死”的监控架构缺陷

把告警规则判断和推送客户端集成在应用内部，将使系统陷入严重的致命性单点崩溃陷阱。

当线上服务遇到灾难性故障（例如 Node.js 进程由于 OOM 被操作系统强杀，或者所在的云服务器实例发生硬件宕机）时，最需要告警系统发出红色警报。然而，因为告警模块也是该应用进程的一部分，它会随着应用的崩溃一同死亡。此时外部将处于一片彻底的寂静中，没有任何告警发出，直至用户投诉涌向客服台。

外部监控系统天然避免了这种同生共死的设计缺陷。当应用宕机时，它对外的 Metric 暴露端点会瞬间静默。外部监控系统在连续数个采样周期探测不到心跳信号后，将自动触发“死人开关（Dead Man's Snitch）”或者“无数据告警（No Data Alert）”，将宕机事故瞬间升级给值班工程师。

## 主线演练：Relay 告警字段契约校验

当应用将告警的评估移交给外部系统后，日志中的关键 JSON 字段名称（如 `error_code`、`http_status`、`latency_ms`）就升格为了跨系统的“告警契约”。一旦开发人员或 Agent 在重构代码时随意更改了日志字段名，外部告警系统就会因为读不到预期指标而陷入“静默失效”状态。

在 Relay 示例中，我们在云端的 Loki / Prometheus 警报管理中定义了如下告警策略。在此之前，我们使用 Prometheus 的 `Alertmanager` 配置文件来声明对 Relay 服务的心跳探测契约：

```yaml
# prometheus-alert-rules.yaml
groups:
  - name: relay_infrastructure_rules
    rules:
      # 死人开关：如果 5 分钟内没有从 relay 获取到任何时序指标，说明服务已完全宕机
      - alert: RelayServiceDeadManSwitch
        expr: absent(up{job="relay-app"})
        for: 5m
        labels:
          severity: critical
        annotations:
          summary: "Relay backend instance has vanished from metrics stream"
          description: "No scrape data received for job relay-app over the last 5 minutes. The service might be offline or blocked."
```

为了防止 Agent 或开发人员在重构时把关键字段改掉，我们在测试套件中为核心日志格式定义了如下契约测试：

```typescript
// test/observability/log-contract.test.ts
import { expect, test } from 'vitest';
import Ajv from 'ajv';

const ajv = new Ajv();

// 声明外部监控系统强依赖的告警日志 JSON Schema 契约
const refundAlertLogSchema = {
  type: 'object',
  required: ['event', 'trace_id', 'error_code', 'refund_amount_cents'],
  properties: {
    event: { type: 'string', const: 'stripe_api_refund_failed' },
    trace_id: { type: 'string' },
    error_code: { type: 'string' }, // 监控系统用此字段来归类统计报错类型
    refund_amount_cents: { type: 'integer' }
  }
};

const validate = ajv.compile(refundAlertLogSchema);

test('Log contract: refund error log must contain exact warning fields for Alertmanager', () => {
  const dummyLogEvent = {
    event: 'stripe_api_refund_failed',
    trace_id: 't_test_9921',
    error_code: 'INSUFFICIENT_FUNDS',
    refund_amount_cents: 12500
  };

  const isValid = validate(dummyLogEvent);
  expect(isValid).toBe(true);
});
```

### 调试挫败细节：字段重构引发的告警静默

在 Relay 项目重构过程中，Agent 曾在执行全局变量重命名时，将 `stripe_client.ts` 捕获异常后抛出的日志字段名进行了优化：

```diff
- logger.error({ error_code: 'STRIPE_ERROR', ... }, 'Refund failed');
+ logger.error({ err_code: 'STRIPE_ERROR', ... }, 'Refund failed');
```

这次修改虽然顺利通过了编译和原本孱弱的单元测试，并在测试环境中以绿灯合规形式合入了主干。然而在部署预发布环境的当晚，第三方支付通道 Stripe 发生了大范围网络抖动。尽管 Relay 线上堆满了支付失败的日志，但外部监控平台配置的告警查询式 `sum(rate(relay_error_count{error_code="STRIPE_ERROR"}[5m]))` 却始终返回为 0。

整个告警线对于此次重大事故毫无反应。这次严重的故障漏报使得团队深刻意识到，必须将告警字段视为 API 级的契约，从而在测试库中补充了上面那套 `Ajv` 日志 Schema 校验测试。后续当 Agent 试图再次简化日志字段名时，CI 会强制在编译期将改动拦截。

## 典型反模式

* **应用自告警设计**：在 Node.js 单进程中起一个内部的告警守护 Cron，当应用由于内存泄露出现未捕获的僵死状况时，告警引擎也一同宕机，导致故障被彻底封锁在黑盒中。
* **盲目修改日志字段**：无规则重构可观测性打印逻辑，将日志字段名由 `latency` 缩写为 `lat`，导致外部 Grafana 指标大面积断流。
* **缺乏死人开关**：只配置了针对“高错误率”的阈值告警，而忽略了整体网络瘫痪、流量完全进不来时的“无数据（No Data）”警报，在系统遭受重大宕机时大盘依然显现为和平的绿色。

## 本章要点

* 监控必须解耦为两端：应用只负责低阻抗抛出信号，告警规则判断与派发全量交由外部。
* 死人开关（Dead Man's Snitch）是抵抗服务整体崩溃的终极防护，不需要应用配合任何代码即可工作。
* 外部告警依赖的日志 JSON 键名属于系统间契约，必须通过 Schema 契约测试（如 Ajv）实施静态或动态防护，防止 Agent 重构越界。

> 信号已暴露，外部契约防线已锁死。但是当告警系统在大半夜把开发团队拉起来时，排障 Agent 该如何快速介入？我们将在第 19 章 `让 AI 直接够到线上` 中探讨如何安全地向 Agent 暴露线上只读通道。
