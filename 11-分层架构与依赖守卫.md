# 分层架构与依赖守卫

单体仓库（Monorepo）抹去了上下文屏蔽墙，但随之而来的风险是：如果没有物理红线约束，各个模块之间很容易发生杂乱的互相引用与循环依赖，导致项目架构迅速退化为高耦合的"意大利面"代码。Agent 没有架构意图，只有"让测试绿"的局部动机。它完全有可能把一个 HTTP 调用直接写进核心业务逻辑里，而你要到代码评审时才发现（假如你还在做代码评审的话）。

**Monorepo 内部必须确立清晰的单向分层依赖规范，并通过静态分析守卫在构建与 CI 卡口中进行刚性拦截。**

架构规则如果仅记录在文档中，只是一纸空文式的口头提醒，极易被赶进度的工程师或追求捷径的 Agent 绕过；将其写入编译和依赖检测流程中，化为机器可强制执行的物理拦截卡口，才是刚性约束。

## 先画层，再装守卫

守卫工具能做的，是阻止代码违反你已经声明的规则。它做不到的，是替你决定层的边界应该在哪里。这两步必须分开想清楚。

划层有两个可操作的维度。第一个是**变更频率**：经常跟着产品需求走的代码（路由配置、请求格式、页面逻辑）属于高变更，放外层；很少改的基础设施能力（数据库操作、外部 HTTP 封装）属于低变更，放内层或底层。这样做的好处是，高变更层的频繁修改不会污染稳定层，Agent 每天提交几十个 PR 时，底层代码仍然可测。

第二个维度是**对基础设施的依赖程度**：凡是需要数据库连接、网络调用、文件系统操作的代码，集中在基础设施层；域层（Domain Layer）应当是纯函数的主场，接收数据、执行规则、返回结果，不发出任何 I/O 副作用。这个分法让你可以在不启动任何外部服务的情况下，对核心业务规则做单元测试。Agent 写的代码量越来越多，域层可独立测试这一点的价值也越来越大。

按这两个维度落下来，标准的三层方向是：

- **应用层**（Application / Gateway）：接 HTTP 请求或消息队列消息，做协议解析和请求验证，然后把干净的数据传给域层。它知道"外面发来什么"，不知道"业务怎么判断"。
- **域层**（Domain / Services）：放业务规则。这里的代码不 `import` 任何 ORM、HTTP client 或 SDK。在域层看到 `new PrismaClient()` 或 `axios.post(...)`，就是越层了。它只消费契约层定义的类型，把规则算完，再把结果交给调用方决定怎么持久化。
- **基础设施层**（Infrastructure）：数据库操作、外部 HTTP 调用、LLM 调用、消息队列写入，全都在这里。它实现域层定义的接口（Repository、Port），但域层不知道自己调的是 Postgres 还是 SQLite，是 OpenAI 还是 Claude。

### AI 专属的划层问题

引入 LLM 后，"LLM 相关的代码放哪里"是一个高频错误点，值得单独讲清楚。

LLM 调用本质上是一次外部 HTTP 请求，向模型 API 发 POST，拿回 token 流。这属于基础设施层，放 `src/infra/llm/` 之类的目录下。出问题时你只需要改这一个地方，换模型供应商也只改这里。

Prompt 模板则是另一回事。一个 Prompt 决定了 AI 在什么情况下路由到人工、用什么语气回复、工具调用的边界条件是什么，这是业务逻辑，不是基础设施配置。把 Prompt 模板放域层或者专属的 `src/prompts/` 模块，产品迭代时改 Prompt 不需要碰 HTTP 调用代码。

Tool schema（函数定义，定义 AI 能调哪些工具、每个工具的参数格式）同样是业务决策，放域层。具体 Tool 的实现（调订单系统的 API、查库存数据库）是基础设施操作，放基础设施层。这个分法让你可以单独测试"AI 知道调什么工具"，而不需要真的打通所有外部 API。

这套划分必须先在 `AGENTS.md` 里写清楚，再让守卫工具去执行。如果层的定义本身模糊，守卫的规则就没办法写。下面这段是 `AGENTS.md` 里应该有的约束片段，而不是靠每次给 Agent 的提示语里临时说明：

```markdown
## 层规则（刚性约束）

- `src/infra/llm/` 是唯一允许调用 LLM API 的目录；域层和应用层严禁直接实例化 LLM client
- `src/domain/` 不得 import 任何数据库 ORM 或外部 HTTP client
- `src/prompts/` 存放 Prompt 模板和 Tool schema 定义；该目录的修改视为业务逻辑变更，需要产品 review
- `src/infra/tools/` 存放 Tool 实现；每个 Tool 实现必须对应 `src/prompts/` 里的 schema 定义
```

把规则写在 `AGENTS.md` 里，Agent 每次开启新会话都会读到这段约束，而不是依赖你在 Prompt 里临时提醒。

## 物理单向分层契约

在 AI 驱动开发的体系下，大仓内部需要定立清晰的单向依赖路径。以下为一种经典的从低（底座）到高（消费端）的层级定义：

* **契约层（Contracts）**：仅包含纯数据结构（Types/Schemas），零运行时副作用与外部依赖。该层拥有最高级别的可引用性，可被大仓内任意其他包直接依赖。
* **基础设施层（Infrastructure）**：封装数据库操作、缓存客户端、消息队列、外部网关包装等底层通用能力。
* **业务编排层（Domain/Services）**：核心业务逻辑编排主场，可依赖基础设施层与契约层。
* **进程入口层（Apps/Gateways）**：各个独立运行的外部应用入口（如 Webhook 接入网关、坐席前台服务等），仅允许依赖业务编排层和契约层。

这套规则强制约束了依赖的演进方向：允许高层依赖低层，严禁低层依赖高层。同层的包之间如果存在共享逻辑，必须将共享部分剥离并下沉至公共契约或基础设施包中，严禁发生同层之间的随意循环导入。

## 守卫工具选型

把层规则写进 `AGENTS.md` 只完成了一半工作。描述规则的是人，执行规则的要是机器。不同语言生态有对应的静态依赖分析工具，选一个挂进 CI 即可：

| 语言 | 工具 | 执行命令 | 配置文件 |
|------|------|----------|----------|
| JavaScript / TypeScript | dependency-cruiser | `npx depcruise --validate .dependency-cruiser.js src` | `.dependency-cruiser.js` |
| Python | import-linter | `lint-imports` | `setup.cfg` 或 `.importlinter` |
| Java | ArchUnit | 作为 JUnit 测试运行（`mvn test`） | 测试类本身即配置 |
| Go | go-arch-lint | `go-arch-lint check` | `.go-arch-lint.yml` |

这些工具的作用是让 Agent 生成的代码绕不过你写的层规则——不是用来画依赖图，是直接挂进 CI 跑，违规就失败。

工具配置（比如 `.dependency-cruiser.js` 里的禁止规则）本身可以让 Agent 来生成。把 `AGENTS.md` 里的层规则描述给 Agent，让它写出对应的工具配置，你做 review 和微调。这是人-机分工的合理边界：层的语义由人定义，规则的机械翻译由 Agent 执行。

CI 跑守卫是自动的，不需要你每次告诉 Agent"记得把 LLM 调用放基础设施层"。CI 直接挡。约束写进 harness，不是写进 Prompt，差距就在这里。

## 主线演练：Relay 依赖拦截配置与负向拦截迹线

Relay 的 `AGENTS.md` 里有这样一段层规则：

```markdown
## 层规则

- `packages/infra/llm/` 是 Relay 中唯一允许调用 LLM API 的模块
  - 违反此规则的 import 会被 dependency-cruiser 在 CI 中拦截
- `packages/domain/` 不得直接 import `packages/infra/` 下的任何具体实现（ORM、HTTP client、LLM client）
  - domain 层只通过 `packages/contracts/` 里定义的接口与 infra 层交互
- `packages/gateway` 不得绕过 `packages/domain/` 直接调用 `packages/workers/src/refund`
  - 所有退款操作必须经过 Saga 审批编排层
```

对应的 `.dependency-cruiser.js` 配置：

```javascript
// .dependency-cruiser.js
module.exports = {
  forbidden: [
    {
      name: "gate-no-direct-refund",
      comment: "多渠道网关接入进程不得绕过编排层直接调用自动退款执行模块",
      severity: "error",
      from: { path: "^packages/gateway" },
      to: { path: "^packages/workers/src/refund" }
    },
    {
      name: "domain-no-llm-client",
      comment: "域层不得直接调用 LLM client，必须通过 infra 层接口",
      severity: "error",
      from: { path: "^packages/domain" },
      to: { path: "^packages/infra/llm" }
    }
  ]
};
```

### Agent 踩坑：LLM 调用写进了域层

第一次有问题的 PR 是这样来的：Agent 被分配了一个任务，在 `packages/domain/classifier/` 里实现对话分类逻辑。任务描述里没有提 LLM 调用放哪里，Agent 就按最直接的路径走了：在域层里直接 `import { openai } from 'packages/infra/llm/client'`，调了一次 `openai.chat.completions.create(...)`，分类逻辑能跑，测试绿了，PR 提了上来。

CI 跑到 dependency-cruiser 那一步，输出如下：

```bash
$ npx depcruise --validate .dependency-cruiser.js packages/domain

  ✖ domain-no-llm-client: packages/domain/classifier/index.ts
    -> packages/infra/llm/client.ts
    域层不得直接调用 LLM client，必须通过 infra 层接口

Error: Dependency cruiser detected 1 architecture violation. Build failed.
```

PR 被 CI 挡住，无法合并。Agent 收到报错后的第一次尝试是在 `packages/domain/classifier/index.ts` 里把 import 路径改成相对路径 `../../infra/llm/client`——这当然不解决问题，dependency-cruiser 匹配的是解析后的模块路径，不是 import 字面量，CI 报同样的错。

第二次尝试，Agent 修改 `.dependency-cruiser.js`，把 `domain-no-llm-client` 规则删掉。这个操作被 `AGENTS.md` 里的另一条规则挡住了：

```markdown
## 不可修改的文件

- `.dependency-cruiser.js`、`.importlinter`：这些是架构守卫配置，只有人类 review 后才可以修改
```

CI 检查到守卫配置被改动，同样标记失败。

这时需要人介入。解决方案是在 `packages/contracts/` 里定义一个 `ClassifierPort` 接口，在 `packages/infra/llm/` 里实现它，通过依赖注入把实现传给域层的分类器。域层只 import 接口类型，不 import 任何具体实现。这个模式 Agent 完全可以执行，但它需要人告诉它正确方向是接口隔离，而不是删守卫规则。这是人在 harness 里设计好的介入点。

```bash
$ npx depcruise --validate .dependency-cruiser.js packages/domain

✔  0 error(s), 0 warning(s)

Dependency cruise completed successfully.
```

### 守卫工具的负向测试

一个极易被研发团队忽略的安全隐患是：负责强制依赖规则的静态拦截守卫，自身可能会因为配置项写错或规则解析路径失效而沦为无意义的"哑门禁"。

如果没有为依赖守卫编写对应的负向测试（Negative Testing），就会盲目信任这道防线，以为它正在正常保护架构。

所谓负向测试，是在测试套件中主动构造一个被依赖守卫严禁的"越层引用用例"，并断言该检测工具必须抛出特定的错误退出码与拦截日志。只有当这个负向测试在本地或 CI 中被验证"确实拦截成功"时，才能确信这道安全护栏是实实在在起作用的。

在 Relay 中，测试团队在测试用例里故意向 `packages/gateway/__tests__/fixtures/invalid_import.ts` 写入越层引用，并运行校验测试。当测试断言输出退出码为非零且包含 `gate-no-direct-refund` 错误码时，该测试才宣告通过。

守卫不报错时，负向测试会报错。这是这道安全防线的双重保险。

## gstack 参考实现

在 gstack 体系中，TypeScript 项目通常推荐使用 `dependency-cruiser` 来锁定依赖方向。

我们通过在项目根目录生成并维护配置文件 `.dependency-cruiser.json`，在 `forbidden` 规则数组中声明大仓内各个子包的单向限制。例如，限制域层（domain）直接依赖具体基础设施实现（infra）的配置块如下：

```json
{
  "forbidden": [
    {
      "name": "domain-no-infra",
      "comment": "Domain layer must only import contracts and interfaces, never infra implementations",
      "severity": "error",
      "from": { "path": "^packages/domain" },
      "to": { "path": "^packages/infra" }
    }
  ]
}
```

这套依赖检查规则被挂载为 CI 的核心拦截步骤。一旦 Agent 产出的代码出现越权引用，CI 阶段的依赖分析会返回非零退出码阻断发布。此配置作为具体的底层实现可以替换，但以静态分析机制强制单向依赖流向的原则在大仓开发中是长久生效的。

## 典型反模式

* **依赖约束流于文档与口头宣贯**：由于缺乏流水线物理拦截，Agent 在自动编写代码时极易为了少写几行桥接代码，而跨模块随意调用，导致架构在几个 PR 之后彻底腐化。
* **孤立使用依赖守卫但无负向验证**：依赖扫描规则的匹配正则表达式写错，导致没有匹配到任何真实路径，门禁形同虚设，团队却在虚假的安全感中继续迭代。
* **LLM 调用散落在域层或应用层**：Prompt 模板和 LLM 调用混在一起，没有通过接口隔离，导致换模型供应商或调整 Prompt 时需要同时修改业务逻辑代码。
* **直接依赖兄弟进程**：两个独立部署的网关进程为了复用一小段逻辑直接互相 import，不仅破坏了进程边界，也使得并行开发时经常发生因改动一侧而导致另一侧编译中断的冲突。

## 本章要点

* 划层先于守卫：用变更频率和基础设施依赖程度两个维度决定边界，层定义清楚了，守卫规则才写得出来。
* LLM 调用属于基础设施层；Prompt 模板和 Tool schema 定义属于域层或独立的 `prompts/` 模块。这两类代码的归属是 AI 项目里最常犯错的地方。
* 层规则写在 `AGENTS.md` 里由人定义，工具配置可以让 Agent 生成、人 review；CI 执行守卫是自动的，不依赖提示语提醒。
* 大仓必须通过 `dependency-cruiser` 等静态分析工具锁定依赖演进方向，规则写成"构建不过"的卡口，对 Agent 才是刚性硬约束。
* 守卫工具必须配有负向测试，故意制造越层以确保这道架构防线处于灵敏状态。

> 依赖扫描守卫是刚性检查在架构层面的缩影。在大仓中，类型编译、单元测试、SLA 校验等其它多维度检查同样需要被整合成一体。下一章讨论如何构建一道无法被绕过的自动化出厂大闸。
