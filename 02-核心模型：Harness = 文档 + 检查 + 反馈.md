# 核心模型：Harness = 文档 + 检查 + 反馈
你在开发中一定遇到过这种荒谬的场景：刚在提示词里严厉警告了三遍“金额必须用分，严禁使用浮点数”，Agent 下一秒依然生成了 `amount: 19.9` 并塞进了数据库；或者为了修补一个报错，Agent 开始在原地疯狂生成几十个无意义的微调 Commit，直到把上下文窗口彻底撑爆。

你无法通过反复叮嘱（Prompt 提示词）去驯服一个概率输出的非确定模型。要想让 AI 可靠且稳定地干活，必须为它建立一个具备刚性物理阻断与自愈能力的脚手架。

**这一能让 Agent 自愈与进化的系统脚手架（Harness），由文档（约束输入边界）、检查（拦截输出偏差）与反馈（驱动闭环自愈）三根承重柱组成。**

这三者并非孤立的功能清单，而是构成一个闭合的控制回路。文档作为冷启动的上下文输入，定义了 Agent 动作的边界；Agent 的产出必须通过检查卡口的拦截；测试与线上的运行结果产出结构化反馈；在此过程中暴露的系统缺陷或逻辑漏洞被重新沉淀为文档与新的检查规则。随着这一回路的循环运转，研发环境的约束力将逐步紧固。


## 文档：约束输入边界

在 Harness 体系中，文档控制的是系统的输入，负责在源头上确立研发环境的不变量与红线规则。

由于 Agent 在每次会话中都是冷启动的，其上下文窗口在起跑线上是一片空白。文档是其获取项目特定规则（Project-specific rules）的唯一来源，包括 `./AGENTS.md` 中的架构红线、接口规范以及历史的设计决策。如果这些规范没有以物理文件的形式沉淀于代码库中，而是散落在历史的对话历史里，一旦会话重置，环境约束便会发生漂移。

文档的回报具有复利效应。在 `./AGENTS.md` 中增加的一条针对特定依赖库的引入禁令，将成为后续所有 Agent 会话的初始输入，从而避免同一个设计错误被多次重复。

## 检查：拦截输出偏差

如果说文档在输入端立起了规则的栅栏，那么检查就是代码合并至主干前的物理拦截门禁。它包括 Lint 静态扫描、编译校验、单元测试、集成测试以及依赖规则守卫。

检查的核心特征是**机器强制性**，而非人为建议。Agent 提交的代码如果无法通过检查，就无法被合入。这不依赖于团队成员是否记得手动运行脚本，也不依赖于评审人员的肉眼洞察。

在机器强制下，“约束大于提醒”的原则得到了最直接的体现。在提示词中叮嘱“金额严禁使用浮点数”，模型在处理长上下文时仍有概率遗忘；但若在构建流水线中配置一条检测浮点操作的静态规则，代码一旦违规，构建便会红线拦截。当研发链路中存在多个并发工作的 Agent 时，这种刚性的检查卡口是唯一的安全防线。



## 反馈：驱动闭环自愈

仅仅把代码拦下还不够，大语言模型必须依赖运行期的结果反馈进行自愈。如果反馈信号模糊（例如只抛出含糊的“运行报错”），Agent 会在盲区中胡乱试错。

反馈的来源从本地到线上贯穿整个研发周期，具体分为以下三类：

1. **检查与测试反馈（本地开发期）**：
   **检查的失败输出本身就是最直接的本地反馈。** 相比于毫无头绪的进程崩溃，诸如 `expected 1200 cents, but got 1199` 这样的精确断言，能够直接为 Agent 指明修正方向，促使其快速实现自我纠正。因此，门禁不仅需要“拦得住”，更需要“说得清”。
2. **端到端自验证反馈（预发/沙箱期）**：
   静态检查与单元测试无法完全覆盖 UI 渲染与复杂业务流逻辑。必须强制 Agent 在隔离的沙箱中启动无头浏览器模拟真实用户路径，核对最终的页面呈现，完成最后一公里的物理验证。
3. **线上生产环境反馈（线上运行期）**：
   源于生产环境的结构化日志与监控指标。这里存在一个关键的设计要求：必须确保 Agent 拥有受控的、最小权限的日志访问通道。如果生产日志仅存储在人类管理员才能登录的后台系统，对 Agent 而言就是信息孤岛。为其提供只读的、限定上下文的日志拉取接口，是其在线上实现问题自主定位的前提。

## Relay 演练：三件套在退款特性中的协同

以 Relay 的自动退款流程为例，展示这三根承重柱如何交织运转：

在 Agent 动手开发前，**文档**环境已经就绪。相对项目根目录的 `./AGENTS.md`（Agent 宪法）明确定义了资金安全硬约束：
```markdown
## 资金安全约束
- 所有金额字段必须使用整数分（Cents），严禁使用浮点数。
- 单次自动退款限额为 50,000 分（500 元）。超出此限额必须调用 `createApprovalRequest`，由人工审批，不得直接调用自动退款服务。
```
Agent 必须在冷启动阶段读取此规则，以此作为编码的边界输入。

在 Agent 完成编码提交时，**检查**门禁开始执行。Agent 在分摊优惠券时误用了浮点数计算。在执行本地 CI 静态校验时，门禁拦截并输出了以下迹线：
```bash
$ npm run lint:finance
> [Lint Error] float_usage_detected: Float operations found in src/services/refund.ts (Line 42). Financial calculations must use Cents integer.
```
这一精确的报错信息转化为**反馈**送回给 Agent，引导它迅速将代码重构为整数运算。

在预发沙箱环境中，一笔退款交易由于网络网关超时触发了重试。Agent 通过受控的日志接口调取了预发日志：
```bash
$ grep "refund_attempt" logs/sandbox.log
{"level":"info","time":"...","conversation_id":"conv_99","msg":"Initiating refund of 1200 cents"}
{"level":"error","time":"...","msg":"Gateway timeout, retrying...","conversation_id":"conv_99"}
{"level":"info","time":"...","conversation_id":"conv_99","msg":"Initiating refund of 1200 cents"}
```
分析迹线后，Agent 发现由于接口未携带唯一幂等键，导致下游支付网关重复扣款。人机协作在此处发生：人在看到 Agent 陷入重试循环的挫败后，引导其在 `./AGENTS.md` 中追加幂等要求，并强制编写了一组网络超时重试的黄金测试用例。

最终，该交易重复扣款的漏洞被成功修补，且转变为 CI 黄金测试库中的永久用例（检查）。针对幂等性的约束被永久写入了文档中。环境约束再次收紧了一格。

## 典型反模式

* **盲目依赖人工 Review**：不建立自动化的测试与编译卡口，将代码质量完全寄托于人类评审的眼力。在 Agent 并发提交高频代码的场景下，这必然会导致漏洞漏网。
* **低质反馈导致调试漂移**：CI 门禁在报错时仅输出泛化的 `Command failed with exit code 1`，而不提供具体的失败堆栈与 Diff 信息，导致 Agent 在修复时失去明确的方向，陷入盲目试错的死循环。
* **上下文失忆**：不维护持久化的 `./AGENTS.md`，每次对话都通过 Prompt 临时灌输业务约束。一旦 Agent 切换至新的会话，之前的规则约束将彻底失效。

## 本章要点

* **Harness 核心支柱**：脚手架由文档（约束输入边界）、检查（拦截输出偏差）与反馈（驱动闭环自愈）三部分互补构成。
* **闭环控制回路**：文档沉淀规则 → 检查执行强制拦截 → 反馈提供修正信息并指导文档与检查的升级。
* **强制性高于建议性**：检查卡口必须由机器自动化运行且具备一票否决权，不可依赖人工记忆。
* **可访问的线上反馈**：必须为 Agent 开放受控的结构化日志拉取权限，否则其在线上故障排查中将彻底失明。
