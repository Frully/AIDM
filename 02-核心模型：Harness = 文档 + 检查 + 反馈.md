# 核心模型：Harness = 文档 + 检查 + 反馈

在提示词中写满三遍“金额必须用分，严禁使用浮点数”，Agent 在处理长上下文时依然有概率生成 `amount: 19.9` 并写入数据库。试图依靠文字反复叮嘱（Prompt）去驯服一个概率输出的非确定性模型，在工程上是不可靠的。当缺乏刚性约束时，Agent 甚至会为了修补一个编译错误，在原地生成几十个无意义的微调 commit 导致上下文窗口彻底撑爆。

**可靠的 AI 驱动开发不依赖更聪明的提示词，而依赖由文档（约束输入边界）、检查（拦截输出偏差）与反馈（驱动闭环自愈）三根承重柱构成的刚性脚手架（Harness）。**

这三者不是零散的功能拼凑，而是互咬的闭环控制系统：文档作为冷启动的确定性上下文输入，划定行为边界；检查卡口在交付前对代码进行刚性物理拦截；测试与运行期的迹线反馈直接喂回给 Agent 驱动其自主修正；在自愈与开发中暴露的系统缺陷，则重新固化为新的文档约束与检查卡口。整个研发环境随着这一回路的运转不断收紧，实现环境质量的阶梯式抬升。


## 文档：约束输入边界

大语言模型在每次会话（Session）中都是冷启动的，上下文处于真空状态。如果不将架构决策、红线指标和接口定义以物理文件的形式沉淀在代码库中，而是散落在漫长的对话记录里，那么一旦会话重置，环境约束就会瞬间清零。

在 Harness 体系中，文档是唯一的持久化记忆载体。我们通过 `./AGENTS.md` 统一构建规则树：

* **全局宪法（`AGENTS.md`）**：它是 Agent 冷启动后必读的第一份规则清单。除了记录全局不变量与红线禁令外，它最核心的任务是作为入口索引，明确列出项目中各项具体设计决策、接口规范和架构决策文档的相对路径。冷启动的 Agent 能够按图索骥，精准加载对应的细节上下文，无需在庞大的 Monorepo 中盲目检索。
* **设计决策文档（ADR）**：记录系统关键的技术取舍与被否决的替代方案。它向 Agent 揭示了代码背后的架构动机，能够防止中途接手的 Agent 因无法理解防错分支的用意，而将为了容错设计的异步隔离层或 Saga 回滚机制当成无用死代码误删。
* **接口与类型定义（Specs）**：将接口契约（如 API 结构、类型定义）作为独立先行任务进行锁定。这构成了并行开发的安全接口桩。

文档的回报具有复利效应。在 `./AGENTS.md` 中增加的一行规则或是一份新模块的设计索引，将立刻成为后续所有 Agent 启动时的前置输入，让系统演进的轨道越来越稳固。


## 检查：拦截输出偏差

检查的核心特征是机器强制的物理门禁，而非可选择性遵守的开发建议。无论是 lint 扫描、类型编译，还是单元/集成测试和依赖守卫，它们必须被配置为“一票否决制”的构建门禁。

“约束大于提醒”在此处是绝对的：将金额禁浮点数的红线配置为静态 Linter 规则，代码一旦违规，构建流水线立即红盘报错，强行切断合入路径。这构成了多 Agent 并行研发时阻断逻辑退化的物理防线。

> [!IMPORTANT]
> **术语红线钉死**：本书用「检查」专指此类由机器执行、结果二元判定的自动化门禁（如 lint、编译、依赖守卫、测试）；而「验证」则是生命周期中的阶段概念，它包含检查门禁、独立评审、业务验收和可逆上线。两词概念完全不同，全书绝不混用。


## 反馈：驱动闭环自愈

仅有拦截是不够的，模型需要能够据此自愈的诊断信号。模糊的反馈（如抛出一个通用的构建失败状态码）只会让 Agent 盲目猜测，进而产生无意义的试错循环。反馈不仅要“拦截住”，更要“说得清”。

反馈贯穿本地至线上的整个研发链条：

1. **本地检查反馈**：测试框架输出的精确 diff 与报错堆栈（如 `expected 1200 cents, but got 1199`）是 Agent 最直接的纠错信号，促使其在本地自动重构。
2. **沙箱端到端反馈**：Playwright 等端到端自动化自验证。静态与单元测试拦不住 UI 渲染塌陷或交互死锁，必须强迫 Agent 启动无头浏览器模拟真实用户路径，核对最终的页面呈现。
3. **生产可观测性反馈**：源自线上结构化日志与指标。为了实现线上排障闭环，必须为 Agent 开放受控的、只读的日志拉取 CLI 通道，解除其在线上故障定位时的“视觉失明”。


## Relay 演练：三件套在退款特性中的协同

以 Relay 客户支持 SaaS 的自动退款流程为例，展示这三根承重柱的闭环交织运转：

1. **文档环境就绪**：在 Agent 动手开发前，相对项目根目录的 `./AGENTS.md`（项目宪法）已明确定义资金硬约束：
   ```markdown
   ## 资金安全约束
   - 所有金额字段必须使用整数分（Cents），严禁使用浮点数。
   - 单次自动退款限额为 50,000 分（500 元）。超出此限额必须调用 `createApprovalRequest`，由人工审批，不得直接调用自动退款。
   ```
   Agent 必须在冷启动阶段读取此规则，以此作为编码的边界输入。
2. **检查门禁拦截**：Agent 在分摊优惠券时误用了浮点数计算。在执行本地 CI 静态校验时，门禁拦截并输出了以下迹线：
   ```bash
   $ npm run lint:finance
   > [Lint Error] float_usage_detected: Float operations found in src/services/refund.ts (Line 42). Financial calculations must use Cents integer.
   ```
3. **反馈驱动自愈**：这一精确报错直接反馈给 Agent，引导它迅速将代码重构为整数运算。
   但在随后的沙箱环境中，一笔退款由于网关超时触发了重发。Agent 调取了预发日志：
   ```json
   {"level":"info","time":"2026-06-29T18:00:01Z","conversation_id":"conv_99","msg":"Initiating refund of 1200 cents"}
   {"level":"error","time":"2026-06-29T18:00:03Z","msg":"Gateway timeout, retrying...","conversation_id":"conv_99"}
   {"level":"info","time":"2026-06-29T18:00:04Z","conversation_id":"conv_99","msg":"Initiating refund of 1200 cents"}
   ```
   分析迹线后，Agent 发现由于接口未携带唯一幂等键，导致下游重复扣款。人机协作在此处发生：人在看到 Agent 陷入重试循环的挫败后，介入并引导其在 `./AGENTS.md` 中追加幂等红线，并补充了测试。

最终，漏洞被修补，且转变为 CI 门禁中的永久用例（检查）。针对幂等性的约束被永久写入了文档中，环境约束再次收紧。


## 典型反模式

* **盲目依赖人工 Review**：不建立自动化的测试与编译卡口，将代码质量完全寄托于人类评审的眼力。在 Agent 并发提交高频代码的场景下，这必然会导致漏洞漏网。
* **低质反馈导致调试漂移**：CI 门禁在报错时仅输出泛化的 `Command failed with exit code 1`，而不提供具体的失败堆栈与 Diff 信息，导致 Agent 在修复时失去明确的方向，陷入盲目试错的死循环。
* **上下文失忆**：不维护持久化的 `./AGENTS.md`，每次对话都通过 Prompt 临时灌输业务约束。一旦 Agent 切换至新的会话，之前的规则约束将彻底失效。


## 本章要点

* **Harness 核心支柱**：脚手架由文档（约束输入边界）、检查（拦截输出偏差）与反馈（驱动闭环自愈）三部分互补构成。
* **闭环控制回路**：文档沉淀规则 → 检查执行强制拦截 → 反馈提供修正信息并指导文档与检查的升级。
* **强制性高于建议性**：检查卡口必须由机器自动化运行且具备一票否决权，不可依赖人工记忆。
* **可访问的线上反馈**：必须为 Agent 开放受控的结构化日志拉取权限，否则其在线上故障排查中将彻底失明。
