# 并行 Worktree：双层隔离与自动装配

在 Relay 早期尝试并行开发时，团队曾向三个独立的 Agent 会话分发了不同的原子任务：A 负责退款决策逻辑，B 负责路由分类器重构，C 负责修复坐席审批界面的前端缺陷。由于未做任何物理隔离，所有会话均直接指向同一个物理目录。仅两分钟后，A 和 B 同时修改了 `src/router/classifier.ts` 导致代码被轮流覆写；而 C 试图启动本地开发服务时，因 3000 端口已被 A 抢占而抛出 `EADDRINUSE` 异常；B 会话写入的脏测试记录也直接污染了 C 的 Playwright 测试沙箱，导致端到端断言全部红盘。在资源冲突中互踩半小时后，两个会话的改动不得不被完全废弃，研发退回单线串行模式。

并行的失效并非由于多会话机制本身，而是因为缺乏隔离栅栏与执行纪律。

**高效的多线并行开发依赖“静态代码隔离”与“动态环境自动装配”构成的双层物理防线，并必须辅以刚性的生命周期清理纪律，否则物理资源冲突会将并发收益吞噬殆尽。**


## 并行能力从多开会话来

虽然大模型客户端或 CLI 工具允许用户同时开启多个会话（Session），以充当独立开发特定任务的虚拟工程师，但会话层面的隔离仅是逻辑上的。一旦逻辑会话落地到物理文件系统与本地运行时环境，由于没有天然边界，便会产生文件冲突、端口冲突和数据污染。只要这三类物理冲突中有一项未得到彻底隔绝，并发协作就会产生比单线串行更为棘手的系统性混乱。


## 第一层：git worktree 隔代码

`git worktree` 允许同一个仓库把多个分支检出到不同物理目录，每个目录有自己的工作区，修改互不可见，但共享同一套提交历史和对象库。

```bash
$ git worktree add ../wt-refund-01 feat/refund-decision
$ git worktree add ../wt-routing-02 feat/classification

$ git worktree list
relay-main      [main]
wt-refund-01    [feat/refund-decision]
wt-routing-02   [feat/classification]
```

会话 A 指向 `wt-refund-01`，会话 B 指向 `wt-routing-02`，两个 Agent 同时改文件落在不同目录，不会相互覆盖。worktree 在注册表层面也做了互锁：同一分支不能同时检出到两个 worktree，防止意外并发写入。这一层解决“文件互踩”，但仅此而已。


## 第二层：脚本隔动态环境

代码隔开后，运行时冲突还在：两个 Agent 的本地服务默认都监听 3000，数据库默认都连同一个实例。解决方案是会话启动时自动给每个 worktree 分配专属端口和专属数据库，写进该目录下的 `.env`：

```ini
# wt-refund-01/.env          # wt-routing-02/.env
PORT=3001                    PORT=3002
DATABASE_URL=.../relay_sandbox_wt_refund_01   DATABASE_URL=.../relay_sandbox_wt_routing_02
REDIS_URL=redis://localhost:6379/1            REDIS_URL=redis://localhost:6379/2
```

只有两层都到位，Agent 才能真正互不干扰地跑完“开发→测试→自验证”整个循环。漏掉任何一层，剩下的一层都只是部分缓解。


## 自动装配：从“能隔离”到“插电即用”

手动挑选空闲端口、建测试库、修改环境变量等繁琐过程，很容易打消团队使用并行的积极性，最终退回单线模式。所以装配必须自动化，且遵守四条硬性原则：

* **会话启动自动触发**：用 Git Hook 拦截 Agent 第一次切入工作目录的时刻自动装配，不靠人手动跑命令。
* **决定性端口派生**：按并发安全分配的“槽位（Slot）”编号，以 `基准端口 + 槽位偏移` 算术派生专属端口。严禁随机端口检测——那会让端口在每次重启时发生漂移，导致多模块联调难以稳定复现。
* **分配机制并发安全**：几条并行线可能在同一秒被拉起，必须使用物理文件锁（flock）对共享槽位表排他加锁，防止两个会话抢到同一端口槽位。
* **装配逻辑幂等**：脚本可重入，同一目录多次运行结果一致，已起好的环境自动复用。

在此过程中，系统仅隔离会引发物理冲突的实体（端口、沙箱数据库名、Redis 实例），复用不冲突的全局资产（大模型 API 密钥、云存储凭证）——从 `.env.template` 派生配置，只替换 `PORT` 与 `DATABASE_URL`，不为每个 worktree 单独申请密钥。Relay 的装配脚本核心是并发安全的槽位分配：

```bash
# scripts/setup_worktree.sh（节选）
WT_NAME=$(basename "$(pwd)")
exec 9>"/tmp/relay_worktree_allocation.lock"
flock -x 9                              # 对共享槽位表排他加锁
SLOT=$(grep "^${WT_NAME} " "$SLOT_FILE" | cut -d' ' -f2 || true)
if [ -z "$SLOT" ]; then                 # 找 1-5 中未占用的最小槽位
  for s in {1..5}; do
    cut -d' ' -f2 "$SLOT_FILE" | grep -q "^$s$" || { SLOT=$s; break; }
  done
  echo "${WT_NAME} ${SLOT}" >> "$SLOT_FILE"
fi
flock -u 9
PORT=$((3000 + SLOT))                    # 决定性端口派生
[ -f .env ] || sed -e "s/{{PORT}}/${PORT}/g" .env.template > .env   # 幂等配置
```

我们将上述装配脚本的规范与生成逻辑固化在 `./AGENTS.md` 的环境约定中。脚本本身由 Agent 生成、人 review 后合入代码库。此后，所有临时 worktree 均通过执行同一套装配逻辑来保障运行环境的完全对齐。这正是 Harness 架构中“文档约束生成”原则的具体落地。


## SessionStart Hook：Agent 的冷启动上下文

装配脚本解决了底层环境资源，但每次在某个 worktree 打开新会话时，Agent 面对的是空白上下文：不知道自己在哪条任务线、当前分支意图是什么、该跑哪个端口。`SessionStart` hook 在会话启动时自动注入这些状态，配置本身提交进代码库、所有 worktree 共享：

```bash
# .claude/wt-session-start.sh — AI 生成、人 review 后提交
echo "当前分支: $(git branch --show-current)"
echo "=== 项目约束 ==="; cat AGENTS.md
echo "=== 当前任务 ==="; grep -E "^\- \[.\]" TODOS.md | head -10
echo "=== 本 Worktree 环境 ==="; grep -E "^(PORT|DATABASE_URL)=" .env
```

这确保了 Agent 在接收用户首条输入前，已加载当前分支名、`./AGENTS.md` 不变量、尚未完成的 `TODOS.md` 任务项和本地 `.env` 端口配置。

在未引入该 Hook 前，Relay 曾发生过一起故障：开发人员在 worktree `RELAY_REFUND_01` 中修改退款逻辑，因开发中断而于次日开启了新会话。由于缺乏状态注入，新 Agent 无法感知分支历史，误将前一日未完成的幂等键校验（Idempotency Key Check）当成“前序遗留冗余代码”直接删除，转而编写新模块。这一退化在代码评审阶段才被发现，延误了交付。该 Hook 建立后，Agent 在启动瞬间即被置于完整的任务上下文之中，彻底杜绝了状态遗忘。


## 隔离之后，纪律才是真正的保障

然而，即便是完备的物理隔离，在缺乏协作纪律时依然会引发逻辑退化。在一次开发中，会话 A 负责退款判定逻辑，中途发现路由分类器中某个公共辅助函数的行为不符合预期，便“顺手”在其 worktree 分支中进行了重构；与此同时，会话 B 正好依赖该函数的旧行为在开发分类逻辑。由于物理目录彼此隔绝，编译与单测均未报警，但该函数的语义改变导致了两条任务线发生隐性的逻辑冲突。直到拉起 Pull Request 触发合并时，才暴露了这一严重的重构错位。

因此，隔离只是基础，纪律才是并行的最终保障：

* **单一任务契约**：每个会话只被授权解决一个职责高度单一的原子任务。接口先行，如果在开发中发现前置依赖或公共组件不合理，必须暂停当前 worktree，在 `./TODOS.md` 中记录并锁定一个独立的重构待办，禁止在当前分支中做越界修改。原子任务对应原子 PR，评审粒度足够小，合并时的逻辑冲突才能得到有效拦截。
* **迭代护栏**：AI Agent 缺乏自我停机的主动意识，一旦陷入死循环会产生巨大的 token 消耗。必须在 `.agents/config.yaml` 或系统配置中设定双重强制保护：`max_iterations: 12`（单次会话轮次上限）与 `session_timeout_ms: 300000`（5 分钟绝对运行时间上限）。一经越界，系统物理强杀，等待人工介入。
* **合并即清理**：worktree 合并后，其本地目录与分配的测试沙箱库必须随之销毁。残留的物理资源若不及时回收，将迅速把后续的可用插槽和端口填满，大幅增加调试与资源维护成本。在 Relay 中，合并脚本的尾部步骤必然挂载 `wt-clean.sh`，用于清除对应的 worktree 物理目录及相关测试数据库。


## gstack 参考实现

gstack 把 worktree 创建、槽位分配、SessionStart 装配封装成内置命令：

```bash
$ gstack sandbox create --id RELAY_REFUND_01 --branch feat/refund-decision  # 底层执行 git worktree add
$ gstack session-start --worktree-id RELAY_REFUND_01                        # 并发安全占槽并导出配置
```

工具是可替换的实现层，要旨是将“物理目录隔离”延展到“端口与数据实例隔离”，通过完全幂等的 Shell 脚本自动装配，并配合生命周期钩子做到“合并即清理”。


## 典型反模式

* **无护栏自动重试**：最昂贵的一类反模式。Agent 编译卡住且缺乏迭代轮次限制，进入“修改→失败→再修改”的死循环。迭代护栏配置必须在开会话前固化在配置文件中，不可等发生失控后再补。
* **随机端口盲目匹配**：本地重启后连接端口频繁漂移，导致前后端联调和数据库连接路径无法固定，网络错误极难稳定复现。
* **槽位分配缺失互斥锁**：多条并行线在同一秒被触发拉起，分配脚本发生并发读写冲突，分到同一端口，因 `EADDRINUSE` 崩溃。
* **陈旧 Worktree 物理堆积**：分支合并后不清目录与沙箱库，运行插槽和本地网络端口被长期占用，造成后续装配故障。
* **不提交 SessionStart hook**：脚本仅存放在本地而不提交到代码仓，新成员或新机器接入时遗漏该机制，导致 Agent 回到冷启动上下文。


## 本章要点

* 并行效能来自多开会话，但逻辑会话隔离不等同于物理隔离，必须依靠代码和动态环境双层防线进行物理划分。
* `git worktree` 保证静态目录与分支物理隔离，文件修改互不可见；装配脚本实现动态网络端口和沙箱实例隔离，运行时完全互不相交。
* 自动装配遵循三项核心设计：进程级排他锁、决定性端口派生、幂等可重入。
* `SessionStart` hook 配合 `./TODOS.md` 自动进行状态注入，确保新开会话获得确定性的任务与不变量上下文。
* 物理隔离必须辅以协作纪律：单一任务契约锁定边界、双重迭代护栏防止死循环开销、合并即清理阻断资源堆积。
