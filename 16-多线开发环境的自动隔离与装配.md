# 多线开发环境的自动隔离与装配

多线并行开发的技术可行性依赖于运行环境的物理隔离。如果我们仅在代码层面隔离了目录，却要依赖人工去手动挑空闲端口、建测试数据库以及修改环境变量，繁琐的环境装配流程很快就会彻底打消团队使用并行的积极性，重新退回到低吞吐的单线开发模式。

**必须在会话启动钩子（SessionStart Hook）中部署并发安全的资源分配与环境装配脚本。使新开辟的每一个 Worktree 目录在首轮开工时，一键自动化装配出独立且不撞车的网络端口与沙箱数据库。**

自动化并不是一个用来点缀的次要工具，而是决定并发 Worktree 体系能否在真实工业研发中应用的关键分水岭。

## 动态装配的四个硬性原则

要让 Agent 在新的 Worktree 中做到“插电即用、无感就绪”，装配脚本的编写必须严格遵循如下四项原则：

* **会话启动自动触发**：使用 Git Hook 拦截 Agent 第一次切入工作目录或启动客户端的时刻，自动触发装配，无需开发人员手动运行配置命令。
* **决定性端口派生**：根据当前并发安全分配的“槽位（Slot）”编号，以 `基准端口 + 槽位偏移` 的算术公式静态派生出该会话的专有端口。严禁使用随机端口探测（那样会导致每一次重启端口都会漂移，使联调极为困难）。
* **分配机制并发安全**：几条并行线可能在同一秒内被多开启动，必须使用物理文件锁（flock）对共享的槽位表进行排他性加锁读取与占用，防止两个会话抢到同一个槽位发生端口碰撞。
* **装配逻辑完全幂等**：装配脚本必须是可重入的。同一 Worktree 目录多次重复运行脚本，结果必须保持完全一致，已起好的环境自动复用，严禁每跑一次就重复派生或修改配置文件。

## 密钥复用与端口隔离

在动态环境隔离中，我们只需隔离那些会引发物理资源冲突的实体（如 API 监听端口、沙箱数据库名、Redis 缓存实例），而应当复用不冲突的全局资产（如大模型调用 API 密钥、云平台存储凭证）。

开发人员不应为每一个 Worktree 各自去申请和配置一套外部 API 密钥，这不仅引入了繁琐的安全运维开销，也增加了密钥泄露的风险。

正确的做法是采用“从主模板派生配置”的方案。装配脚本自动读取根目录下的 `.env.template`（包含全局共享的只读密钥），通过字符串替换（如 `sed` 命令）仅修改其中的 `PORT` 与 `DATABASE_URL` 字段，并生成专属于该目录的 `.env` 配置文件。这既保证了资源隔离的纯净度，又规避了到处手动配置密钥的麻烦。

## 主线演练：Relay 自动化装配 Hook 脚本与执行迹线

在 Relay 示例项目中，大仓提供了一键装配脚本 `scripts/setup_worktree.sh`。该脚本在 Agent 切入工作区或初始化环境时自动调用：

```bash
#!/usr/bin/env bash
# scripts/setup_worktree.sh
set -euo pipefail

# 1. 确定当前 Worktree 目录名作为唯一标识
WT_NAME=$(basename "$(pwd)")
LOCK_FILE="/tmp/relay_worktree_allocation.lock"

# 2. 并发安全地分配槽位 (Slot)
exec 9>"$LOCK_FILE"
flock -x 9

SLOT_FILE="/tmp/relay_worktree_slots.db"
touch "$SLOT_FILE"

# 查找当前 Worktree 是否已分配槽位，若没有则分配一个空闲的 (1-5)
SLOT=$(grep "^${WT_NAME} " "$SLOT_FILE" | cut -d' ' -f2 || true)
if [ -z "$SLOT" ]; then
  # 找出 1-5 中未被占用的最小数字
  USED_SLOTS=$(cut -d' ' -f2 "$SLOT_FILE" | sort -n)
  SLOT=1
  for s in {1..5}; do
    if ! echo "$USED_SLOTS" | grep -q "^$s$"; then
      SLOT=$s
      break
    fi
  done
  echo "${WT_NAME} ${SLOT}" >> "$SLOT_FILE"
fi

flock -u 9

# 3. 计算端口与数据库映射
PORT=$((3000 + SLOT))
DB_NAME="relay_sandbox_${WT_NAME}"

# 4. 基于模板生成专属环境变量配置文件
if [ ! -f .env ]; then
  sed -e "s/{{PORT}}/${PORT}/g" \
      -e "s/{{DB_NAME}}/${DB_NAME}/g" \
      -e "s/{{REDIS_DB}}/${SLOT}/g" \
      .env.template > .env
  echo "[INFO] Generated isolated environment config in .env for ${WT_NAME} (Port: ${PORT}, DB: ${DB_NAME})"
fi
```

当 Agent 第一次进入 `wt-refund-01` 目录开展工作时，该脚本自动执行并输出：

```bash
$ bash ../scripts/setup_worktree.sh
[INFO] Generated isolated environment config in .env for wt-refund-01 (Port: 3001, DB: relay_sandbox_wt-refund-01)
```

通过这套机制，各线开发环境实现了零人工干预的就绪，为并发开发彻底扫清了环境装配摩擦力。

## 典型反模式

* **随机端口盲目匹配**：每一次重启环境都使用系统随机分配的临时端口，导致前后端联调以及数据库连接的配置路径频繁漂移，无法稳定复现网络问题。
* **槽位分配缺乏进程互斥锁**：当三条并行线同时被自动化流程拉起时，因槽位读取冲突，多线同时分到了相同的端口，导致运行时抛出 `EADDRINUSE` 错误崩溃。
* **环境初始化缺乏幂等校验**：重复执行启动命令导致本地测试库被清空重置，破坏了 Agent 在上一次调试时积累的测试脏数据状态。

## 本章要点

* 并行开发必须配备在会话启动时自动触发的 `SessionStart` 动态环境装配机制，做到新线插电即用。
* 动态隔离聚焦于网络端口、隔离数据库等冲突实体，复用主仓共享的外部接口密钥凭证。
* 装配脚本必须具备并发安全加锁、静态端口派生以及幂等可重入三大硬性特质，彻底消除手动维护成本。

> 并行环境的自动装配确保了开发线的高吞吐与零碰撞。但当代码最终合入并投入运行后，我们必须确保它产生的行为在生产环境是可追踪和可诊断的。我们将在第 17 章 `写出来就可诊断` 中探讨如何设计机器友好的结构化日志系统。
